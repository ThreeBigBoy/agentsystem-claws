# 技术方案评审纪要规范

> 本规范为 `architecture-review` 技能的配套 REFERENCE，固化技术方案评审纪要的**内容结构、判定标准、输出格式**，为所有项目提供自检留痕的标准模板和实践参考。
> 
> 与 `project-analysis/REFERENCE/技术方案与架构产出物-最小结构与自检.md` 中的「四、评审纪要输出」保持一致，本规范侧重**操作层面的模板和示例**。

---

## 一、评审纪要定位

### 1.1 什么是技术方案评审纪要

**技术方案评审纪要**是技术方案自检过程的**详细留痕文档**，记录：

- 评审了什么（自检项）
- 发现了什么（评审发现）
- 结论是什么（判定结论：通过/有条件通过/不通过）
- 为什么是这个结论（判定理由、引用依据）
- 如何改进（修改建议）
- 修改后如何（修改后复核）

### 1.2 评审纪要 vs 迭代日志

| 对比维度 | 迭代日志 | 技术方案评审纪要 |
|---------|---------|-----------------|
| **定位** | 谁在什么时间做了什么事（日志型） | 评审了什么、结论是什么、为什么（评审型） |
| **详细程度** | 简明扼要（一行记录） | 详细完整（多章节文档） |
| **用途** | 追溯执行历史 | 追溯评审决策、支撑架构质量改进 |
| **存放** | `docs/项目事件日志.md` | `docs/project-prd-changes/[change-id]/records/技术方案-[change-id]-评审纪要.md` |

### 1.3 为什么必须留痕

1. **可追溯**：后续查看技术方案为何如此设计，可参考评审纪要中的决策逻辑
2. **可对齐**：团队成员/其他 Agent 可通过评审纪要快速理解技术方案的成熟度和关键决策
3. **可改进**：积累评审经验，形成可复用的评审模式
4. **可审计**：质量审计时可查看评审过程是否规范、结论是否合理
5. **可传承**：新成员可通过历史评审纪要学习架构设计规范和最佳实践

## 三、评审判定标准（速查表）

### 3.1 单项评审判定

| 判定 | 图标 | 含义 | 后续行动 |
|-----|------|------|---------|
| **通过** | ✓ | 内容完整、逻辑自洽、符合架构最佳实践 | 无需修改，进入下一项 |
| **有条件通过** | △ | 基本可用，但存在 minor 问题或建议优化项 | 记录问题，可在当前阶段或后续迭代优化 |
| **不通过** | ✗ | 缺失、逻辑不自洽、不满足落地要求 | 必须修改，修改后复核 |

### 3.2 整体评审结论

| 综合判定 | 条件 | 后续行动 |
|---------|------|---------|
| **通过** | 所有自检项均通过（或仅 minor 建议，无阻塞问题） | 可进入 coding-implement 阶段 |
| **有条件通过** | 存在有条件通过项，但无阻塞性问题 | 记录优化项，可并行进入编码阶段，但须在迭代中完成优化 |
| **不通过** | 存在不通过项 | 必须修改技术方案，重新评审 |

## 四、评审维度参考框架（详细版）

### 4.1 架构层评审

| 维度 | 评审要点 | 检查方法 | 常见不通过原因 |
|-----|---------|---------|---------------|
| **架构合理性** | 总体架构设计是否合理？模块/层次划分是否清晰？边界与职责是否明确？ | 检查「架构与模块」章节；对照 project.md 架构模式 | 模块职责重叠；层次混乱；边界模糊 |
| **可扩展性** | 架构是否支持未来扩展？新增功能是否需要大规模重构？ | 检查是否有扩展性设计；是否预留扩展点 | 紧耦合设计；扩展需要修改核心模块 |
| **技术选型 ROI** | 技术栈选型是否合理？是否引入不必要的复杂度？ | 检查技术选型理由；对比替代方案 | 过度设计（如简单需求引入微服务）；技术选型与团队能力不匹配 |

### 4.2 实现层评审

| 维度 | 评审要点 | 检查方法 | 常见不通过原因 |
|-----|---------|---------|---------------|
| **需求满足度** | 技术方案是否 100% 满足 PRD 期望实现需求和目标？ | 对照 PRD 功能列表逐项检查技术方案覆盖情况 | 遗漏 PRD 功能点；技术方案与需求不匹配 |
| **接口设计清晰度** | 接口契约是否清晰？数据模型是否合理？与现有系统兼容性如何？ | 检查「接口与数据」章节；检查字段类型、约束、示例 | 接口定义模糊（如字段类型未定义）；数据模型冗余或不完整 |
| **流程完整性** | 关键业务流程是否有清晰描述？状态流转是否完整？ | 检查「关键流程与执行逻辑」章节；对照「执行逻辑图」 | 流程描述缺失关键环节；状态流转不完整 |
| **异常处理完备性** | 异常处理策略是否完备？安全与性能要点是否覆盖？ | 检查「异常、安全与性能」章节 | 异常处理缺失；安全考虑不足（如未考虑鉴权）；性能指标未定义 |

### 4.3 落地层评审

| 维度 | 评审要点 | 检查方法 | 常见不通过原因 |
|-----|---------|---------|---------------|
| **可实现性** | 技术方案是否可被前端/后端按图实现？实现路径是否清晰？ | 检查是否有足够的实现细节；是否可被 coding-implement 技能直接使用 | 描述过于抽象，无法指导编码；缺少关键实现细节 |
| **可验证性** | 技术方案是否可被 code-review/func-test 对照验证？ | 检查是否有明确的验收点；是否可对照 specs 验证 | 验收标准模糊；无法对照验证 |
| **可追溯性** | 与 PRD、specs 的追溯关系是否明确？ | 检查「与需求/PRD 的对应」章节；检查与 specs 的对应关系 | 与 PRD 不一致；与 specs 冲突；追溯关系缺失 |

## 五、常见问题与处理建议

### 5.1 架构设计过于简略

**现象**: 技术方案只有简单几句话描述架构，无模块划分、无层次说明

**处理**:
- 判定：✗ 不通过
- 修改建议：补充「架构与模块」章节，明确系统/模块/层次、边界与职责、与 project.md 的对应关系
- 引用依据：架构设计必须「可被前端/后端按图实现」

### 5.2 接口定义模糊

**现象**: 接口定义写「传入用户ID」「返回用户信息」，无字段类型、无约束、无示例

**处理**:
- 判定：✗ 不通过
- 修改建议：补充接口契约：请求方法/路径、请求参数（名称/类型/必填/约束）、响应格式（字段/类型/示例）、错误码
- 引用依据：接口设计必须「可被 code-review/func-test 对照验证」

### 5.3 与 PRD 不一致

**现象**: 技术方案的功能范围与 PRD 不一致，或技术实现与需求描述冲突

**处理**:
- 判定：✗ 不通过（严重）/ △ 有条件通过（轻微差异）
- 修改建议：对齐技术方案与 PRD，确保 100% 满足需求；如有技术约束导致无法满足，需在技术方案中明确说明并征得产品确认

### 5.4 缺少异常处理

**现象**: 技术方案只有正常流程描述，无异常处理策略

**处理**:
- 判定：△ 有条件通过（简单功能）/ ✗ 不通过（复杂功能）
- 修改建议：补充「异常、安全与性能」章节，覆盖关键异常场景、处理策略、降级方案

### 5.5 无图示或图示未引用

**现象**: 技术方案提到「见架构图」「见流程图」，但实际无图或图未存放

**处理**:
- 判定：△ 有条件通过
- 修改建议：补充架构图/流程图，存放于 `docs/project-prd-changes/[change-id]/architecture/`，并在技术方案中正确引用

### 5.6 命名不规范

**现象**: 技术方案文件命名为 `设计.md`、`方案.md` 等无识别性名称

**处理**:
- 判定：△ 有条件通过
- 修改建议：技术方案正文**仅**保留 `openspec/changes/[change-id]/design.md`；若在 `docs/project-prd-changes/` 存在第二份「技术方案-*.md」正文，应合并进 `design.md` 后删除重复。

## 六、评审纪要示例（节选）

### 示例：自检项 #5 关键流程与图示评审

```markdown
### 自检项 #5: 关键流程与图示

- **评审内容**: 关键流程是否有文字描述或执行逻辑图/数据流图？图示是否已存放并在 design.md 或 project-rules 中引用？
- **评审发现**: 
  - 「阶段化执行」关键流程有文字描述，但缺少图示
  - 「/run → checkpoint → /resume」流程只有文字，无序列图
  - 「数据流」章节提到「见数据流图」，但 architecture/ 目录下无该图
- **判定结论**: △ 有条件通过
- **判定理由**: 
  - 文字描述基本清晰，可满足编码实现需求
  - 但涉及多模块交互（MCP 工具层、LangGraph 后端、Checkpoint 存储），仅有文字易产生歧义
  - 引用数据流图缺失，影响可读性和可验证性
  - 根据架构最佳实践，跨模块交互流程建议补充序列图或执行逻辑图
- **修改建议**: 
  - 补充「/run 阶段化执行」序列图，展示：客户端 → MCP 工具层 → /run API → LangGraph 执行 → Checkpoint 存储 → /resume API → 结果返回
  - 补充「数据流图」，展示数据在各模块间的流向
  - 图示存放于 `docs/project-prd-changes/[change-id]/architecture/`，并在 design.md 中引用
- **引用依据**: 
  - `skills/project-analysis/REFERENCE/技术方案与架构产出物-最小结构与自检.md` 第 1 节「1.3 技术架构图、执行逻辑图、数据流图等产出物的存放与引用」
  - 架构最佳实践：跨模块交互建议用图示辅助理解，减少歧义
- **修改后复核**: 
  - 待补充图示后复核
```

## 八、PRD 评审纪要 vs 技术方案评审纪要

| 对比维度 | PRD 评审纪要 | 技术方案评审纪要 |

| **评审对象** | PRD（产品需求文档） | 技术方案（design.md） |
