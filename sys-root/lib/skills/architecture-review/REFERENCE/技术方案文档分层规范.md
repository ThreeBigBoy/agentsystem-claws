# 技术方案文档分层规范

> **说明**: 明确 `design-brief.md` 与 `design.md` 的分层关系，确保技术方案评审时先检查概述层，再检查实现层。

---

## 文档分层体系

```
┌─────────────────────────────────────────────────────────────────┐
│                    技术方案文档分层体系                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Layer 1: 概述层 (Overview)                                      │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━                                      │
│  文档: design-brief.md                                           │
│  目标读者: 非技术人员、产品经理、项目经理、新加入开发者          │
│  内容特点:                                                       │
│    • 架构图、流程图（可视化为主）                                │
│    • 文字说明（无代码或极少代码）                                │
│    • 目录索引（快速定位design.md内容）                           │
│    • 产出物清单、执行人映射、输入信息源                          │
│  必须先于: design.md 存在                                        │
│                                                                  │
│                           ↓ 引用/索引                           │
│                                                                  │
│  Layer 2: 实现层 (Implementation)                                │
│  ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━                                 │
│  文档: design.md                                                 │
│  目标读者: 开发者、架构师、Reviewer                              │
│  内容特点:                                                       │
│    • 代码示例（核心实现逻辑）                                    │
│    • 接口协议（API定义、数据模型）                               │
│    • 详细设计（状态机、算法、配置）                              │
│    • 引用 brief 中的架构图作为实现依据                           │
│  必须后于: design-brief.md 存在                                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 评审顺序规范

### 模式A：单文档模式（默认）

**检查** `openspec/changes/[change-id]/design.md` 是否存在。

**检查要点**:
| 检查项 | 判定标准 |
|-------|---------|
| 文档存在性 | design.md 必须存在 |
| 概述章节 | 是否有文字说明 + 架构图（无代码） |
| 详细设计章节 | 是否有代码示例 + 接口协议 |
| 结构规范性 | 6章节结构是否完整 |
| 与PRD一致性 | 是否满足PRD需求 |
| 可理解性 | 非技术人员能否理解概述章节 |

**评审结论**:
- ✓ **通过**: design.md 结构完整，概述清晰，代码示例充分
- △ **有条件通过**: 结构基本完整，但需要优化概述章节或补充代码示例
- ✗ **不通过**: 缺少核心章节或代码示例严重不足

### 模式B：双文档模式（特殊情况）

**首先检查** `openspec/changes/[change-id]/design-brief.md` 是否存在。

**如果不存在 brief**:
- 检查 design.md 行数
- 如果 <3000行：按单文档模式评审
- 如果 ≥3000行：建议产出 brief 以提升可读性（但不是强制）

**如果存在 brief**:
- 按双文档模式评审（参见下文）

## 单文档模式评审要点（design.md）

### 概述章节检查（前3章）
| 检查项 | 判定标准 |
|-------|---------|
| 章节存在性 | 第1-3章为概述章节（文字+架构图） |
| 代码隔离 | 概述章节应无代码或<5处伪代码 |
| 架构图 | 核心架构图是否包含 |
| 可理解性 | 非技术人员能读懂概述章节 |

### 实现章节检查（后3章）
| 检查项 | 判定标准 |
|-------|---------|
| 代码示例 | 核心实现是否有代码示例 |
| 接口协议 | API定义是否清晰 |
| 一致性 | 实现章节是否与概述章节一致 |

---

## 双文档模式评审要点（brief + design）

### 当存在 design-brief.md 时

**检查要点**:
| 检查项 | 判定标准 |
|-------|---------|
| 文档存在性 | design-brief.md 作为辅助文档存在 |
| 架构图完整性 | 核心架构图是否包含 |
| 产出物清单 | 10步闭环产出物清单是否完整（可选） |
| 可理解性 | 非技术人员能否理解 |
| 目录索引 | 指向design.md的快速定位表（可选） |

**然后检查 design.md**:
| 检查项 | 判定标准 | 与brief的关系 |
|-------|---------|--------------|
| 结构规范性 | 6章节结构是否完整 | 应该与brief的架构描述对应 |
| 代码示例 | 核心实现是否有代码示例 | 应该实现brief中描述的架构 |
| 接口协议 | API定义是否清晰 | 应该符合brief中的接口概述 |
| 与brief一致性 | 实现是否与概述一致 | 必须保持一致 |

## 两个文档的引用关系

### design-brief.md 引用 design.md

```markdown
## 快速定位指引（For Agent）

| 想了解 | 查看本章节 | 详细代码位置（design.md） |
|--------|-----------|------------------------|
| 整体架构 | [第2章](#2-核心架构图) | - |
| Step 4实现 | [第4章](#4-step-4双技能协同评审) | 约5017-5256行 |
| 自动发现机制 | [第5章](#5-可插拔扩展架构) | 约1557-1800行 |
```

**作用**: brief 作为目录索引，指向 design.md 的具体实现位置。

### design.md 引用 design-brief.md

```markdown
## 1. 概述

本技术方案的架构设计和流程说明详见 [design-brief.md](../design-brief.md)。

核心架构图引用：
![架构图](./design-brief.md#2-核心架构图)
```

**作用**: design.md 在详细实现前，引用brief中的架构图作为实现依据。

## 评审时发现的问题类型

### 问题类型1: 缺少概述层

**表现**: 只有 design.md，没有 design-brief.md

**判定**: ✗ 不通过

**理由**: 
- 非技术人员无法理解技术方案
- 缺少目录索引，Agent难以快速定位
- 产出物不完整

**修改建议**: 
1. 先产出 design-brief.md（架构图、流程图、文字说明）
2. 确保 brief 中的架构设计合理
3. 再基于 brief 产出 design.md（代码实现）

### 问题类型2: 概述与实现不一致

**表现**: design-brief.md 中描述的架构与 design.md 中的实现不符

**判定**: △ 有条件通过 或 ✗ 不通过

**示例**:
```markdown
# design-brief.md 中描述：
"Step 4 双技能协同评审：先执行 architecture-review，再执行 technical-design-review"

# design.md 中实现：
"Step 4 只执行 architecture-review，没有 technical-design-review"
```

**修改建议**:
1. 以 design-brief.md 中的架构设计为准
2. 修改 design.md 的实现，使其与brief一致
3. 或者，如果实现更合理，则更新brief中的架构描述

### 问题类型3: 概述层过于详细（包含代码）

**表现**: design-brief.md 中包含大量代码示例

**判定**: △ 有条件通过

**理由**: 
- brief 的目标是非技术人员，代码应该放在design.md中
- brief 应该聚焦在"是什么"和"为什么"，而非"怎么做"

**修改建议**:
1. 将brief中的代码示例移动到design.md
2. brief中保留架构图、流程图、文字说明
3. 添加指向design.md具体章节的链接

### 问题类型4: 实现层缺少代码示例

**表现**: design.md 只有文字描述，没有代码示例或接口协议

**判定**: △ 有条件通过 或 ✗ 不通过

**理由**:
- 开发者需要代码示例才能按图实现
- 接口协议不明确会导致实现偏差

**修改建议**:
1. 在design.md中添加核心代码示例
2. 定义清晰的接口协议（API、数据模型）
3. 确保代码示例与brief中的架构一致

## 产出物规范性检查清单

### 概述层检查清单 (design-brief.md)

```markdown
- [ ] 文档存在：`openspec/changes/[change-id]/design-brief.md`
- [ ] 核心架构图：包含系统架构图、数据流图
- [ ] 工作流程图：10步闭环流程可视化
- [ ] 产出物清单：10步闭环产出物清单完整
- [ ] 执行人映射：各Step负责人明确
- [ ] 输入信息源：各Step输入信息源定义
- [ ] 执行规范栈：MUST/SHOULD/MAY规范定义
- [ ] 无代码或极少代码：<5处核心伪代码
- [ ] 可理解性：非技术人员能读懂
- [ ] 目录索引：指向design.md的快速定位表
```

### 实现层检查清单 (design.md)

```markdown
- [ ] 文档存在：`openspec/changes/[change-id]/design.md`
- [ ] 6章节结构：符合技术方案最小结构
- [ ] 与brief一致：实现与概述架构一致
- [ ] 与PRD一致：满足PRD需求
- [ ] 代码示例：核心实现有代码示例
- [ ] 接口协议：API定义清晰
- [ ] 引用brief：在概述章节引用brief
```
