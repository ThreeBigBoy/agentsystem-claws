## 触发场景

- 用户输入「评审技术方案结构」「检查架构设计规范性」「技术方案文档自检」「审查 design.md 结构」等指令
- 用户要求对已产出的技术方案进行**文档规范性检查**、内容完整性验证、与PRD一致性检查
- 用户需要在进入 coding-implement 阶段前，确保技术方案文档达到「可被前端/后端按图实现」标准
- 用户需要回溯补录技术方案评审纪要（基于历史对话和已产出文档）
- **注意**: 如果用户询问"技术方案是否生产就绪""是否有安全/稳定/观测模块""代码实现是否完整"，应使用 `technical-design-review` 技能

## 整体流程

### 0. 加载技术方案评审规范

**必须首先读取**以下两个 REFERENCE 文档：

1. **`agentsystem/skills/architecture-review/REFERENCE/技术方案文档分层规范.md`** ⭐新增必读⭐
   - 重点阅读「文档分层体系」了解 design-brief.md（概述层）与 design.md（实现层）的关系
   - 重点阅读「评审顺序规范」了解必须先检查brief，再检查design
   - 重点阅读「两个文档的引用关系」了解如何验证一致性

2. **`agentsystem/skills/project-analysis/REFERENCE/技术方案与架构产出物-最小结构与自检.md`**
   - 重点阅读「一、最小结构（模板）」了解技术方案应包含的 6 个章节
   - 重点阅读「二、技术方案文档命名规范」了解命名要求
   - 重点阅读「三、自检清单」了解 9 项自检项
   - 重点阅读「四、评审纪要输出」了解评审纪要的内容结构和输出格式

### 1. 确定评审对象

- 从用户输入或上下文中识别本次评审的 **change-id**
- 若未给出 change-id，应向用户询问

### 2. 检查技术方案文档（确定文档模式）

**Step 2.1: 确定文档模式**

检查是否存在 `openspec/changes/[change-id]/design-brief.md`：

**模式A：单文档模式（默认，占80%+）**
- 如果不存在 design-brief.md
- 检查 `design.md` 是否包含：
  - [ ] 概述章节（前3章：文字+架构图，无代码或<5处伪代码）
  - [ ] 详细设计章节（后3章：代码示例+接口协议）
- 进入 **步骤3: 单文档模式评审**

**模式B：双文档模式（特殊，占<20%）**
- 如果存在 design-brief.md（通常因为代码示例过多>3000行）
- 先检查 brief（概述层），再检查 design（实现层）
- 验证两层一致性
- 进入 **步骤3: 双文档模式评审**

**Step 2.2: 单文档模式评审要点**

检查 `openspec/changes/[change-id]/design.md`:

| 检查项 | 判定标准 |
|-------|---------|
| 文档存在性 | design.md 必须存在 |
| 概述章节 | 前3章应为概述（文字+架构图，无代码或<5处伪代码） |
| 详细设计章节 | 后3章应有代码示例和接口协议 |
| 结构规范性 | 6章节结构是否完整 |
| 与PRD一致性 | 是否满足PRD需求 |
| 可理解性 | 非技术人员能读懂概述章节 |

**Step 2.3: 双文档模式评审要点**

如果存在 design-brief.md:

检查 brief（概述层）：
- 读取 design-brief.md 全文
- 检查核心架构图、产出物清单、可理解性（无代码或<5处伪代码）
- 记录 brief 的评审状态

检查 design.md（实现层）：
- 读取 `openspec/changes/[change-id]/design.md`
- 同时读取关联的 PRD 文档
- 理解技术方案的当前状态

**Step 2.4: 验证两层一致性（仅双文档模式）**

| 检查项 | brief中的描述 | design中的实现 | 一致性判定 |
|-------|--------------|---------------|-----------|
| 架构设计 | 系统架构图描述 | 代码实现是否符合架构 | 一致/不一致 |
| 流程设计 | 工作流程图描述 | 状态机实现是否符合流程 | 一致/不一致 |
| 产出物定义 | 产出物清单 | 实际产出是否符合清单 | 一致/不一致 |

**如果不一致**：
- 判定：△ **有条件通过** 或 ✗ **不通过**（视偏离程度）
- 建议：以brief中的架构设计为准，修改design.md的实现；或如果实现更合理，则更新brief

### 3. 执行系统化评审

按照「三、自检清单」的 9 项内容逐项评审：

| 序号 | 自检项 | 评审要点 |
|-----|-------|---------|
| 1 | 变更目标与范围 | 是否写清架构/技术层面的目标与影响范围？ |
| 2 | 与 project.md 一致 | 是否与项目宪法（架构模式、技术栈、目录结构）一致？ |
| 3 | 与需求/PRD 可追溯 | 是否与 PRD、specs 相关 Requirements 可互相引用、无冲突？ |
| 4 | 接口与数据 | 接口契约、数据模型是否清晰？与现有系统兼容性如何？ |
| 5 | 关键流程与图示 | 关键流程是否有文字描述或图示？图示是否已存放并引用？ |
| 6 | 异常与边界 | 是否覆盖关键异常、安全与性能要点、非目标？ |
| 7 | project-rules 从属关系 | project-rules 下文档是否标明为 project.md 的补充约束层？ |
| 8 | 与 openspec 一致 | 与 proposal、specs、tasks 是否可互相引用、无冲突？ |
| 9 | 唯一技术方案正文 | 技术方案是否**仅**存在于 `openspec/changes/[change-id]/design.md`（不在 `docs/project-prd-changes/` 另写一份「技术方案-*.md」正文）？ |

**评审方式**：
- 对照 REFERENCE 中的「一、最小结构」逐条检查技术方案内容
- 发现不满足项时，记录：评审发现、判定结论、判定理由、修改建议、引用依据
- 评审时须围绕「架构层、实现层、落地层」三个底层逻辑展开（见 REFERENCE「四、评审维度参考框架」）

### 4. 形成评审判定

对每项自检项给出判定：

- **✓ 通过**：内容完整、逻辑自洽、符合架构最佳实践
- **△ 有条件通过**：基本可用，但存在 minor 问题或建议优化项
- **✗ 不通过**：缺失、逻辑不自洽、不满足落地要求

**综合判定**：
- **通过**：所有自检项均通过（或仅 minor 建议，无阻塞问题）
- **有条件通过**：存在有条件通过项，但无阻塞性问题
- **不通过**：存在不通过项，必须修改技术方案

**评审结论与后续动作映射（重要）**：

| 评审结论 | 是否可以进入下一阶段 | 后续动作 |
|---------|-------------------|---------|
| **✓ 通过** | ✅ **可以** | 直接进入下一阶段（Step 5: 编码实现） |
| **△ 有条件通过** | ❌ **不可以** | **必须**修复问题清单中的 minor 建议，**重新评审**通过后，才能进入下一阶段 |
| **✗ 不通过** | ❌ **不可以** | **必须**修改技术方案，**重新评审**通过后，才能进入下一阶段 |

> ⚠️ **重要澄清**：「有条件通过」≠ 「可以进入下一阶段」。只有「100% 通过」才是真正的通过。详见 `memory/patterns/pattern-review-fix-loop.md`

### 5. 产出评审纪要

在 `docs/project-prd-changes/[change-id]/records/` 或 `openspec/changes/[change-id]/` 下创建评审纪要文档，命名：
- 推荐：`技术方案-[change-id]-评审纪要.md`
- 兼容：`技术方案评审纪要.md`

评审纪要必须包含以下章节：

#### 1. 评审基本信息

- 评审日期
- 评审人（Agent 角色：架构 Agent / 主 Agent）
- 技术方案版本
- 所属 change-id
- 评审依据（引用的 REFERENCE 文档路径）

#### 2. 评审过程记录

对每项自检项，按以下格式记录：

```markdown
### 自检项 #1: 变更目标与范围

- **评审内容**: design.md 是否写清本变更在架构/技术层面的目标与影响范围？
- **评审发现**: [详细描述技术方案中该部分的内容，是否满足要求]
- **判定结论**: ✓ 通过 / △ 有条件通过 / ✗ 不通过
- **判定理由**: [详细说明理由]
- **修改建议**: [给出具体修改建议]
- **引用依据**: [架构最佳实践、相关规范、技术文档等]
- **修改后复核**: [如已修改，记录复核结果]
```

#### 3. 问题发现与处理

汇总所有发现的问题：

| 序号 | 自检项 | 原始判定 | 问题类型 | 修改建议 | 状态 |
|-----|-------|---------|---------|---------|------|
| 1 | 架构与模块 | ✗ 不通过 | 缺失 | 补充模块边界说明 | 待修改 |

#### 4. 整体评审结论

- **综合判定**: 通过 / 有条件通过 / 不通过
- **判定理由**: 总结性说明（架构层、实现层、落地层评审结论）
- **后续行动项**: 明确的下一步行动

#### 5. 附录

- 参考文档列表
- 相关 memory 条目
- 自检清单原始链接

### 6. 反馈评审结论

向用户反馈：

1. **评审结果摘要**：综合判定（通过/有条件通过/不通过）
2. **关键问题**：阻塞性问题（如有）
3. **建议优化项**：minor 问题或改进建议
4. **评审纪要路径**：告知用户评审纪要文档存放位置
5. **后续行动建议**：
   - 若通过：可进入 Step 5（编码实现）
   - 若有条件通过：**不能进入下一阶段**。必须：① 修复问题清单中的 minor 建议 → ② 执行重新评审 → ③ 转为「通过」后，才能进入下一阶段
   - 若不通过：**不能进入下一阶段**。必须：① 修改技术方案解决阻塞性问题 → ② 执行重新评审 → ③ 转为「通过」后，才能进入下一阶段

**评审修复循环**：
```
首次评审 ──→ 有条件通过/不通过 ──→ 修复问题 ──→ 重新评审 ──→ 通过？──→ 否 → 继续修复
                                                         └──────→ 是 → 进入下一阶段
```

## 与 OpenSpec 的对应关系

| 产出物 | 位置 | 说明 |
|--------|------|------|
| 技术方案正文 | **`openspec/changes/[change-id]/design.md`（唯一）** | 评审对象 |
| PRD 文档 | `docs/project-prd-changes/[change-id]/PRD-[change-id]-[关键词].md` | 评审参考（确保技术方案与需求一致） |
| 评审纪要 | `docs/project-prd-changes/[change-id]/records/技术方案-[change-id]-评审纪要.md` | 本技能产出 |
| 规范依据 | `skills/project-analysis/REFERENCE/技术方案与架构产出物-最小结构与自检.md` | 必须读取的 REFERENCE |

## 执行后必做收尾

按 `skills-rules-for-agent.md` 通用约定，执行本技能后必须：

1. **向迭代日志追加记录**：

   **首次评审记录**（所有评审都必须记录）：
   - 文件：`docs/项目事件日志.md`
   - 格式：`- [日期] | [change-id] | architecture-review | 技术方案评审完成，综合判定：[通过/有条件通过/不通过]，评审纪要路径：[路径]`

   **重新评审记录**（如首次评审为「有条件通过」或「不通过」，修复后必须重新评审并记录）：
   - 格式：`- [日期] | [change-id] | architecture-review | 技术方案重新评审完成，修复项：[N 项]，综合判定：[通过]，评审纪要路径：[路径]`

   **示例**：
   ```markdown
   # 首次评审（有条件通过）
   - 2026-03-16 | check-langgraph-backend | architecture-review | 技术方案评审完成，综合判定：有条件通过，评审纪要路径：docs/project-prd-changes/check-langgraph-backend/records/技术方案-check-langgraph-backend-阶段化执行-评审纪要.md

   # 修复后重新评审（通过）
   - 2026-03-16 | check-langgraph-backend | architecture-review | 技术方案重新评审完成，修复项：1 项，综合判定：通过，评审纪要路径：docs/project-prd-changes/check-langgraph-backend/records/技术方案-check-langgraph-backend-阶段化执行-重新评审纪要.md
   ```

2. **更新技术方案状态**（可选）：
   - 在技术方案文档头部添加评审状态标记：
   ```markdown
   **评审状态**: ✓ 通过（[日期]）| 评审纪要: [路径]
   ```

## 注意事项

- **必须先读取 REFERENCE**：执行本技能前必须先读取 `技术方案与架构产出物-最小结构与自检.md`，确保评审标准统一
- **对照 PRD 评审**：技术方案必须与 PRD 对照评审，确保 100% 满足需求，无遗漏无冲突
- **详细记录**：评审纪要不是简单打勾，必须包含「评审发现、判定理由、修改建议、引用依据」
- **客观公正**：评审须基于 REFERENCE 规范，避免主观判断；判定理由必须有依据支撑
- **建设性反馈**：即使不通过，也要给出具体、可执行的修改建议
- **支持回溯评审**：用户可以要求基于历史对话和已产出文档回溯补录评审纪要
