# 技术方案与架构产出物 — 最小结构与自检

> 架构 Agent 执行 project-analysis 时，产出的 **design.md（技术方案）**、**project-rules/** 下文档以及**技术架构图、执行逻辑图、数据流图**等须符合本结构，并完成自检，使产出达到**可被前端/后端按图实现、可被 code-review/func-test 对照验证、可追溯**。  
> 与 `request-analysis/REFERENCE/迭代需求说明-PRD最小结构与自检.md` 形成「需求 → 技术方案 → 实现 → 验收」的配套规范。

---

## 一、最小结构（模板）

产出 **变更级技术方案**（`openspec/changes/[change-id]/design.md`）或 **工程补充约束**（`project-rules/` 下文档）时，须覆盖下列约定；具体章节可按项目与变更范围裁剪，但核心维度不缺失。

### 1.1 design.md（变更级技术方案）最小结构

当本次变更需要 **design.md**（见 OpenSpec 4.5：跨多服务/新架构模式/新外部依赖/安全性能或迁移复杂性/编码前技术决策）时，建议覆盖：

| # | 章节（建议） | 内容要点 |
|---|--------------|----------|
| 1 | **变更目标与范围** | 本变更在架构/技术层面的目标；影响模块、层次、数据流边界。 |
| 2 | **架构与模块** | 涉及的系统/模块/层次；新增或调整的边界与职责；与 project.md、project-rules 的对应关系。 |
| 3 | **接口与数据** | 新增或变更的接口（API/事件/消息）、数据模型或存储；与现有契约的兼容性。 |
| 4 | **关键流程与执行逻辑** | 核心业务流程、状态流转、关键路径；可引用「执行逻辑图」或「序列图」路径。 |
| 5 | **异常、安全与性能** | 异常处理策略、安全与合规要点、性能与容量考虑；非目标或后续迭代说明。 |
| 6 | **与需求/PRD 的对应** | 与 `docs/project-prd-changes/[change-id]/` 下 PRD、specs 中 Requirements 的追溯关系；必要时列出关键 Requirement ID 或小节。 |

- **图示引用**：若存在技术架构图、执行逻辑图、数据流图，在相应章节写明「见 `docs/project-prd-changes/[change-id]/architecture/xxx.md`」或「见 `project-rules/` 下 xxx」；图示与正文一致、可追溯。

### 1.2 project-rules/ 下文档建议结构

作为 **project.md 的补充约束层**，建议按主题分文件，每类文档覆盖：

| 类型（示例文件名） | 内容要点 |
|--------------------|----------|
| **核心概念/名词**（如 `glossary.md`、`core-concepts.md`） | 领域与工程中的关键术语、缩写、与平台/框架的对应关系。 |
| **产品/模块架构**（如 `architecture.md`、`module-boundaries.md`） | 模块/子系统划分、层级、边界与职责；可配架构图（见下节「设计产出物」）。 |
| **数据概念模型**（如 `data-model.md`） | 核心实体、字段、关系与存储约定；可配 ER 图或数据流图。 |
| **关键信息流/数据流**（如 `flows.md`、`data-flows.md`） | 全链路或关键场景下的数据流、调用顺序；可配流程图、序列图。 |
| **实现约束与最佳实践**（如 `implementation-constraints.md`） | 安全、性能、可观测性、可扩展性等约束；与 project.md 的从属关系。 |

- 在文内标明与 **openspec/project.md** 的从属关系；图示存放与引用见下节。

### 1.3 技术架构图、执行逻辑图、数据流图等产出物的存放与引用

| 产出物类型 | 说明 | 建议存放位置 | 在 design.md / project-rules 中的引用方式 |
|------------|------|--------------|--------------------------------------------|
| **技术架构图** | 系统/模块/层次结构、边界与依赖关系。 | `docs/project-prd-changes/[change-id]/architecture/` 或 `project-rules/` 下同目录 | 在 design.md 或 project-rules 对应章节写明「见 `architecture/xxx.png`」或「见 `project-rules/architecture-diagram.md`」。 |
| **执行逻辑图 / 流程图** | 关键业务流程、状态机、决策分支。 | `docs/project-prd-changes/[change-id]/architecture/` 或 `flows/` 子目录 | 在 design.md「关键流程与执行逻辑」或 project-rules flows 文档中引用。 |
| **数据流图** | 数据在模块/服务间的流向与转换。 | 同上或 `project-rules/` 下 `data-flows.md` 内嵌或附图 | 在 design.md「接口与数据」或 project-rules 数据流文档中引用。 |
| **序列图 / 时序图** | 跨模块/服务调用顺序与交互。 | 同上 | 在 design.md 或 project-rules 流程/接口章节引用。 |

- **格式**：图可为 Mermaid（.md 内嵌）、PNG/SVG（需配简短说明）；若为用户提供的图，可经 **image-analysis** 解析后，将结构化描述写入 design.md / project-rules，原图存上述目录并注明来源。
- **与 request-analysis 设计产出物区分**：PRD 侧的线框图/交互图侧重产品与交互，存放在 `docs/project-prd-changes/[change-id]/design-assets/`（wireframes、flows、mockups）；架构侧的技术架构图/执行逻辑图/数据流图侧重技术实现，可存放在 `docs/project-prd-changes/[change-id]/architecture/` 或 `project-rules/`，并在 design.md 与 project-rules 中引用。

## 二、技术方案正文：唯一文件 `design.md`

### 2.1 单一事实来源（强制）

- **变更级技术方案正文**有且仅有**一份**：**`openspec/changes/[change-id]/design.md`**（文件名固定为 `design.md`，与 OpenSpec / usr-rules 一致）。
- **禁止**在 `docs/project-prd-changes/[change-id]/` 下再维护一份「技术方案-*.md」作为第二套技术方案正文；与 PRD 重复的表述以 PRD 为准，技术决策与实现细节以 **`design.md`** 为准。
- **图示**（架构图、流程图等）可放在 `docs/project-prd-changes/[change-id]/architecture/` 等目录，**必须在 `design.md` 内引用**，不单独充当「第二份技术方案」。

### 2.2 与「技术方案评审纪要」的区分

| 类型 | 路径 | 说明 |
|------|------|------|
| **技术方案正文** | `openspec/changes/[change-id]/design.md` | 架构与实现决策的完整表述 |
| **技术方案评审纪要** | `docs/project-prd-changes/[change-id]/records/` 下（命名见 architecture-review 技能） | 评审过程与结论留痕，**不是**第二份方案正文 |

## 三、自检清单（产出后必过）

执行 project-analysis 并产出 design.md 或 project-rules 后，按下列项自检；不满足时补全再同步 proposal、specs、tasks。

| # | 自检项 | 说明 |
|---|--------|------|
| 1 | **变更目标与范围** | design.md 是否写清本变更在架构/技术层面的目标与影响范围？ |
| 2 | **与 project.md 一致** | 本次产出是否与 openspec/project.md 的架构模式、技术栈、目录结构一致？若有更新，是否已同步修订 project.md？ |
| 3 | **与需求/PRD 可追溯** | design.md 或 project-rules 是否与 docs/project-prd-changes/[change-id]/ 下 PRD、specs 中相关 Requirements 可互相引用、无冲突？ |
| 4 | **接口与数据** | 若有新增/变更接口或数据模型，是否已写清契约、兼容性与存储约定？ |
| 5 | **关键流程与图示** | 关键流程是否有文字描述或执行逻辑图/数据流图？图示是否已存放并在 design.md 或 project-rules 中引用？ |
| 6 | **异常与边界** | 是否覆盖关键异常、安全与性能要点、非目标或后续迭代？ |
| 7 | **project-rules 从属关系** | project-rules 下文档是否在文内标明为 project.md 的「补充约束层」？ |
| 8 | **与 openspec 一致** | design.md、project-rules 与 proposal、specs 是否可互相引用、无冲突？ |
| 9 | **唯一 design.md** | 技术方案是否只存在于 `openspec/changes/[change-id]/design.md`，且未在 `docs/project-prd-changes/` 重复维护第二份技术方案正文？ |

## 四、评审纪要输出（自检留痕）

### 4.1 定位与必要性

**技术方案自检 ≠ 简单打勾**。自检是一个「自我评审」过程，需要将评审的详细过程、结论、依据、建议留痕，形成**评审纪要**，以便于：

- **追溯**：后续查看技术方案为何如此设计，可参考评审纪要中的决策逻辑
- **对齐**：团队成员/其他 Agent 可通过评审纪要快速理解技术方案的成熟度和关键决策
- **改进**：积累评审经验，形成可复用的评审模式

### 4.2 评审纪要内容结构

每次技术方案自检后，须在 `docs/project-prd-changes/[change-id]/records/` 或 `openspec/changes/[change-id]/` 下创建**评审纪要文档**，命名建议：
- `技术方案-[change-id]-评审纪要.md`（遵循新命名规范）
- `技术方案评审纪要.md`（传统命名，仍兼容）

评审纪要须包含：

| 章节 | 内容要求 |
|-----|---------|
| **1. 评审基本信息** | 评审日期、评审人（Agent 角色）、技术方案版本、所属 change-id |
| **2. 评审过程记录** | 逐条过自检清单的详细记录 |
| **3. 问题发现与处理** | 发现的问题、判定结论（通过/有条件通过/不通过）、修改建议、引用依据 |
| **4. 整体评审结论** | 综合判定：通过 / 有条件通过 / 不通过；后续行动项 |
| **5. 附录** | 参考文档、相关记忆、自检清单原始链接 |

### 4.3 评审过程记录格式

对自检清单的每一项，须记录：

```markdown
### 自检项 #1: 变更目标与范围

- **评审内容**: design.md 是否写清本变更在架构/技术层面的目标与影响范围？
- **评审发现**: [详细描述技术方案中该部分的内容，是否满足要求]
- **判定结论**: ✓ 通过 / △ 有条件通过 / ✗ 不通过
- **判定理由**: [如果通过，简述通过的逻辑；如果不通过或有条件通过，详细说明理由]
- **修改建议**: [如果不通过或有条件通过，给出具体修改建议]
- **引用依据**: [支撑判定观点的参考：架构最佳实践、相关规范、技术文档等]
- **修改后复核**: [修改后再次评审的结果和日期]
```

### 4.4 评审判定标准

| 判定 | 含义 | 后续行动 |
|-----|------|---------|
| **✓ 通过** | 该项内容完整、逻辑自洽、符合架构最佳实践 | 无需修改，进入下一项 |
| **△ 有条件通过** | 该项基本可用，但存在 minor 问题或建议优化项 | 记录问题，可在当前阶段或后续迭代优化 |
| **✗ 不通过** | 该项缺失、逻辑不自洽、不满足落地要求 | 必须修改，修改后复核 |

### 4.5 整体评审结论标准

| 综合判定 | 条件 | 后续行动 |
|---------|------|---------|
| **通过** | 所有自检项均通过（或仅 minor 建议，无阻塞问题） | 可进入 coding-implement 阶段 |
| **有条件通过** | 存在有条件通过项，但无阻塞性问题 | 记录优化项，可并行进入编码阶段，但须在迭代中完成优化 |
| **不通过** | 存在不通过项 | 必须修改技术方案，重新评审 |

### 4.6 评审维度参考框架

评审须围绕以下底层逻辑展开：

#### 架构层

| 维度 | 评审要点 |
|-----|---------|
| **架构合理性** | 总体架构设计是否合理？模块/层次划分是否清晰？边界与职责是否明确？ |
| **可扩展性** | 架构是否支持未来扩展？新增功能是否需要大规模重构？ |
| **技术选型 ROI** | 技术栈选型是否合理？是否引入不必要的复杂度？ |

#### 实现层

| 维度 | 评审要点 |
|-----|---------|
| **需求满足度** | 技术方案是否 100% 满足 PRD 期望实现需求和目标？ |
| **接口设计清晰度** | 接口契约是否清晰？数据模型是否合理？与现有系统兼容性如何？ |
| **流程完整性** | 关键业务流程是否有清晰描述？状态流转是否完整？ |
| **异常处理完备性** | 异常处理策略是否完备？安全与性能要点是否覆盖？ |

#### 落地层

| 维度 | 评审要点 |
|-----|---------|
| **可实现性** | 技术方案是否可被前端/后端按图实现？实现路径是否清晰？ |
| **可验证性** | 技术方案是否可被 code-review/func-test 对照验证？ |
| **可追溯性** | 与 PRD、specs 的追溯关系是否明确？ |

### 4.7 与迭代日志的区别

| 对比 | 迭代日志 | 技术方案评审纪要 |
|-----|---------|-----------------|
| **定位** | 谁在什么时间做了什么事（日志型） | 评审了什么、结论是什么、为什么（评审型） |
| **内容** | 执行方、技能、change-id、产出物路径 | 自检项、评审发现、判定结论、理由、建议、依据 |
| **粒度** | 单次技能执行 | 技术方案自检的完整过程 |
| **存放** | `docs/项目事件日志.md`（项目级单一文件） | `docs/project-prd-changes/[change-id]/records/技术方案-[change-id]-评审纪要.md`（按变更归档） |

## 五、与技能流程的对应

- **project-analysis 执行流程**：读取 request-analysis 产出与 project.md、project-rules → 评估是否需更新 project.md / project-rules → **产出或更新** `openspec/changes/[change-id]/design.md` 与 project-rules → **按本节「一、最小结构」与「三、自检清单」执行并自检** → 将结论反馈到 proposal、specs、tasks。
- **产出物路径**：技术方案正文**仅** `openspec/changes/[change-id]/design.md`（见「二」）。
- **architecture-review 技能触发**：当用户输入「评审技术方案」「检查架构设计」「技术方案自检」等指令时，触发 **architecture-review** 技能，按本节「四、评审纪要输出」规范执行评审并产出评审纪要。

## 六、设计产出物（架构图、执行逻辑图、数据流图）补充约定

当变更涉及**多模块、多服务、复杂数据流或状态机**时，仅文字描述易产生歧义，建议将下列产出物作为 **技术方案的配套**，与 design.md、project-rules 一并使用，便于前端/后端按图实现与 code-review 对照。

| 类型 | 说明 | 典型用途 |
|------|------|----------|
| **技术架构图** | 系统/子系统/模块层次、边界、依赖与部署单元。 | 新服务/新模块引入、跨模块边界划分、与 project.md 架构模式对齐。 |
| **执行逻辑图 / 流程图** | 关键业务流程、状态流转、分支与异常路径。 | 订单/支付/审批等流程；与 PRD 场景描述、specs Scenario 对应。 |
| **数据流图** | 数据在模块/服务间的输入输出与转换。 | 与 project-rules 数据模型、接口契约一致；ETL 或跨服务数据流。 |
| **序列图 / 时序图** | 跨模块/服务调用顺序与消息交互。 | 接口设计、事件驱动、与 specs 中交互场景对应。 |

- **存放**：`docs/project-prd-changes/[change-id]/architecture/` 或 `project-rules/` 下；文件命名建议含主题（如 `system-architecture.md`、`order-flow.mmd`、`data-flow.svg`）。
- **在 design.md / project-rules 中引用**：在对应章节写明「架构见 `architecture/xxx`」「执行逻辑见 `architecture/yyy`」，便于实现与评审按图核对。
- **与 image-analysis**：用户提供已有架构图/流程图时，可经 image-analysis 解析后写入上述文档并保存原图，建立「图 → 文档」可追溯。
