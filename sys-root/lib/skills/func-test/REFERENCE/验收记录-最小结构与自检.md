# 验收记录 — 最小结构与自检

> 测试 Agent 执行 func-test 时，产出的 **功能验收记录**（`[change-id]-func-test.md`）须符合本结构，并完成自检，使产出可被主 Agent/开发对照、可追溯、可支撑「是否通过验收」结论。  
> 记录文件约定详见同目录 `spec-func-test-openspec-integration.md`；本节约定**最小结构**与**产出后自检清单**。

---

## 一、最小结构（必含）

产出 **`docs/project-prd-changes/[change-id]/records/[change-id]-func-test.md`** 时，须覆盖以下 5 类内容：

| # | 章节 | 内容要点 |
|---|------|----------|
| 1 | **基本信息** | change-id；测试日期、环境、测试者。 |
| 2 | **范围说明** | 本次测试覆盖/不覆盖哪些模块或场景；与 specs 中 Requirements/Scenarios 或需求验收 Checklist 的对应关系。 |
| 3 | **用例列表与结果汇总** | 用例列表（可引用 Requirement/Scenario）；每项结果：通过/未通过/需人工判定；通过/未通过统计。 |
| 4 | **问题与 bug 列表** | 未通过项与测试中发现的问题；严重级别与位置；建议修复或回归方式。 |
| 5 | **结论与建议** | 是否推荐通过本次验收；OpenSpec 两轮 validate 结果（第一轮 `openspec validate [change-id]`、第二轮 `openspec validate --strict`）；是否需要补充测试或回归。 |

- **OpenSpec 规定动作**：第一轮在测试前或测试中执行 `openspec validate [change-id]`；第二轮在输出验收记录后执行 `openspec validate --strict`，两轮结果须记入验收记录。
- **与 specs/Checklist 的对应**：用例与结果须与 `specs/*/spec.md` 的 Requirements + Scenarios 及 `docs/project-prd-changes/[change-id]/` 下需求验收 Checklist 可对应。

---

## 二、自检清单（产出后必过）

执行 func-test 并输出验收记录后，按下列项自检；不满足时补全再给出验收结论。

| # | 自检项 | 说明 |
|---|--------|------|
| 1 | **绑定 change-id** | 记录是否明确绑定单一 change-id，且路径为 `docs/project-prd-changes/[change-id]/records/`？ |
| 2 | **五类结构完整** | 是否包含基本信息、范围说明、用例与结果汇总、问题与 bug 列表、结论与建议？ |
| 3 | **两轮 validate 已执行并记录** | 是否已执行第一轮 `openspec validate [change-id]` 与第二轮 `openspec validate --strict`，并将结果写入记录？ |
| 4 | **用例与 spec/Checklist 对应** | 用例是否与 specs 中 Requirements/Scenarios 或需求验收 Checklist 可对应？结果是否可追溯？ |
| 5 | **问题进 tasks** | 功能错误或高优先级问题是否已转化为 `tasks.md` 任务项并可引用本记录？ |
| 6 | **结论明确** | 是否给出「是否推荐通过本次验收」及理由？ |

---

## 三、与技能流程的对应

- **func-test 步骤 6**：输出验收记录时，须采用本节「一、最小结构」；产出后执行「二、自检清单」，通过后再给出是否推荐通过验收的结论。
- **与 code-review**：验收记录与评审记录同目录，便于按变更综合决策与归档。
