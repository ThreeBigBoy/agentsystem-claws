# 通用 Code Review 规范

本规范用于指导在 OpenSpec 项目中进行 Code Review，不绑定具体语言或框架，重点关注：

- 是否满足需求与场景（Requirements + Scenarios）；
- 是否遵守项目架构与分层约定；
- 代码质量、安全性、性能与可维护性；
- 文档与实现的一致性。

---

## 一、Review 前准备

- 明确本次 Review 的 `change-id` 与范围（文件、模块、提交集等）；
- 阅读与本次变更相关的：
  - `openspec/changes/[change-id]/proposal.md`、`design.md`、`specs/*/spec.md`、`tasks.md`；
  - `openspec/project.md` 与 `project-rules/`；
  - 如有 `info-database/`、`info-service-interface/` 相关文档，一并参考。

---

## 二、Review 维度清单

建议从以下维度系统化检查代码（可按项目需要增减）：

1. **需求符合性与场景覆盖**
   - 实现是否覆盖所有声明的 Scenario（正常路径、边界情况、异常场景等）；
   - 是否存在未文档化的隐含行为或超出本次变更范围的改动。

2. **架构与分层**
   - 代码是否落在正确的层（例如表现层不包含业务决策，业务层不直接访问存储等）；
   - 是否遵守后端分层总纲与前后端分层规范中的依赖方向约束；
   - 是否出现层与层之间的循环依赖或越层调用。

3. **代码质量与可维护性**
   - 命名是否清晰表达意图（类/函数/变量/文件）；
   - 函数/类是否过长或职责过多；
   - 是否存在明显重复代码，可以抽取成复用模块。

4. **安全性与鲁棒性**
   - 对外部输入是否进行了必要校验与过滤；
   - 是否正确处理错误与异常（例如区分可预期业务错误与系统错误）；
   - 是否存在潜在的安全风险（如注入、越权、敏感信息泄露等）。

5. **性能与资源使用（视项目关注度）**
   - 关键路径上是否存在明显的低效实现（多次重复查询、大量无缓存 IO 等）；
   - 是否有不必要的阻塞调用或同步等待。

6. **日志、监控与可观测性**
   - 关键操作是否有足够的日志与指标埋点；
   - 日志是否遵守项目日志规范（级别、字段、敏感信息处理等）。

7. **测试与可测试性**
   - 是否为关键路径或复杂逻辑编写了合理的单元/集成测试；
   - 代码结构是否便于测试（依赖注入、分层清晰、避免隐藏状态）。

---

## 三、问题记录与优先级

- 对每个发现的问题，建议记录以下信息：
  - **位置**：文件、函数/类名、行附近描述；
  - **问题类型**：需求偏差 / 架构越层 / 代码质量 / 安全 / 性能 / 可观测性 / 测试等；
  - **严重级别**：Blocking / Major / Minor；
  - **建议**：尽量给出具体、可执行的改进方案。

- 在输出到 `docs/project-prd-changes/[change-id]/records/[change-id]-code-review.md` 时，可按优先级排序问题，便于集中处理。

---

## 四、与 OpenSpec 文档的一致性

- 检查代码是否与以下文档保持一致：
  - `specs/*/spec.md` 中的 Requirements + Scenarios；
  - `project.md` 与 `project-rules/` 中的架构/命名/数据/接口约定；
  - `info-database/` 与 `info-service-interface/` 中的数据结构与接口说明。

- 如发现不一致，应判断：
  - 优先修正代码以符合已达成的规范；
  - 如确有必要调整规范，应：
    - 先与相关方达成共识；
    - 再更新规范文档，并在 review 记录中写明原因。
