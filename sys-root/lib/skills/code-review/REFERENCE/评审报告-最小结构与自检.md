# 评审报告 — 最小结构与自检

> 架构 Agent 执行 code-review 时，产出的 **Code Review 记录**（`[change-id]-code-review.md`）须符合本结构，并完成自检，使产出可被开发按项整改、可被主 Agent/测试对照、可追溯。  
> 记录文件约定详见同目录 `spec-code-review-openspec-integration.md`；本节约定**最小结构**与**产出后自检清单**。

---

## 一、最小结构（必含）

产出 **`docs/project-prd-changes/[change-id]/records/[change-id]-code-review.md`** 时，须覆盖以下 4 类内容：

| # | 章节 | 内容要点 |
|---|------|----------|
| 1 | **基本信息** | change-id；评审日期与轮次（如 First/Second Round）；reviewer 与参与者；涉及模块/文件列表。 |
| 2 | **整体结论** | 通过 / 需修改 / 需重大调整；关键风险点或注意事项。 |
| 3 | **问题清单** | 按优先级（Blocking / Major / Minor）列出问题；每项含：位置（文件/函数或模块）、描述、类型与建议方案；可与 specs 中 Requirement/Scenario 对应。 |
| 4 | **后续行动** | 需补充到 `tasks.md` 的任务项（Blocking/Major 须写入）；是否需要追加变更提案或更新规范文档。 |

- **与 specs 的对应**：问题描述中可引用 `specs/*/spec.md` 中的 Requirement ID 或 Scenario，便于开发按 spec 整改。
- **与 tasks 的联动**：Blocking/Major 级问题须在 `openspec/changes/[change-id]/tasks.md` 中新增或更新任务项，任务描述中可引用本记录中的问题编号。

---

## 二、自检清单（产出后必过）

执行 code-review 并输出评审记录后，按下列项自检；不满足时补全再视为完成。

| # | 自检项 | 说明 |
|---|--------|------|
| 1 | **绑定 change-id** | 记录是否明确绑定单一 change-id，且路径为 `docs/project-prd-changes/[change-id]/records/`？ |
| 2 | **四类结构完整** | 是否包含基本信息、整体结论、问题清单、后续行动？ |
| 3 | **问题分级** | 问题是否按 Blocking / Major / Minor 分级？每项是否有位置、描述与建议？ |
| 4 | **与 specs 可追溯** | 涉及需求符合性时，是否引用 specs 中 Requirement 或 Scenario？ |
| 5 | **Blocking/Major 进 tasks** | Blocking/Major 级问题是否已同步到 `tasks.md` 并可在任务中引用本记录？ |
| 6 | **结论可操作** | 整体结论与后续行动是否具体、可执行（非空泛评价）？ |

---

## 三、与技能流程的对应

- **code-review 步骤 5**：输出 review 记录时，须采用本节「一、最小结构」；产出后执行「二、自检清单」，通过后再视为本次评审完成。
- **与 func-test**：评审记录与验收记录同目录（`docs/project-prd-changes/[change-id]/records/`），便于按变更归档与综合决策。
