# 复盘报告模板

> 本模板为 `retrospective-analysis` 技能的配套 REFERENCE，提供复盘报告的标准结构和编写指南。
> 
> 根据复盘类型（问题诊断型、经验沉淀型、流程优化型、学习成长型、项目结项型），可适当调整内容侧重。

---

## 报告头信息

```markdown
# 复盘报告：[主题/事件名称]

**复盘时间**: YYYY-MM-DD  
**复盘范围**: YYYY-MM-DD 至 YYYY-MM-DD（最近 N 天 / 某个 change-id / 某个项目）  
**核心 change-id**: [change-id]（如适用）  
**复盘类型**: [问题诊断型 / 经验沉淀型 / 流程优化型 / 学习成长型 / 项目结项型]  
**复盘人**: [Agent 角色，如 主 Agent / 架构 Agent / 产品经理 Agent]  
**复盘状态**: [进行中 / 完成]
```

## 一、复盘背景与目标

### 1.1 复盘触发原因

**说明**: 为什么进行这次复盘？是什么触发了复盘？

**示例**:
```markdown
本次复盘由 [事件/问题/里程碑] 触发：

- [如果是问题]: 出现了什么问题？影响范围？紧急程度？
  示例: "check-langgraph-backend 自检脚本出现 POST /run 超时问题，
         导致无法验证 LangGraph 后端健康状态"

- [如果是里程碑]: 完成了什么重要工作？为什么重要？
  示例: "完成了 LangGraph 后端阶段化执行架构的重构，
         需要复盘整个过程，提炼经验"

- [如果是定期复盘]: 周期性的回顾与总结
  示例: "每月一次的工作复盘，回顾本月的工作成果与改进空间"
```

### 1.2 复盘目标

**说明**: 希望通过这次复盘达成什么目标？

**示例**:
```markdown
1. **问题诊断**（如适用）: 找出问题的根本原因，提出系统性解决方案
2. **经验沉淀**: 将本次工作的经验提炼为可复用的模式，沉淀为 memory
3. **流程优化**: 识别流程中的改进点，提出优化建议
4. **团队学习**: 让团队成员从本次工作中学习，提升能力
```

### 1.3 复盘范围与边界

**说明**: 复盘覆盖哪些内容？不覆盖哪些内容？

**示例**:
```markdown
**包含范围**:
- check-langgraph-backend change-id 的完整工作过程
- 从问题发现到规范升级的 2 天工作内容
- 相关的 PRD、技术方案、代码实现、评审过程

**不包含范围**:
- 其他 change-id 的工作（除非有关联）
- 尚未完成的工作（如第二阶段架构重构）
- 个人工作习惯（聚焦团队协作与流程）
```

## 四、分析原因（阶段 3）

### 4.1 关键问题根因分析（5 个 Why）

**说明**: 对关键问题执行 5 个 Why 分析，找到根本原因

**示例**:
```markdown
**问题**: 为什么自检脚本出现超时？

Why 1: 为什么超时？
→ 执行耗时 7~10 分钟，超过了脚本 120s 超时

Why 2: 为什么执行这么久？
→ 一次性执行完整的 check-langgraph-backend change-id，包含 8 个任务

Why 3: 为什么必须一次性执行？
→ /run API 设计假设：单次调用完成完整 workflow

Why 4: 为什么这样设计？
→ 早期需求简单，没有考虑长时间运行的场景

Why 5: 能否改变设计？
→ 可以，阶段化执行：/run(change_id, phase="env-check")

**根本原因**: 
架构设计假设（一次性全量执行）与实际场景（长时间运行）不匹配

**解决方案**: 
引入阶段化执行模式，将大事务拆成可观测的小步骤
```

### 4.2 问题分类分析

**说明**: 将问题分类，区分内因/外因、可控/不可控

**示例**:
```markdown
| 问题 | 内因/外因 | 可控/不可控 | 系统性/偶然 | 处理策略 |
|-----|----------|-----------|------------|---------|
| 超时问题 | 内因（架构设计） | 可控 | 系统性 | 架构重构 |
| 命名拼写错误 | 内因（疏忽） | 可控 | 偶然 | 修复+规范检查 |
| 用户突破性思维 | 外因（用户） | 不可控 | - | 学习+沉淀 |
| 评审时间长 | 内因（流程） | 可控 | 系统性 | 优化流程/工具 |
```

### 4.3 思维模式差异分析（如适用）

**说明**: 分析 AI vs 用户（或不同角色）的思维差异

**示例**:
```markdown
| 维度 | AI 思维模式 | 用户思维模式 | 差异影响 |
|-----|-------------|--------------|---------|
| **问题定义** | 超时时间不够长 | 大事务不可分割 | 解决思路完全不同 |
| **解决层次** | 工具层（调参） | 框架层（架构重构） | 治标 vs 治本 |
| **思考深度** | 当前约束下优化 | 重新定义约束 | 渐进 vs 突破 |
| **关注焦点** | 如何让当前方案 work | 如何设计更健康的方案 | 短期 vs 长期 |

**学习启示**:
- 面对问题时，先问「为什么」而不是「如何解决」
- 区分本质约束和偶然约束，敢于突破偶然约束
- 从「解决问题」升维到「重新定义问题空间」
```

## 使用说明

### 如何使用本模板

1. **复制模板**: 复制本模板内容到新的复盘报告
2. **填空内容**: 根据实际复盘内容，填充各章节
3. **适当裁剪**: 根据复盘类型，裁剪不必要的章节
4. **保持结构**: 保持 5 阶段结构（目标→结果→原因→经验→行动）
5. **质量自检**: 完成后按「复盘质量自检（9 项）」检查

### 不同复盘类型的内容侧重

| 复盘类型 | 侧重章节 | 可裁剪章节 |
|---------|---------|-----------|
| **问题诊断型** | 阶段 3（分析原因） | 阶段 5 可简化 |
| **经验沉淀型** | 阶段 4（提炼经验） | 阶段 3 可简化 |
| **流程优化型** | 阶段 4、5（经验、行动） | 阶段 2 可简化 |
| **学习成长型** | 阶段 4（提炼经验） | 阶段 3 可简化 |
| **项目结项型** | 全章节 | 不裁剪 |

### 复盘报告质量门禁

- **通过**: 7 项及以上通过，无不通过项
- **有条件通过**: 5-6 项通过，无不通过项
- **不通过**: 少于 5 项通过，或有不通过项

## 复盘报告存放路径规范（分布式管理架构）

### ⚠️ 核心原则: 数据就近存储（Data Locality）

- **框架级复盘** → 存储在 **agentsystem**（框架仓库）
- **项目级/change-id 级复盘** → 存储在 **对应的业务项目中**
- agentsystem 不替业务项目管理其复盘

### 复盘类型与管理责任

| 复盘类型 | 管理仓库 | 存放路径 | 说明 |
|---------|---------|---------|------|
| **框架级复盘** | agentsystem | `agentsystem 内框架级复盘（路径以仓库实际为准，如 docs/retrospective/framework/）[YYYY-QN]/复盘-[主题]-YYYY-MM-DD.md` | 框架能力成长复盘 |
| **项目级复盘** | 业务项目 | `[业务项目]/docs/retrospective/[YYYY-MM]/复盘-[主题]-YYYY-MM-DD.md` | 业务项目交付复盘 |
| **change-id 级复盘** | 业务项目 | `[业务项目]/docs/project-prd-changes/[change-id]/records/复盘报告-YYYY-MM-DD-[主题].md` | 单个变更详细复盘 |

### 路径选择决策树

```
复盘主题是什么？
    │
    ├─ agentsystem 能力成长（框架、流程、技能、Memory 改进）
    │   → 框架级复盘
    │   → 管理方: agentsystem
    │   → 路径: agentsystem 内框架级复盘（路径以仓库实际为准，如 docs/retrospective/framework/）[YYYY-QN]/
    │
    └─ 业务项目交付（Shopify Theme、其他项目等）
        │
        ├─ 整体项目回顾（跨 change-id）
        │   → 项目级复盘
        │   → 管理方: 业务项目
        │   → 路径: [业务项目]/docs/retrospective/[YYYY-MM]/
        │
        └─ 单个 change-id 详细分析
            → change-id 级复盘
            → 管理方: 各项目自身
            → 路径: 
                - 业务项目: [业务项目]/docs/project-prd-changes/[change-id]/records/
                - agentsystem: agentsystem 内变更记录（路径以仓库实际为准，如 docs/project-prd-changes/）[change-id]/records/
```
