# 目录结构规范 V3（基于完整质量闭环）

> 本规范定义基于 8+1 完整质量闭环流程的目录结构，覆盖从需求分析到复盘的全生命周期文档管理。
> 
> **适用对象**:
> - agentsystem（框架仓库）
> - 所有业务项目（如 Proj01ShopifyTheme）

## agentsystem 目录结构（框架仓库）

> **说明**：框架仓库**不**采用业务侧已废弃的「并行需求目录树」；规范与技能正文在 **`usr-devclaw/usr-rules/`** 与 **`sys-root/lib/`**。

```
agentsystem/
├── usr-devclaw/usr-rules/              # usr-rules：OpenSpec.md、PROJECT-CONFIG.md 等
├── docs/                               # 仓库级说明文档（如 AGENTS 职责区分）
├── sys-root/lib/
│   ├── skills/                         # 可复用技能（每技能 SKILL.md + REFERENCE/）
│   └── memory/                         # patterns / playbooks / preferences / …
├── workspace/                          # 业务项目 claw-config 软链挂载（按项目配置）
└── README.md
```

> 若 agentsystem **自身**需要记录框架级复盘或变更材料，路径以仓库内 **`docs/retrospective/`**（或团队约定目录）为准。

## 关键设计原则

### 1. PRD 与过程记录在 `docs/project-prd-changes/`，技术方案正文在 `openspec/changes/`

- **PRD** 与 **records/**（评审纪要、代码评审、验收、复盘等）在 **`docs/project-prd-changes/[change-id]/`**。
- **技术方案正文**仅在 **`openspec/changes/[change-id]/design.md`**，**不要**在 `docs/project-prd-changes/` 再维护一份「技术方案-*.md」。

### 2. 清晰的层级关系（示意）

```
docs/project-prd-changes/
└── [change-id]/
    ├── PRD-xxx.md
    └── records/                 # 评审纪要、验收、change-id 级复盘等

openspec/changes/
└── [change-id]/
    ├── proposal.md
    ├── tasks.md
    ├── design.md                # 技术方案正文（唯一）
    └── specs/...
```

### 3. 区分需求侧核心文档 vs OpenSpec 变更 vs 过程记录

- **需求侧核心**：`docs/project-prd-changes/[change-id]/` 下 PRD 等。
- **变更侧技术方案**：**仅** `openspec/changes/[change-id]/design.md`。
- **过程记录**：`docs/project-prd-changes/[change-id]/records/`。

### 4. 复盘 vs 执行偏差记录：术语严格区分

**重要**："复盘"和"执行偏差记录"是两个不同的术语，不可混用。

| 术语 | 定义 | 存放位置 | 产出时机 | 文档结构 |
|------|------|---------|---------|---------|
| **执行偏差记录** (Execution Deviation) | 执行过程中发现问题/偏差时的**即时记录** | `runtime-logs/execution-deviations/` | 问题发现时立即记录 | 自由格式，重在快速记录事实 |
| **复盘** (Retrospective) | 变更/阶段结束后的**系统性总结** | `docs/retrospective/[level]/` | 变更归档后 | 必须含：目标→过程→根因→改进→模式沉淀 |

**关键区别**：
- 执行偏差记录 = 运行时审计，问"发现了什么偏差？"（即时事实记录）
- 复盘 = 方法论提炼，问"我们做得怎么样？为什么？下次如何更好？"（事后系统总结）

**禁止的做法**：
- 把 runtime-logs/execution-deviations/ 下的文档称为"复盘"
- 试图"先写执行偏差记录草稿，再迁移到 retrospectives/"——复盘一次性产出到正确位置，不需要草稿迁移

### 5. 复盘 3 级分离

| 复盘级别 | 存放位置 | 关注焦点 |
|---------|---------|---------|
| **change-id级** | `changes/[change-id]/records/` | 单个 change-id 执行过程 |
| **项目级** | `[项目]/retrospectives/[YYYY-MM]/` | 项目整体交付（跨 change-id）|
| **框架级** | `retrospectives/framework/[YYYY-QN]/` | 框架能力成长（跨项目模式提炼）|

## 与 OpenSpec 的对应关系

| OpenSpec 结构 | 需求侧 / 说明 |
|--------------|----------------|
| `openspec/changes/[change-id]/proposal.md` | 与 `docs/project-prd-changes/[change-id]/` 下 PRD **互相引用** |
| `openspec/changes/[change-id]/tasks.md` | 任务清单，与 PRD、design 对齐 |
| **`openspec/changes/[change-id]/design.md`** | **技术方案正文唯一载体**（不在 `docs/project-prd-changes/` 重复一份技术方案正文） |
| `openspec/changes/[change-id]/specs/` | 与归档后 `openspec/specs/` 对齐 |

## 实施建议
