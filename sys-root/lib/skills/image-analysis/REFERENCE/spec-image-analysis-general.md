# 通用图片分析规范

本规范用于指导对用户提供的图片进行**结构化解析**，输出可被需求、设计与实现环节复用的文本描述，适用于各类应用工程。

---
## 一、图片类型与解析侧重

| 图片类型 | 主要解析维度 | 输出用途 |
|----------|--------------|----------|
| 界面设计稿/原型图 | 布局结构、组件与控件、文案、视觉规范（颜色/字体/间距）、交互入口 | 需求说明、spec Scenario、前端实现与验收 |
| 界面截图 | 当前可见的布局、文案、状态、控件位置与样式 | 需求澄清、问题复现、验收对比 |
| 架构图/系统图 | 节点（服务/模块/系统）、连线与关系、层次与边界 | project.md、project-rules/、design.md |
| 数据流图/流程图 | 节点、流向、数据/操作标注、分支与汇聚 | project-rules/ 信息流与数据流、spec 流程类 Scenario |
| 其他示意图 | 根据内容提取关键实体、关系与标注文案 | 按需写入需求或设计文档 |

---
## 二、界面类图片解析要点

- **布局与层级**
  - 描述整体结构（如：顶部导航、主内容区、底部栏）；
  - 重要区块的上下级与占位关系（可辅以简要层级列表或缩进描述）。

- **组件与控件**
  - 可交互元素：按钮、链接、输入框、选择器、标签页等；
  - 展示型组件：标题、正文、图片、列表、卡片、表格等；
  - 对关键控件注明类型与大致位置（如「主操作按钮位于右下角」）。

- **文案与数据**
  - 图中可见的标题、说明、占位文案、示例数据；
  - 若为多语言或动态内容，标注「占位/示例」及建议的数据来源（如接口字段、配置项）。

- **状态与交互**
  - 若图代表某一状态（如加载中、空状态、错误态），明确标注；
  - 图中可见的交互线索（如箭头、标注、说明文字）整理为「预期交互行为」列表。

- **视觉规范（可选）**
  - 可识别的颜色、字体层级、间距规律，以「设计约定」形式简要列出，便于实现时对齐。

---
## 三、架构/流程图解析要点

- **节点**
  - 每个节点代表的实体（系统、服务、模块、存储、外部方）；
  - 节点类型或角色（如 API、DB、Queue、第三方）。

- **关系与流向**
  - 连线的方向与含义（调用、数据流、事件等）；
  - 标注在连线上的文字或符号含义。

- **层次与边界**
  - 分组、泳道、虚线框等表达的边界（如系统边界、部署边界）；
  - 层级关系（如前端 / 网关 / 服务 / 数据）。

- **输出形式**
  - 建议输出为「节点列表 + 关系列表」或「层级结构 + 关键数据流/调用链」的文本描述，便于写入 project-rules 或 design.md。

---
## 四、解析结果的组织形式

- **客观与推断分离**
  - **客观描述**：图中直接可见的内容（布局、文案、节点、连线等）；
  - **推断或建议**：需要结合业务或上下文得出的结论，应单独标注「推断」或「需确认」。

- **结构化输出**
  - 使用标题、列表、表格等使下游可直接引用；
  - 若有多张图，按「图标识 + 内容」分段，避免混淆。

- **歧义与缺失**
  - 对模糊、不清晰或图中未体现的部分，明确写出「图中未体现」「需与设计/产品确认」，不擅自假设。

---
## 五、与项目语境的结合

- 解析前若已读取 `openspec/project.md` 或 `project-rules/` 中的术语、模块命名、技术栈，应在描述中尽量使用一致用语。
- 解析结果用于需求或 spec 时，应满足「可验收」：即实现与测试能根据该描述判断是否达标。
