# 后端分层整体架构总纲

本总纲在 OpenSpec 语境下，对后端各层的职责边界与依赖方向做统一说明，便于在 `coding-implement` 技能下进行一致的编码实现与评审。

涵盖的层次：

- 表现层（Presentation）：Controller / API Handler / 接口入口；
- 应用层（Application）：用例编排与事务边界；
- 业务层（Business/Domain）：核心业务规则与领域模型；
- 服务层（Service）：可复用的能力中心（结算、风控、推荐等）；
- 数据层（Data/Repository）：数据访问与持久化；
- 适配层（Adapter/Gateway）：对接外部系统与平台；
- 基础设施层（Infrastructure）：日志、监控、配置、缓存、消息等技术底座。

---

## 一、依赖与调用方向（自上而下）

从「请求进入系统」的视角，推荐的调用/依赖方向为：

1. **表现层 → 应用层**
   - 表现层接收外部请求（HTTP/RPC/消息），完成参数解析与基础校验后，将语义化请求交给应用层；
   - 表现层依赖应用层的用例服务接口，不直接访问业务层、数据层或适配层。

2. **应用层 → 业务层 / 服务层 / 数据层 / 适配层**
   - 应用层负责用例编排：按照流程调用业务层、服务层、仓储与适配层；
   - 理想情况下，应用层更多依赖业务层与服务层，由业务层/服务层间接使用数据层与适配层；
   - 仅在简单场景或约定允许的情况下，应用层可以直接访问数据层（例如查询只读视图）。

3. **业务层 ↔ 服务层**
   - 业务层以领域模型与规则为中心，可调用服务层复用跨用例能力；
   - 服务层也可封装部分复杂业务组合逻辑，供多个业务模块与用例调用；
   - 二者之间的依赖应通过清晰接口管理，避免循环依赖与过深耦合。

4. **业务层/服务层 → 数据层 / 适配层**
   - 当需要持久化或读取数据时，通过仓储接口访问数据层；
   - 当需要与外部系统交互时，通过适配层/Gateway 接口发起调用；
   - 业务层和服务层不直接依赖具体 ORM/HTTP/MQ 客户端实现。

5. **所有上层 → 基础设施层**
   - 通过统一的日志/监控/缓存/消息等接口使用基础设施能力；
   - 不直接依赖具体基础设施产品或 SDK，便于替换实现与统一治理。

## 二、各层职责速览

- **表现层（Presentation）**
  - 入口：Controller、API Handler、消息消费入口；
  - 只做协议适配、基础校验、响应封装与请求级日志；
  - 依赖：应用层接口、基础设施层（日志、监控、鉴权结果）。

- **应用层（Application）**
  - 负责用例编排与事务边界：按步骤调用业务层、服务层、数据层与适配层；
  - 聚焦「一次用户/系统操作」从请求到结果的完整流程；
  - 依赖：业务层、服务层、数据层（仓储）、适配层、基础设施层。

- **业务层（Business/Domain）**
  - 承载核心业务规则与领域模型，不直接操作基础设施；
  - 决定在给定场景下系统「应该如何行为」；
  - 依赖：仓储接口（数据层抽象）、服务层能力接口、基础设施层（日志/指标）。

- **服务层（Service）**
  - 提供跨用例、跨模块可复用的能力（如定价、风控、推荐等）；
  - 侧重能力抽象与复用，而非单个用例的完整流程；
  - 依赖：业务层（可选）、数据层、适配层、基础设施层。

- **数据层（Data/Repository）**
  - 与数据库/缓存/NoSQL 交互，将存储模型与领域/DTO 映射；
  - 提供语义化数据访问接口，不实现业务决策；
  - 依赖：基础设施层（连接池、驱动、缓存客户端等），与 `info-database/`、数据模型文档保持一致。

- **适配层（Adapter/Gateway）**
  - 与第三方系统、平台 API、消息系统等交互，提供防腐层；
  - 对上暴露稳定接口，对下封装外部 SDK/协议；
  - 依赖：基础设施层（HTTP 客户端、MQ 客户端、配置、日志/监控等）。

- **基础设施层（Infrastructure）**
  - 提供日志、监控、配置、缓存、消息、任务调度等通用能力；
  - 被所有上层通过统一接口使用，不包含业务规则。

## 三、常见反模式与约束

- **表现层直接访问数据层或适配层**：
  - 破坏用例编排在应用层的集中控制，导致逻辑分散；
  - 约束：表现层只能调用应用层服务，不直接访问仓储或外部系统。

- **业务规则下沉到数据层或适配层**：
  - 使业务逻辑与存储/外部系统耦合，难以测试与演进；
  - 约束：数据层与适配层只做数据/协议转换与 IO，不作业务决策。

- **事务与重试逻辑散落在多层**：
  - 导致事务边界不清与行为难以推断；
  - 约束：应用层负责事务边界；重试与幂等策略在设计中有统一约定，并在适配层/基础设施层实现。

- **绕过基础设施层直接使用底层组件**：
  - 导致日志/监控/缓存等能力碎片化、难以统一治理；
  - 约束：所有层通过基础设施层提供的统一接口访问这些能力。

## 四、与 OpenSpec 与 coding-implement 的衔接

- 在使用 `coding-implement` 进行编码时，应先根据具体实现内容：
  - 在本总纲中确认应落在哪一层；
  - 再结合对应层的 `spec-backend-*.md` 细则进行实现。

- 在 `project-rules/` 中，可依据本总纲绘制项目实际采用的分层架构图与依赖关系，并在需要时对本总纲做适度裁剪（如合并某些层）。

- 在评审代码或排查问题时，可用本总纲作为「是否越层」「是否职责不清」的检查清单。
