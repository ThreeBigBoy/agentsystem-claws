# 后端应用层编码规范

应用层负责「用例编排」：接收上层请求，协调业务层与基础设施，完成一个完整的应用场景（Use Case），并将结果返回给表现层或调用方。

---

## 一、职责与边界

- **负责用例流程，不实现细粒度业务规则**
  - 例如：下单流程中，应用层负责「验证请求 → 调用领域服务计算价格 → 请求库存锁定 → 持久化订单 → 发送通知」的顺序；
  - 具体的价格计算、库存判断等规则放在业务层或领域服务中。

- **协调依赖而非拥有逻辑**
  - 通过依赖注入持有业务服务接口、仓储接口、外部服务客户端等；
  - 尽量避免在应用层写大量 if/else 描述业务细节。

---

## 二、接口设计

- **以用例为单位设计 Application Service**
  - 如：`PlaceOrderService`、`RegisterUserService`、`GenerateReportService`；
  - 每个服务方法对应一个完整的用例而非简单 CRUD。

- **输入输出契约**
  - 输入应为清晰的请求 DTO，包含验证后的业务必需字段；
  - 输出为结果 DTO，尽量避免直接返回领域实体。

---

## 三、事务与一致性

- **应用层负责事务边界**
  - 在一个用例流程中，明确事务开始与结束位置；
  - 避免在业务层或数据层散落事务控制代码；
  - 如使用分布式事务或最终一致性机制，应在此层作出协调策略。

- **与 OpenSpec 对齐**
  - 对涉及事务和一致性的场景，应在 `specs/*/spec.md` 或 `design.md` 中明确说明；
  - 代码实现需要与这些说明保持一致。

---

## 四、错误与返回码

- **统一的错误包装**
  - 捕获业务层与基础设施的异常，在应用层转换为统一的错误模型（供表现层使用）；
  - 避免将内部实现细节（如具体库错误）直接暴露给上层。

---

## 五、测试与可维护性

- **集成/用例级测试**
  - 对关键应用服务应有集成测试，覆盖主干流程与关键异常分支；
  - 可使用仓储或外部服务的测试替身。
