# 后端数据层编码规范（仓储 / DAO）

数据层负责与持久化存储交互（数据库、缓存、NoSQL 等），并为上层提供语义化的数据访问接口。

---

## 一、职责与边界

- **与具体存储技术打交道**
  - 负责构建查询、执行 SQL/ORM 调用、处理连接与会话；
  - 不实现业务规则决策（如是否允许下单、是否满足优惠条件）。

- **对象映射**
  - 负责在持久化模型与领域模型/DTO 之间进行数据转换；
  - 避免在各层重复编写字段映射逻辑。

---

## 二、接口设计

- **仓储接口的语义化**
  - 方法应表达业务含义，如 `findPaidOrdersByUser`、`saveCart`，而不是暴露原始 SQL 片段；
  - 对通用查询（分页、排序等）使用统一参数结构。

---

## 三、性能与一致性

- **索引与查询优化**
  - 常用查询应有合适索引支持；
  - 避免在代码中隐式触发 N+1 查询等性能陷阱。

- **事务与锁**
  - 遵循项目事务边界约定，避免在数据层开启或管理跨用例的长事务；
  - 对竞争访问需有明确锁策略（行锁、悲观锁、乐观锁版本号等）。

---

## 四、错误处理与重试

- **错误分类**
  - 区分暂时性错误（如连接超时）与永久性错误（如约束冲突）；
  - 对可重试错误使用统一的重试策略，不在调用处各自实现。

---

## 五、与数据库文档的一致性

- **对齐 info-database 与数据模型文档**
  - 数据层的表/字段/索引使用情况应与 `info-database/` 中的表结构说明及 `project-rules/` 中的数据模型描述一致；
  - 如有变更，需同步更新相关文档与 DDL。
