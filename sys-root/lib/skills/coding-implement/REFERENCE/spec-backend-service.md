# 后端服务层编码规范（Service Layer）

服务层用于组织可复用的业务能力或跨模块逻辑，介于纯领域规则与应用层之间，可作为多个用例或系统组件的共享能力中心。

---

## 一、职责与边界

- **复用能力中心**
  - 将多个用例或模块共用的逻辑抽取到服务层，以减少重复实现；
  - 不承载完整用例编排，而是聚焦于某一类能力（如结算服务、风控服务、推荐服务等）。

- **与业务层/应用层关系**
  - 业务层可调用服务层，服务层也可在内部封装复杂的业务规则；
  - 应用层通过服务层复用能力，但不应过度依赖其内部实现细节。

---

## 二、接口设计

- **能力导向的接口**
  - 以能力命名服务与方法，如 `PricingService.calculatePrice`、`RiskService.evaluate`；
  - 输入输出尽量与领域模型或专用 DTO 对齐，避免泄露存储或外部系统细节。

---

## 三、可扩展性与演进

- **可配置与可扩展**
  - 对于策略性较强的逻辑（如规则评分、推荐算法），预留配置与扩展点；
  - 避免将可配置项硬编码在服务逻辑中。

- **与 OpenSpec 的映射**
  - 若服务层能力对应重要业务能力，应在 `specs/*/spec.md` 或项目级规范中进行描述；
  - 服务输出的关键指标可纳入监控与验收标准。
