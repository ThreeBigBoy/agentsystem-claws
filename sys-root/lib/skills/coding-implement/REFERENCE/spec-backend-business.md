# 后端业务层编码规范

业务层负责承载领域规则与核心业务决策，是系统行为的中心。其职责是「根据输入的业务请求与领域状态，按照业务规则给出决策与结果」，而非直接处理 IO、协议或持久化细节。

---

## 一、职责与边界

- **只关注业务语义，不直接操作基础设施**
  - 不直接执行数据库查询、HTTP 调用、缓存操作等；
  - 通过仓储接口、服务接口或应用层提供的抽象获取/变更数据。

- **以领域模型为中心**
  - 使用领域实体、值对象与领域服务描述业务概念与规则；
  - 避免在业务层中使用无语义的 Map / JSON 结构进行核心决策。

- **与 OpenSpec 的关系**
  - 业务规则应能映射到 `specs/*/spec.md` 中的 Requirements 与 Scenarios；
  - 当业务决策空间发生变化时，应同步更新对应的规范文档。

---

## 二、接口设计与命名

- **方法命名体现业务语义**
  - 如：`placeOrder`、`cancelSubscription`、`applyDiscountForCart`；
  - 避免使用 `process`、`handle` 等模糊动词。

- **输入输出模型清晰**
  - 输入参数应为业务对象或专门的请求 DTO，而非无结构的 Map；
  - 输出应清晰表达操作结果及必要的领域状态变化。

---

## 三、业务规则实现原则

- **单一职责与模块化**
  - 将复杂规则拆分为可复用的子规则或策略；
  - 对经常变化的规则（如折扣策略、准入规则）使用策略模式、规则引擎或配置驱动等方式。

- **显式的前置条件与后置条件**
  - 在方法开头校验关键前置条件，并在失败时返回明确的业务错误；
  - 在完成关键操作后，对不变量进行校验或断言。

- **错误处理与领域错误**
  - 区分技术错误（如网络超时）与领域错误（如余额不足、库存不足）；
  - 领域错误应使用具备业务含义的错误码或异常类型。

---

## 四、测试与可维护性

- **可测试性优先**
  - 业务层代码应在不依赖外部服务的情况下易于进行单元测试；
  - 依赖资源通过接口注入（依赖倒置），便于构造测试替身。

- **文档与代码同步**
  - 对重要业务流程，建议在 `docs/project-prd-changes/`（PRD/方案侧）或 `project-rules/` 中有对应的流程图或说明；
  - 当业务规则变更时，先更新规范与设计，再实现代码变更。
