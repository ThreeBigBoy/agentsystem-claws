# 前端编码实现规范（通用）

本规范用于指导在 OpenSpec 项目中进行前端编码实现（包括但不限于 Web 页面、前端组件、主题模板、前端脚本等），确保：

- 前端实现与 `openspec/changes/[change-id]/specs/*/spec.md` 中的 Requirements + Scenarios 一一对应；
- 符合项目级宪法 `openspec/project.md` 与工程补充约束 `project-rules/` 中的前端约定；
- 在结构上清晰区分视图层、状态管理、数据访问与跨切关注点（错误处理、日志、埋点等）。

---

## 一、分层与职责边界（前端视角）

在前端工程中，建议至少区分以下层次（具体命名根据项目可调整）：

- **视图层（View / UI）**
  - 负责纯粹的展示与基础交互，将状态通过 props / 输入参数接收，而不过多持有业务逻辑。
  - 典型形式：React/Vue 组件、Liquid 模板片段、UI Section、静态模版等。

- **容器层 / 页面层（Page / Container）**
  - 负责页面级状态管理、数据加载与视图组合。
  - 处理路由、权限校验结果映射成 UI 状态，但不直接编码底层数据访问（由数据层或 service 提供）。

- **状态管理层（State / Store）**
  - 管理跨组件共享状态（如全局 store、上下文等）。
  - 定义状态结构、变更 action、派发与订阅机制。

- **数据访问层（Client / API 封装）**
  - 封装与后端接口或平台 API 的交互逻辑（HTTP 请求、GraphQL 调用、SDK 调用等）。
  - 只暴露语义化方法（如 `fetchOrderList`），而非散落在组件中的原始请求调用。

- **跨切关注点（Cross-cutting Concerns）**
  - 包括：错误处理、全局 Loading、埋点/统计、A/B 实验、国际化、主题样式与设计系统等。
  - 建议通过统一的工具模块或高阶组件/钩子实现，避免在各处复制粘贴。

---

## 二、代码组织与命名约定

- **遵循项目级约定**
  - 目录结构、命名风格（kebab-case / camelCase / PascalCase）、文件后缀等，应首先以 `openspec/project.md` 与 `project-rules/` 的约定为准。

- **组件命名与粒度**
  - 原则：可复用、职责单一、语义清晰。
  - 容器组件：以页面或业务场景命名，如 `OrderListPage`、`ProductDetailContainer`。
  - 展示组件：以 UI 元素命名，如 `PriceTag`、`CountdownTimer`、`BadgeList`。

- **文件命名**
  - 一个组件一个文件，文件名与导出的主组件名称保持一致。
  - 若有紧密相关的 hook 或样式，可采用相邻文件或同目录命名约定（如 `MyComponent.tsx` + `MyComponent.hooks.ts` + `MyComponent.css`）。

---

## 三、数据与状态管理规范

- **状态来源优先级**
  1. 父组件传入的 props；
  2. 本地组件内部 state（与 UI 紧密相关）；
  3. 全局状态或共享 store（仅限多个页面/组件确有共享需求时）。

- **避免状态重复**
  - 不要在多个层次维护同一业务状态的重复副本，优先确保“单一可信来源”（Single Source of Truth）。

- **接口调用与错误处理**
  - 接口调用逻辑应封装在数据访问层/客户端模块，组件仅调用语义化方法。
  - 对错误进行分类处理（用户可感知 / 开发者调试 / 静默重试），并按照 project-rules 中的规范记录日志与上报埋点。

---

## 四、UI 行为与交互规范

- **严格对齐 Requirements + Scenarios**
  - 编码前应逐条阅读对应 `specs/*/spec.md` 的 Scenario；
  - 为每种状态（成功、加载中、空数据、异常、权限不足等）提供明确 UI 响应；
  - 尽量避免隐藏逻辑，所有状态与交互分支应在代码中清晰可见。

- **无障碍与国际化（如适用）**
  - 按项目规范支持必要的无障碍属性（如 ARIA 标签等）；
  - 文案与可见字符串应集中管理（如 i18n 资源文件），避免硬编码在多个组件中。

---

## 五、样式与设计系统

- **统一的设计系统/组件库**
  - 优先复用项目约定的基础组件库与样式系统（如设计 Token、主题变量等）；
  - 避免在业务组件中直接写大量硬编码样式。

- **响应式与适配**
  - 根据项目目标设备（PC / Mobile / Multi-platform）设置合理的断点与布局策略；
  - 注意性能与可维护性，避免过多深层嵌套与复杂选择器。

---

## 六、测试与可观测性

- **前端测试**
  - 关键组件与页面应具备基本的单元测试或集成测试；
  - 对复杂交互（如多步表单、倒计时逻辑等）建议增加关键路径用例。

- **日志与埋点**
  - 遵守项目埋点规范，关键业务事件与错误均应有埋点或日志记录；
  - 避免在 UI 组件中直接记录大量日志，将日志逻辑集中在专门的工具模块。

---

## 七、与 OpenSpec 的衔接

- 在实现每一块前端功能前，先确认对应的：
  - `change-id`；
  - Capability 与 Requirement ID；
  - Scenario 描述。
- 实现完成后，应能回答：
  - 哪段代码实现了哪条 Requirement？
  - 在哪些场景下，UI 会进入对应状态？
  - 如有新增前端可观察指标（如点击率、转化率等），是否在文档与任务中有所体现？
