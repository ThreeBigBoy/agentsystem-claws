# 任务拆分 spec

用于将需求分析结果拆解为 `openspec/changes/[change-id]/tasks.md` 中可执行、可勾选的任务列表。

## 1. 原则

- **可验证**：每项任务完成时有明确验收（如“实现某接口”“通过某测试”“文档已更新”）。
- **小步可交付**：单任务粒度建议可在半天内完成，避免过大块。
- **顺序合理**：按依赖排序（如先设计/建表再实现接口再联调）；与变更的 proposal、design、spec 一致。

## 2. 任务分类建议

| 阶段 | 示例任务 |
|------|----------|
| 准备/设计 | 确认接口契约、更新 design.md、创建/更新数据模型或配置 |
| 实现 | 按模块/层次实现（如数据层 → 业务层 → 接口层 → 表现层），每层可拆为多条 |
| 联调与测试 | 接口联调、端到端测试、边界与异常场景测试 |
| 文档与收尾 | 更新 README/API 文档、补充 CHANGELOG、运行 OpenSpec 校验 |

## 3. 格式要求

- 使用 Markdown 任务列表：`- [ ]` 未完成，`- [x]` 已完成。
- 可按阶段分节（如 `## 1. 准备与设计`、`## 2. 实现`），节内为任务列表。
- 任务描述简洁明确，必要时附验收标准或文件路径。

## 4. 示例（tasks.md 片段）

```markdown
# Tasks: [变更标题]

## 1. 准备与设计
- [ ] 1.1 在 design.md 中确定 API 路径与请求/响应格式
- [ ] 1.2 创建/更新数据库表结构（或配置文件）

## 2. 实现
- [ ] 2.1 实现 [capability] 数据层（Repository/DAO）
- [ ] 2.2 实现业务逻辑层
- [ ] 2.3 实现接口层（Controller/Route）
- [ ] 2.4 实现前端组件/页面（若适用）

## 3. 联调与测试
- [ ] 3.1 接口联调并通过核心场景
- [ ] 3.2 补充边界与异常测试

## 4. 文档与收尾
- [ ] 4.1 更新 API 文档或 README
- [ ] 4.2 运行 `openspec validate [change-id]` 并修复问题
```

## 5. 与 OpenSpec 的衔接

- 实施时按 tasks.md 从上到下执行，完成一项将 `- [ ]` 改为 `- [x]`。
- 变更部署后使用 `openspec archive <change-id> --yes` 归档，此时 specs 更新；tasks 作为历史记录保留在 changes 目录中。
