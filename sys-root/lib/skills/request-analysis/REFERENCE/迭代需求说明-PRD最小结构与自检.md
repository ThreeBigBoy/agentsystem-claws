# 迭代需求说明 / PRD 最小结构与自检

> 产品经理 Agent 执行 request-analysis 时，产出 docs/project-prd-changes 下的迭代需求说明或 PRD 须符合本结构，并完成自检，使产出达到**可商业化、可技术落地、可验收、可衡量**。  
> 依据：优秀产品经理实践复盘；采用本规范的项目可在 docs/project-prd-changes 下保留复盘文档供内部参考，规范本身不依赖具体项目路径。

---

## 一、最小结构（模板）

产出 `docs/project-prd-changes/[change-id]/` 下的**迭代需求说明**或**功能需求说明书 + 市场研究与产品方案**时，须覆盖以下 8 类内容（可合并为单文档或拆为多文档，与项目约定一致）。

### 1. 需求来源与背景

- 问题/机会描述；来源（用户反馈、走查、战略）。
- 背景：与 MVP/版本目标的关系。

### 2. 问题价值分析与优先级

- 每项问题的价值（谁受益/受损、对目标的影响）；P0/P1/P2 及理由。
- Non-Goals 与边界（可与 proposal 共享）。

### 3. 竞品/平台调研（可选但推荐）

- 同类功能在官方/竞品中的做法；结论与对本方案的影响。

### 4. 本迭代目标与成功标准

- 3～5 条可验证的迭代目标。
- 「如何衡量拿到目标」：与需求验收 Checklist、func-test、OpenSpec validate 的对应。

### 5. 产品方案要点

须按**可被前端/后端/测试 Agent 按需开发**的粒度撰写；原则：从 1 个产品功能点落地倒推需考虑的方方面面，兼顾用户体验、性能、实现成本。

- **若需求涉及 C 端（前台）**：须包含 **前端（toC）产品方案**，见 [六、产品方案要点编写指引](#六产品方案要点编写指引) 之 toC 要点。
- **若需求涉及 B 端（后台/管理/编辑器）**：须包含 **后端（toB）产品方案**，见 [六、产品方案要点编写指引](#六产品方案要点编写指引) 之 toB 要点。
- 通用：用户流程（谁-在什么情况-做什么-系统如何响应）；功能逻辑与信息架构；关键交互与异常/边界；必要时拆为 5.1 toC、5.2 toB、5.3 汇总。
- **场景描述（必选）**：每个功能点须补充**场景描述**，用 1～3 个典型场景说明「谁、在什么情况下、为什么需要该能力」，支撑该功能点的使用逻辑与能力必要性；写法见 [六、产品方案要点编写指引](#六产品方案要点编写指引)。
- **5 补充（可选但推荐）**：若涉及 toC/toB 的**界面布局效果与交互细节**，仅文字易产生歧义，建议补充**布局线框图、功能界面视觉图、交互链路设计图**等，作为 PRD 配套；存放与引用方式见 [七、PRD 补充物（设计产出物）](#七prd-补充物设计产出物)。

### 6. 功能列表与验收标准

- 功能列表、优先级；每项验收标准（可与 specs 中 Requirements/Scenario 对应）。

### 7. 需求验收 Checklist

- 可逐条勾选的验收项，与 specs 和 func-test 记录对应。

### 8. 与 openspec 的对应关系

- change-id；proposal、tasks、specs、design 路径；本目录与验收记录的追溯关系。

## 二、PRD 文档命名规范

### 2.1 命名原则

PRD 文档的命名应满足以下原则：

| 原则 | 说明 | 示例 |
|-----|------|------|
| **可识别性** | 从文件名即可识别这是 PRD 文档 | 包含 "PRD" 或 "迭代需求说明" 关键词 |
| **可追溯性** | 文件名包含 change-id，便于与 OpenSpec 变更目录对应 | `PRD-[change-id].md` |
| **可读性** | 文件名简洁明了，便于在文件列表中快速定位 | 避免过长，控制在 50 字符以内 |
| **一致性** | 同一项目内保持命名风格一致 | 统一采用 "PRD-" 前缀或统一采用 "迭代需求说明" |

### 2.2 推荐命名格式

#### 格式 1：PRD-前缀风格（推荐）

```
PRD-[change-id]-[关键词].md
```

- **change-id**: 变更 ID，kebab-case（如 `check-langgraph-backend`、`add-homepage`）
- **关键词**（可选）: 简短描述本次迭代的主题，便于快速理解（如 `一键自检`、`首页改版`）

**示例**:
- `PRD-check-langgraph-backend-一键自检.md`
- `PRD-add-homepage-首页改版.md`
- `PRD-update-auth-flow-登录优化.md`

#### 格式 2：迭代需求说明风格（传统）

```
迭代需求说明-[change-id]-[关键词].md
```

**示例**:
- `迭代需求说明-check-langgraph-backend-一键自检.md`
- `迭代需求说明-add-homepage-首页改版.md`

### 2.3 命名升级建议

对于新项目或正在进行规范升级的项目，建议：

1. **优先采用 PRD-前缀格式**：更清晰识别文档类型
2. **必须包含 change-id**：确保与 `openspec/changes/[change-id]/` 目录一一对应
3. **可附加关键词**：增强可读性，但非强制

### 2.4 多文档拆分命名

当 PRD 拆分为多文档时：

| 文档类型 | 命名格式 | 示例 |
|---------|---------|------|
| **主 PRD 文档** | `PRD-[change-id]-[关键词].md` | `PRD-add-homepage-首页改版.md` |
| **市场研究与产品方案** | `PRD-[change-id]-市场研究.md` | `PRD-add-homepage-市场研究.md` |
| **功能需求说明书** | `PRD-[change-id]-功能需求.md` | `PRD-add-homepage-功能需求.md` |
| **需求验收 Checklist** | `PRD-[change-id]-验收清单.md` | `PRD-add-homepage-验收清单.md` |
| **评审纪要** | `PRD-[change-id]-评审纪要.md` | `PRD-add-homepage-评审纪要.md` |

### 2.5 存放位置

无论采用何种命名格式，PRD 文档统一存放于：

```
docs/project-prd-changes/[change-id]/
```

## 三、自检清单（产出后必过）

执行 request-analysis 并产出 docs/project-prd-changes 后，按下列项自检；不满足时补全再同步 proposal、tasks、specs。

| # | 自检项 | 说明 |
|---|--------|------|
| 1 | **价值分析** | 是否写清每项问题/需求对「谁、在什么场景」的影响及优先级理由？ |
| 2 | **竞品/调研** | 是否做了同类功能调研（或注明「本次未做，建议后续补」）？ |
| 3 | **迭代目标** | 是否写出 3～5 条可验证的迭代目标？ |
| 4 | **成功衡量** | 是否定义「如何判断本迭代达成目标」（与验收 Checklist、func-test 对应）？ |
| 5 | **产品方案** | 是否有用户流程、功能逻辑或信息架构要点（谁-在什么情况-做什么-系统如何响应）？ |
| 5b | **产品方案 toC/toB 可落地** | 若涉及 C 端/B 端，是否按「六、产品方案要点编写指引」区分 toC 与 toB，并写清前端/后端/测试可依循的细节（布局、对象、操作、字段、异常、取数或数据流）？ |
| 5b2 | **场景描述** | 每个功能点是否补充场景描述（谁、在什么情况下、为什么需要该能力），以支撑使用逻辑与能力必要性？ |
| 5c | **设计产出物（线框图/视觉图/交互图）** | 若涉及 toC/toB 界面布局与交互细节，是否已补充或引用线框图、界面视觉图、交互链路图等（或明确注明「由后续设计阶段产出」/「沿用平台标准，不单独出图」）？ |
| 6 | **异常与边界** | 是否覆盖关键异常、空状态、非目标与边界？ |
| 7 | **PRD 结构** | 是否具备：市场研究与产品方案（或需求来源/价值）、功能需求说明（或功能列表+验收标准）、需求验收 Checklist？ |
| 8 | **与 openspec 一致** | docs/project-prd-changes 与 proposal、specs 是否可互相引用、无冲突？ |
| 9 | **文档命名规范** | PRD 文档命名是否符合「二、PRD 文档命名规范」要求（含 PRD/迭代需求说明前缀、change-id）？ |

## 四、评审纪要输出（自检留痕）

### 4.1 定位与必要性

**自检 ≠ 简单打勾**。自检是一个「自我评审」过程，需要将评审的详细过程、结论、依据、建议留痕，形成**评审纪要**，以便于：

- **追溯**：后续查看 PRD 为何如此设计，可参考评审纪要中的决策逻辑
- **对齐**：团队成员/其他 Agent 可通过评审纪要快速理解 PRD 的成熟度和关键决策
- **改进**：积累评审经验，形成可复用的评审模式

### 4.2 评审纪要内容结构

每次 PRD 自检后，须在 `docs/project-prd-changes/[change-id]/records/` 下创建**评审纪要文档**，命名建议：
- `PRD-[change-id]-评审纪要.md`（遵循新命名规范）
- `PRD评审纪要.md`（传统命名，仍兼容）

评审纪要须包含：

| 章节 | 内容要求 |
|-----|---------|
| **1. 评审基本信息** | 评审日期、评审人（Agent 角色）、PRD 版本、所属 change-id |
| **2. 评审过程记录** | 逐条过自检清单的详细记录 |
| **3. 问题发现与处理** | 发现的问题、判定结论（通过/有条件通过/不通过）、修改建议、引用依据 |
| **4. 整体评审结论** | 综合判定：通过 / 有条件通过 / 不通过；后续行动项 |
| **5. 附录** | 参考文档、相关记忆、自检清单原始链接 |

### 4.3 评审过程记录格式

对自检清单的每一项，须记录：

```markdown
### 自检项 #1: 价值分析

- **评审内容**: 是否写清每项问题/需求对「谁、在什么场景」的影响及优先级理由？
- **评审发现**: [详细描述 PRD 中该部分的内容，是否满足要求]
- **判定结论**: ✓ 通过 / △ 有条件通过 / ✗ 不通过
- **判定理由**: [如果通过，简述通过的逻辑；如果不通过或有条件通过，详细说明理由]
- **修改建议**: [如果不通过或有条件通过，给出具体修改建议]
- **引用依据**: [支撑判定观点的参考：行业最佳实践、相关规范、竞品案例等]
- **修改后复核**: [修改后再次评审的结果和日期]
```

### 4.4 评审判定标准

| 判定 | 含义 | 后续行动 |
|-----|------|---------|
| **✓ 通过** | 该项内容完整、逻辑自洽、符合行业最佳实践 | 无需修改，进入下一项 |
| **△ 有条件通过** | 该项基本可用，但存在 minor 问题或建议优化项 | 记录问题，可在当前阶段或后续迭代优化 |
| **✗ 不通过** | 该项缺失、逻辑不自洽、不满足落地要求 | 必须修改，修改后复核 |

### 4.5 整体评审结论标准

| 综合判定 | 条件 | 后续行动 |
|---------|------|---------|
| **通过** | 所有自检项均通过（或仅 minor 建议，无阻塞问题） | 可进入 proposal/tasks/specs 阶段 |
| **有条件通过** | 存在有条件通过项，但无阻塞性问题 | 记录优化项，可并行进入后续阶段，但须在迭代中完成优化 |
| **不通过** | 存在不通过项 | 必须修改 PRD，重新评审 |

### 4.6 评审维度参考框架

评审须围绕以下底层逻辑展开：

#### 价值层

| 维度 | 评审要点 |
|-----|---------|
| **需求真实性** | PRD 是否有真实的需求场景？是否符合需求价值逻辑（解决谁的什么问题）？是否有值得做的必要性？ |
| **技术 ROI** | 技术方案是否能确定实现需求目标？实现过程是否 ROI 合理（投入产出比）？ |

#### 执行层

| 维度 | 评审要点 |
|-----|---------|
| **产品方案完整性** | 产品设计方案是否贴合场景需求？是否有完整对应的需求功能点拆解清单？产品流程链路、概念模型、界面设计、交互设计、数据处理、埋点采集、非功能性要求等不同维度是否均清晰阐述且逻辑自洽？ |
| **产品方案合理性** | 既不存在过度设计的情况，也不存在设计方案无法有效满足场景需求的情况？是否有明确的结果指标和过程衡量指标体系设计？ |
| **技术方案匹配度** | 技术方案是否 100% 满足 PRD 期望实现需求和目标？总体架构设计是否合理、细节功能技术实现方案是否清晰、逻辑自洽？ |
| **技术方案可行性** | 实现路径选型 ROI 最优？有落地可行性且可预见能稳定高效运行？ |
| **行业最佳实践** | 整体是否符合 PRD/技术方案撰写的行业最佳实践标准（无论前端产品 PRD、后端产品 PRD、数据产品 PRD，适用不同行业）？ |

### 4.7 与迭代日志的区别

| 对比 | 迭代日志 | PRD 评审纪要 |
|-----|---------|-------------|
| **定位** | 谁在什么时间做了什么事（日志型） | 评审了什么、结论是什么、为什么（评审型） |
| **内容** | 执行方、技能、change-id、产出物路径 | 自检项、评审发现、判定结论、理由、建议、依据 |
| **粒度** | 单次技能执行 | PRD 自检的完整过程 |
| **存放** | `docs/项目事件日志.md`（项目级单一文件） | `docs/project-prd-changes/[change-id]/records/PRD-[change-id]-评审纪要.md`（按变更归档） |

## 五、与技能流程的对应

- **request-analysis 步骤 2**：产出 `docs/project-prd-changes/[change-id]/` 时（V3 规范统一存放到 changes/ 目录），**迭代类或 PRD 类**需求须采用本节「一、最小结构」；产出后执行「三、自检清单」，通过后再进行步骤 3～5（openspec 目录、proposal、tasks、specs）。
- **产出物命名**：推荐采用「二、PRD 文档命名规范」中的 `PRD-[change-id]-[关键词].md` 格式，也可采用 `迭代需求说明-[change-id]-[关键词].md` 传统格式，与 [新增类需求分析spec](新增类需求分析spec.md)、[修改类需求分析spec](修改类需求分析spec.md) 中 documents 约定一致。
- **prd-review 技能触发**：当用户输入「评审 PRD」「检查 PRD 质量」「PRD 自检」等指令时，触发 **prd-review** 技能，按本节「四、评审纪要输出」规范执行评审并产出评审纪要。

## 六、产品方案要点编写指引

产品方案要点应足够详细，使**前端 Agent、后端 Agent、测试 Agent** 能按需求开发与验收。按功能点落地时需考虑：用户体验、性能、实现成本；以下为 toC 与 toB 的常用要点（实际不限于此，按项目与功能点按需裁剪）。

### 6.1 前端（toC）产品方案

针对 **C 端用户** 在前台看到什么、能操作什么、出现什么反馈。需求要点一般包括：

- **场景描述**：每个功能点 1～3 个典型场景，格式如「**谁**（角色）在**什么情况**下（入口/前置条件）**为什么需要**该能力（目标/结果）」；用于支撑使用逻辑与能力必要性。
- **界面布局**：整体结构、分区、响应式或断点。
- **构成内容要素**：展示字段、具体图/文/视频等不同内容定义、数据来源（哪个 API、哪个配置、哪个 locale）。
- **视觉方案**：风格、主题、可访问性要求（若有）。
- **内容展示与用户互动交互**：默认态、悬停/点击/提交等交互、反馈（toast、跳转、错误提示）。
- **异常兜底处理**：无数据、加载失败、权限不足、网络错误时的展示与降级。
- **内容取数逻辑**：数据从哪来（接口、theme 配置、locale 文件）、何时请求、缓存或静态策略。

**推荐写法**：按功能点分小节（如 #### 多语言、#### 页头/页脚），每点下用**列表**写出各维度（看到什么、能操作什么、反馈、布局与内容、异常兜底、取数逻辑），避免单表列数过多导致阅读体验差；若用表格，建议单表不超过 3～4 列（如 功能点 | 方案要点），将要点写在单元格内分段或列表。

### 6.2 后端/管理端（toB）产品方案

针对 **B 端用户** 在后台/管理/编辑器中看到什么、能操作什么、出现什么反馈。需求要点一般包括：

- **场景描述**：每个功能点 1～3 个典型场景，格式如「**谁**（角色）在**什么情况**下（入口/前置条件）**为什么需要**该能力（目标/结果）」；用于支撑使用逻辑与能力必要性。
- **界面布局**：管理界面分区、列表与详情关系、导航。
- **操作对象定义**：核心实体（如 section group、section、block、商品、订单等）及层级关系。
- **操作交互能力说明**：添加、删除、隐藏、移动、选中、编辑参数；是否支持批量操作；是否支持上传图/视频；是否调用平台通用控件（如媒体选择器、collection 选择器）。
- **编辑某对象时的具体内容说明**：  
  - 有哪些参数字段（类型、必填/可选、默认值）。  
  - 字段与字段间的层级、联动关系。  
  - 哪些字段定义成同一组（groups）。  
  - 是否根据某参数动态控制部分参数组的显隐或生效与否。  
  - 是否涉及 collection/商品级关联或其他平台资源选择。
- **视觉风格方案要求**：与平台设计规范或现有后台一致（若适用）。
- **批量操作、模版一键高效操作能力**：若有则说明范围与结果。
- **数据流转逻辑**：编辑后写入哪里（如 theme JSON、数据库表）、如何同步到前台或下游。

**推荐写法**：按功能点分小节（如 #### 页头/页脚 section group、#### Section 下自定义区块），每点下用**列表**写出各维度（操作对象、界面布局、操作能力、参数字段与联动、视觉与批量、数据流转），避免单表列数过多；若用表格，建议单表不超过 3～4 列，要点在单元格内分段或列表。

## 七、PRD 补充物（设计产出物）

当 toC/toB 涉及**界面布局效果与交互细节**时，仅文字描述易产生歧义，建议将以下设计产出物作为 **PRD 补充**，与迭代需求说明/功能需求说明书配套使用，便于前端、后端、测试按图实现与验收。

### 7.1 建议的补充物类型

| 类型 | 说明 | 典型用途 |
|------|------|----------|
| **布局线框图（Wireframe）** | 关键页面或区块的结构、分区、元素大致位置与层级，可低保真。 | toC 前台关键页（如首页、商品页、语言选择区）；toB 编辑器关键视图（如「页头/模板/页脚」三区、section 列表、设置面板分区）。 |
| **功能界面详细视觉图（Mockup/视觉稿）** | 关键界面的视觉呈现：字体、颜色、间距、组件样式；可为高保真或关键态截图+标注。 | toC 首屏、CTA 区、多语言选择器样式；toB 设置分组折叠后的面板样式、block 列表与 section 并排布局。 |
| **交互链路设计图（Flow）** | 用户操作路径、状态流转、分支（正常/异常）；可用流程图、状态图或步骤列表+箭头。 | toC 访客切换语言后的跳转与重渲染；toB 商户添加 section → 拖拽排序 → 编辑设置 → 保存 的完整链路；异常分支（如保存失败、无权限）。 |

以上可二选一或组合使用；若项目有设计规范或设计阶段，可由设计阶段产出后放入 docs/project-prd-changes 并在 PRD 中引用。

### 7.2 存放位置与 PRD 引用

- **存放**：`docs/project-prd-changes/[change-id]/` 下，可建子目录如 `wireframes/`、`mockups/`、`flows/`，或统一 `design-assets/`；文件命名建议含功能点或页面名（如 `home-sections-wireframe.png`、`editor-header-footer-flow.md`）。
- **在 PRD 中引用**：在迭代需求说明或功能需求说明书的对应小节（如 5.1 toC、5.2 toB）中写明「布局与交互见：`docs/project-prd-changes/[change-id]/wireframes/xxx.png`」或「交互链路见：`docs/project-prd-changes/[change-id]/flows/xxx.md`」，便于前端/后端/测试按图实现。
- **与 image-analysis 技能**：若需求分析时用户提供了设计稿或截图，可经 image-analysis 解析后，将解析结论写入 PRD 并保存原图至上述目录，建立「图 → 文档」可追溯。

### 7.3 何时需要、何时可省略

- **建议补充**：界面与交互较复杂、多状态、多角色、易歧义时；或需与设计/视觉对齐时。
- **可注明不单独出图**：完全沿用平台标准（如仅用 Shopify 主题编辑器标准控件、Dawn 参考布局）时，可在 PRD 中写明「toC/toB 布局与交互沿用 [平台/参考主题] 标准，不单独产出线框图/视觉图」；自检时勾选「已明确沿用标准」即可。
- **由后续设计阶段产出**：若本次仅需求分析、设计阶段在后，可在 PRD 中注明「线框图/视觉图/交互链路图由设计阶段产出后补充至 docs/project-prd-changes/[change-id]/」，并在自检项 5c 中勾选「已注明由后续设计产出」。
