# PRD 评审技能（prd-review）

## 触发场景

- 用户输入「评审 PRD」「检查 PRD 质量」「PRD 自检」「审查需求文档」等指令
- 用户要求对已产出的 PRD 进行质量检查、完整性校验、逻辑审查
- 用户需要在进入 proposal/tasks/specs 阶段前，确保 PRD 达到「可商业化、可技术落地、可验收、可衡量」标准
- 用户需要回溯补录 PRD 评审纪要（基于历史对话和已产出文档）

## 整体流程

### 0. 加载 PRD 评审规范（强制顺序）

路径以 **`agentsystem` 仓库根**为基准（多根工作区：`sys-root/lib/skills/...`）。

1. **`request-analysis/REFERENCE/迭代需求说明-PRD最小结构与自检.md`**
   - 「一、最小结构」「二、PRD 文档命名」「**三、自检清单**（9 项）」「**四、评审纪要输出**」
   - 「**六、产品方案要点编写指引**」—— **§6.1 前端（toC）**、**§6.2 后端/管理端（toB）**：与自检项 **#5b** 配套，是 **C/B 端可落地深度** 的权威展开；**不得**仅用 9 项表格打分替代本节。
2. **`prd-review/REFERENCE/PRD评审纪要规范.md`**
   - 纪要章节、附录 A 模板、与迭代日志分工。

> **常见失误（须避免）**：只对照「三、自检清单」打 ✓，但未按 **§6.1/§6.2** 做 **维度对齐**，导致「自检通过但 C 端产品深度不足」——本技能 **§3.5** 为强制补位。

### 0.1 与 request-analysis 的衔接（前置建议，减少返工）

- **不修改** `request-analysis` SKILL 正文亦可执行本技能；但为减少「PRD 过薄 → prd-review 打回」，建议在 **`request-analysis` 产出 PRD（Step 2）时即包含**：**迭代类型一句话** + **「产品方案深度裁剪说明」骨架**（工程验证型）或 **§6.1/§6.2 级要点**（全量产品型）。
- **含 UI/路由/小程序时**：建议同步给出 **页面或路由清单草稿**、**关键 mock 字段**，或明确「路由与数据契约见 Step 3 `design.md`」——便于 **§3.8 附录 C** 一次对齐。
- 评审时若 PRD 缺失上述骨架且 #5b 仍偏泛 → 按 **§3.5** 要求补全后再判 **✓**。

### 1. 确定评审对象

- 从用户输入或上下文中识别本次评审的 **change-id**
- 若未给出 change-id，应向用户询问
- 检查 `docs/project-prd-changes/[change-id]/` 目录是否存在 PRD 文档
  - 按命名规范检查以下可能的文件名：
    - `PRD-[change-id]-[关键词].md`（推荐新格式）
    - `PRD-[change-id].md`
    - `迭代需求说明-[change-id]-[关键词].md`
    - `迭代需求说明.md`
    - `功能需求说明书.md`
- 若找不到 PRD 文档，告知用户并请求确认路径

### 2. 读取 PRD 文档

- 读取找到的 PRD 文档全文
- 理解 PRD 的当前状态：完整版还是简略版？已覆盖 8 类内容中的哪些？
- 记录 PRD 版本信息（文档头部的版本号或最后更新日期）

### 3. 执行系统化评审（9 项自检）

按照「三、自检清单」的 9 项内容逐项评审：

| 序号 | 自检项 | 评审要点 |
|-----|-------|---------|
| 1 | 价值分析 | 是否有受益方分析？优先级 P0/P1/P2 理由是否充分？ |
| 2 | 竞品/调研 | 是否做了同类功能调研？结论是否影响本方案？ |
| 3 | 迭代目标 | 是否写出 3～5 条可验证的迭代目标？ |
| 4 | 成功衡量 | 是否定义衡量标准？是否与验收 Checklist 对应？ |
| 5 | 产品方案 | 是否有用户流程、功能逻辑、信息架构？ |
| 5b | toC/toB 可落地 | 是否区分 toC/toB？细节是否足够开发使用？ |
| 5b2 | 场景描述 | 每个功能点是否有「谁-在什么情况-为什么需要」场景？ |
| 5c | 设计产出物 | 界面布局是否有线框图/视觉图/交互图（或注明沿用标准）？ |
| 6 | 异常与边界 | 是否覆盖关键异常、空状态、非目标？ |
| 7 | PRD 结构 | 是否具备 8 类内容？结构是否完整？ |
| 8 | 与 OpenSpec 一致 | 与 proposal、specs 是否可互相引用、无冲突？ |
| 9 | 文档命名规范 | 是否符合「PRD-[change-id]-[关键词].md」命名规范？ |

**评审方式**：
- 对照 REFERENCE 中的「一、最小结构」逐条检查 PRD 内容
- 发现不满足项时，记录：评审发现、判定结论、判定理由、修改建议、引用依据
- 评审时须围绕「价值层」和「执行层」两个底层逻辑展开（见 `迭代需求说明-PRD最小结构与自检.md`「**四、评审维度参考框架**」及本技能配套 **`REFERENCE/PRD评审纪要规范.md`**「**四、评审维度参考框架**」）

### 3.5 产品方案深度审查（强制；与 #5b 配套）

在逐项自检的同时，必须完成 **REFERENCE §6.1（toC）与/或 §6.2（toB）维度对齐**，并在评审纪要 **「附录 A」** 落表（见 **`prd-review/REFERENCE/PRD评审纪要规范.md`**）。维度至少覆盖 §6.1/§6.2 所列要点（可按项目删减列，但须在表中注明「不适用」理由）。

**PRD 迭代类型（先判定，再定 #5b 口径）**：

| 类型 | 含义 | 对 #5b / §6.1 的要求 |
|------|------|----------------------|
| **全量 C/B 端产品迭代** | 面向真实用户主路径、可独立验收上线 | PRD 须在正文或附件中达到 §6.1/§6.2 可开发粒度（布局分区、要素、交互反馈、取数逻辑等）；**不得**仅以「占位」带过 |
| **工程/架构/技术验证型** | 壳、路由、合规、联调验证为主 | 允许 **显式裁剪** §6.1/§6.2 深度，但 PRD **必须**包含独立小节（或表格）：**「产品方案深度裁剪说明」**，写明各维度是 **本迭代覆盖 / 延后至某 change-id / 由 design 承接**；纪要附录 A 须与之一致。**禁止**在无该小节的情况下将 #5b 评为 ✓「通过」 |

**涉及未成年人、亲子、教育、内容安全时**：须在 PRD 或附录中体现 **合规与产品约束**（时长、隐私、内容边界等）**已写清或已声明延后**，否则 #5b 或专项自检不得 ✓。

**与 #5b 判定关系**：
- 若 PRD **未**提供「深度裁剪说明」且 toC/toB 方案仍停留在 IA 级 → #5b 最高 **△ 有条件通过**（须修 PRD 或补附录后再评）
- 若已提供裁剪说明且附录 A 对齐 → #5b 可 **✓**，但综合结论须标明 **「通过（限定：工程验证型）」** 或 **「通过（全量 C 端）」**，避免与上市级 PRD 混淆

### 3.6 专项风险域（命中即须显式评审）

若 PRD 涉及以下**任一**主题，须在评审纪要中增加 **「专项风险域」** 小节（或勾选 `REFERENCE` 中的专项表），并核对 PRD 是否已写清约束或已声明延后；**未写清且未声明延后**的，对应项 **不得 ✓**（至少 **△**，直至补全）：

| 域 | 典型关注点（示例） |
|----|-------------------|
| 未成年人 / 亲子 | 使用时长、家长同意、适龄内容、收集个人信息边界 |
| 教育 / 培训 | 宣传合规、效果承诺、资质表述 |
| UGC / 内容安全 | 审核责任、举报、违规处置 |
| 支付 / 金融 | 主体资质、虚拟支付、合规路径 |

（与 **§3.5**「涉及未成年人…」条款叠加执行。）

### 3.7 可观测性 / 埋点（可选附录 B）

- 若 PRD 涉及可量化业务目标、转化漏斗或需线上排障：检查是否声明 **埋点 / 日志 / 实验** 需求，或明确 **本迭代不埋点、延后至某 change-id**。
- 若有要求，在评审纪要中增加 **附录 B（可选）**，见 **`REFERENCE/PRD评审纪要规范.md`**；与 memory `pattern-observable-small-steps` 等可对照，**不强制**每项目必有。

### 3.8 编码与测试可执行性（附录 C；含 UI / 路由 / 多端时强制）

> **目的**：避免 PRD 在产品语义上「通过」，但开发/测试仍缺 **页面路径、数据契约、可执行用例**，导致实现依赖口头对齐。

**何时必须落「附录 C」**（在评审纪要中；模板见 **`REFERENCE/PRD评审纪要规范.md`**）：

- 本 change 含 **用户可见界面**（小程序/Web/App/桌面客户端等）、**路由/多页面**，或 **需按步骤验收的交互**；**且**
- PRD 正文 **未**以表格形式给出：页面/模块与 **功能 ID（F）**、**路由或页面路径**、**关键 mock/字段**、**最小测试场景（Given/When/Then 简写）** 的对照。

**可接受的等价物**（须在纪要附录 C 中 **一行写清引用**，并确认文件已存在或可随 Step 3 补全）：

- **`openspec/changes/[change-id]/design.md`** 中已含 **页面/路由清单 + 数据契约**；和/或
- **`tasks.md`** 已按任务拆到可映射 F-ID；和/或
- **`specs/.../spec.md`** 中 Scenario 已细到可转测试用例。

**工程验证型**：允许大量「见 design.md §x」，但 **不得**三者皆空（PRD 无表、design 无路由、spec 无场景）。

**纯后端 / 无 UI**：附录 C 可填 **「不适用」** 并说明理由（如仅 API/批任务）。

**与 #5b / 综合判定的关系**：

- 若触发「须附录 C」条件且 PRD + 引用文档仍无法回答「测什么页面、什么数据、什么算过」→ #5b 或整体结论至少 **△**，直至补齐或声明等效物。
- 全量 C 端：**附录 C**（或 PRD 内嵌同等表格）应接近 **可写自动化用例** 的粒度。

### 4. 形成评审判定

对每项自检项给出判定：

- **✓ 通过**：内容完整、逻辑自洽、符合行业最佳实践
- **△ 有条件通过**：基本可用，但存在 minor 问题或建议优化项
- **✗ 不通过**：缺失、逻辑不自洽、不满足落地要求

**综合判定**：
- **通过**：所有自检项均为 **✓**（含经 §3.5 对齐后的 #5b），且无阻塞问题；**须**在纪要「整体结论」中写明 **迭代类型标签**（全量 C 端 / 工程验证型等）
- **有条件通过**：存在 **△** 项 → **不得**记为最终通过；须走修复环直至全部为 ✓（见 `pattern-review-fix-loop`）
- **不通过**：存在 **✗** 项，必须修改 PRD

> **禁止**：在 **未**完成 **§3.5 附录 A** 的情况下给出「综合判定：通过」。  
> **禁止**（补充）：在触发 **§3.8**（含 UI/路由）且 **附录 C 未填写**、PRD/`design.md`/`specs` 中也**无等效可引用内容**的情况下，将综合判定记为 **通过**。

**评审结论与后续动作映射（重要）**：

| 评审结论 | 是否可以进入下一阶段 | 后续动作 |
|---------|-------------------|---------|
| **✓ 通过** | ✅ **可以** | 直接进入下一阶段（Step 3: 工程结构分析） |
| **△ 有条件通过** | ❌ **不可以** | **必须**修复问题清单中的 minor 建议，**重新评审**通过后，才能进入下一阶段 |
| **✗ 不通过** | ❌ **不可以** | **必须**修改 PRD，**重新评审**通过后，才能进入下一阶段 |

> ⚠️ **重要澄清**：「有条件通过」≠ 「可以进入下一阶段」。只有「100% 通过」才是真正的通过。详见 `memory/patterns/pattern-review-fix-loop.md`

### 5. 产出评审纪要

在 `docs/project-prd-changes/[change-id]/records/` 下创建评审纪要文档，命名：
- 推荐：`PRD-[change-id]-评审纪要.md`
- 兼容：`PRD评审纪要.md`

评审纪要必须包含以下章节：

#### 1. 评审基本信息

- 评审日期
- 评审人（Agent 角色：产品经理 Agent / 主 Agent）
- PRD 版本
- 所属 change-id
- 评审依据（引用的 REFERENCE 文档路径）

#### 2. 评审过程记录

对每项自检项，按以下格式记录：

```markdown
### 自检项 #1: 价值分析

- **评审内容**: 是否写清每项问题/需求对「谁、在什么场景」的影响及优先级理由？
- **评审发现**: [详细描述 PRD 中该部分的内容，是否满足要求]
- **判定结论**: ✓ 通过 / △ 有条件通过 / ✗ 不通过
- **判定理由**: [详细说明理由]
- **修改建议**: [给出具体修改建议]
- **引用依据**: [行业最佳实践、相关规范、竞品案例等]
- **修改后复核**: [如已修改，记录复核结果]
```

#### 3. 问题发现与处理

汇总所有发现的问题：

| 序号 | 自检项 | 原始判定 | 问题类型 | 修改建议 | 状态 |
|-----|-------|---------|---------|---------|------|
| 1 | 价值分析 | ✗ 不通过 | 缺失 | 补充受益方分析 | 待修改 |

#### 4. 整体评审结论

- **综合判定**: 通过 / 有条件通过 / 不通过
- **判定理由**: 总结性说明
- **后续行动项**: 明确的下一步行动

#### 5. 附录

- 参考文档列表
- 相关 memory 条目
- 自检清单原始链接

#### 6. 附录 A：产品方案深度对齐（强制）

- **内容**：按 **§3.5** 与 **`REFERENCE/PRD评审纪要规范.md`** 填写 **REFERENCE §6.1 / §6.2** 维度表（本迭代覆盖 / 延后 / design 承接 / 不适用）
- **位置**：置于纪要「附录」内，作为 **附录 A**（不可仅口头在对话中说明）

#### 7. 附录 B：可观测性 / 埋点（可选）

- **何时写**：§3.7 触发时；或 PRD 已写清数据/实验诉求时，在纪要中复述对齐结论。
- **模板**：见 **`REFERENCE/PRD评审纪要规范.md`**。

#### 8. Gate-PRD（人工终稿）提示

- **PRD-R 综合判定「通过」≠ 终稿定版**：人类 **Gate-PRD** 书面确认见 **`REFERENCE/PRD评审纪要规范.md`**「**Gate-PRD 人工终稿确认清单**」（与 `usr-devclaw/usr-rules/OpenSpec.md` §2.2.1 对齐）。
- Agent 可在用户可见回复中提示：**下一步须完成 Gate-PRD 留痕后再进入 Step 3**。

#### 9. 附录 C：编码与测试落地对齐（条件强制）

- **内容**：按 **§3.8** 与 **`REFERENCE/PRD评审纪要规范.md`** 填写；可与附录 A、B 并列。
- **纯后端/无 UI**：可声明不适用并简述理由。

### 7. 反馈评审结论

向用户反馈：

1. **评审结果摘要**：综合判定（通过/有条件通过/不通过）
2. **关键问题**：阻塞性问题（如有）
3. **建议优化项**：minor 问题或改进建议
4. **评审纪要路径**：告知用户评审纪要文档存放位置
5. **后续行动建议**：
   - 若通过：可进入 Step 3（工程结构分析）；若 §3.8 曾打 △，提示优先补 **design.md 路由/数据** 或 PRD 表格，再开编码
   - 若有条件通过：**不能进入下一阶段**。必须：① 修复问题清单中的 minor 建议 → ② 执行重新评审 → ③ 转为「通过」后，才能进入下一阶段
   - 若不通过：**不能进入下一阶段**。必须：① 修改 PRD 解决阻塞性问题 → ② 执行重新评审 → ③ 转为「通过」后，才能进入下一阶段

**评审修复循环**：
```
首次评审 ──→ 有条件通过/不通过 ──→ 修复问题 ──→ 重新评审 ──→ 通过？──→ 否 → 继续修复
                                                         └──────→ 是 → 进入下一阶段
```

## 执行后必做收尾

按 `skills-rules-for-agent.md` 通用约定，执行本技能后必须：

1. **向迭代日志追加记录**：

   **首次评审记录**（所有评审都必须记录）：
   - 文件：`docs/项目事件日志.md`
   - 格式：`- [日期] | [change-id] | prd-review | PRD 评审完成，综合判定：[通过/有条件通过/不通过]，评审纪要路径：[路径]`

   **重新评审记录**（如首次评审为「有条件通过」或「不通过」，修复后必须重新评审并记录）：
   - 格式：`- [日期] | [change-id] | prd-review | PRD 重新评审完成，修复项：[N 项]，综合判定：[通过]，评审纪要路径：[路径]`

   **示例**：

## 注意事项

- **必须先读取 REFERENCE**：执行本技能前必须先读取 `迭代需求说明-PRD最小结构与自检.md`（含 **§六**）与 `prd-review/REFERENCE/PRD评审纪要规范.md`，确保评审标准统一
- **详细记录**：评审纪要不是简单打勾，必须包含「评审发现、判定理由、修改建议、引用依据」
- **客观公正**：评审须基于 REFERENCE 规范，避免主观判断；判定理由必须有依据支撑
- **建设性反馈**：即使不通过，也要给出具体、可执行的修改建议
- **支持回溯评审**：用户可以要求基于历史对话和已产出文档回溯补录评审纪要
- **评审留痕制度**：详见 memory **`patterns/pattern-prd-architecture-review-audit-trail.md`**（纪要型 vs 日志型、审计追溯）；本技能产出须与该模式一致。
- **功能ID格式**：统一使用 `F-\d+(?:\.\d{1,2})?` 格式，如 `F-1`、`F-1.1`、`F-2.15`。禁止使用旧格式 `FR-1.1`。

### 功能ID格式规范

功能ID用于唯一标识PRD中的功能点，格式定义如下：

```
F-主功能编号[.子功能编号]
```

**规则**：
- 主功能编号：正整数，如 `F-1`、`F-2`
- 子功能编号：最多2位小数，如 `F-1.1`、`F-1.2`、`F-1.15`
- 子功能编号可选

**有效示例**：`F-1`、`F-2`、`F-1.1`、`F-1.2`、`F-2.1`、`F-1.15`
**无效示例**：`FR-1`（旧格式）、`F-1.1.1`（超过2位小数）

## 附录A: 第一性原理评审框架（v1.5+）

### 核心原则

评审PRD时，首先问自己：**这份PRD能回答哪些本质问题？**

如果读者看完后无法回答这些问题，说明PRD设计得不够本质。

### 问题诊断：为什么"关键词罗列"不够？

| 问题 | 表现 | 后果 |
|------|------|------|
| **可被刷分** | 堆砌功能点但无优先级 | PRD看起来全但实际抓不住重点 |
| **无法区分深度** | 写一行"安全性"和写十行都得满分 | 无法反映真实产品思考 |
| **违背第一性原理** | 关注"有没有功能"而非"为什么需要" | PRD不反映真实价值 |

### PRD的第一性原理维度

PRD是"为什么做、做什么、怎么做"的蓝图，包含：

| 维度 | 本质问题 | 评审必须回答 | 质量标准 |
|------|----------|-------------|---------|
| **必要性** | 为什么要做 | 需求是否真实？是否直击项目短板？ | 有受益方分析、P0/P1理由 |
| **合理性** | 这样做对吗 | 产品方案在用户视角是否成立？ | 有场景分析、无明显设计坑 |
| **完整性** | 功能全覆盖了吗 | 功能点是否无遗漏、无过度实现？ | 有功能清单、有验收标准 |
| **可落地性** | 能否开发使用 | toC/toB细节是否足够开发使用？ | §6.1/§6.2维度对齐 |
| **可衡量性** | 如何验证成功 | 成功指标是否定义？ | 有结果/过程指标 |

### 与9项自检的映射关系

| 自检项 | 对应第一性原理维度 | 质量指标 |
|--------|-------------------|---------|
| 1.价值分析 | 必要性 | 受益方分析、优先级理由 |
| 2.竞品/调研 | 合理性 | 调研结论影响本方案 |
| 3.迭代目标 | 必要性 + 可衡量性 | 3-5条可验证目标 |
| 4.成功衡量 | 可衡量性 | 衡量标准与验收对应 |
| 5.产品方案 | 完整性 | 用户流程、功能逻辑、信息架构 |
| 5b.toC/toB可落地 | 可落地性 | §6.1/§6.2深度对齐 |
| 6.异常与边界 | 完整性 + 合理性 | 异常、空状态、非目标覆盖 |
| 7.PRD结构 | 完整性 | 8类内容完整 |
| 8.与OpenSpec一致 | 可落地性 | 可追溯引用 |
| 9.文档命名规范 | 完整性 | 命名规范 |

### 四维需求评审法的地位

| 四维 | 对应第一性原理 | 说明 |
|------|---------------|------|
| 必要性 | 价值层 | 为什么做 |
| 合理性 | 价值层 | 这样做对吗 |
| 优先级 | 可衡量性 | 值得排在前面 |
| 内容质量 | 完整性 + 可落地性 | 可执行/可验收粒度 |
