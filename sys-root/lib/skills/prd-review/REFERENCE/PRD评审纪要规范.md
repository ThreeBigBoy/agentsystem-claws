# PRD 评审纪要规范

> 本规范为 `prd-review` 技能的配套 REFERENCE，固化 PRD 评审纪要的**内容结构、判定标准、输出格式**，为所有项目提供自检留痕的标准模板和实践参考。
> 
> 与 `request-analysis/REFERENCE/迭代需求说明-PRD最小结构与自检.md` 中的「四、评审纪要输出」保持一致，本规范侧重**操作层面的模板和示例**。
>
> **仓库路径**：凡写 `agentsystem` 内技能与 REFERENCE，默认指 **`agentsystem/sys-root/lib/skills/`**（与 `prd-review/SKILL.md` 一致）。

---

## 一、评审纪要定位

### 1.1 什么是评审纪要

**评审纪要**是 PRD 自检过程的**详细留痕文档**，记录：

- 评审了什么（自检项）
- 发现了什么（评审发现）
- 结论是什么（判定结论：通过/有条件通过/不通过）
- 为什么是这个结论（判定理由、引用依据）
- 如何改进（修改建议）
- 修改后如何（修改后复核）

### 1.2 评审纪要 vs 迭代日志

| 对比维度 | 迭代日志 | PRD 评审纪要 |
|---------|---------|-------------|
| **定位** | 谁在什么时间做了什么事（日志型） | 评审了什么、结论是什么、为什么（评审型） |
| **详细程度** | 简明扼要（一行记录） | 详细完整（多章节文档） |
| **用途** | 追溯执行历史 | 追溯评审决策、支撑质量改进 |
| **存放** | `docs/项目事件日志.md` | `docs/project-prd-changes/[change-id]/records/PRD-[change-id]-评审纪要.md` |

### 1.3 为什么必须留痕

1. **可追溯**：后续查看 PRD 为何如此设计，可参考评审纪要中的决策逻辑
2. **可对齐**：团队成员/其他 Agent 可通过评审纪要快速理解 PRD 的成熟度和关键决策
3. **可改进**：积累评审经验，形成可复用的评审模式
4. **可审计**：质量审计时可查看评审过程是否规范、结论是否合理

## 三、评审判定标准（速查表）

### 3.1 单项评审判定

| 判定 | 图标 | 含义 | 后续行动 |
|-----|------|------|---------|
| **通过** | ✓ | 内容完整、逻辑自洽、符合行业最佳实践 | 无需修改，进入下一项 |
| **有条件通过** | △ | 基本可用，但存在 minor 问题或建议优化项 | 记录问题，可在当前阶段或后续迭代优化 |
| **不通过** | ✗ | 缺失、逻辑不自洽、不满足落地要求 | 必须修改，修改后复核 |

### 3.2 整体评审结论

| 综合判定 | 条件 | 后续行动 |
|---------|------|---------|
| **通过** | 所有自检项均为 **✓**；**附录 A** 完成；若含 **UI/路由**（见 SKILL **§3.8**）则 **附录 C** 已填或已有 **声明等效** 的 `design.md`/`specs` 节 | 可进入下一阶段；注明 **迭代类型** |
| **有条件通过** | 存在 **△** 项 | **不得**作为最终通过；须修复并重新评审直至全 ✓（见 `pattern-review-fix-loop`；**禁止**「并行进入下一阶段」） |
| **不通过** | 存在 **✗** 项 | 必须修改 PRD，重新评审 |

> **与旧版表述的差异**：「有条件通过即可并行推进」易与 **`pattern-review-fix-loop`** 冲突；**以 `prd-review` SKILL **v1.4+** 与 memory 为准**。

## 四、评审维度参考框架（详细版）

### 4.1 价值层评审

| 维度 | 评审要点 | 检查方法 | 常见不通过原因 |
|-----|---------|---------|---------------|
| **需求真实性** | PRD 是否有真实的需求场景？是否符合需求价值逻辑（解决谁的什么问题）？是否有值得做的必要性？ | 检查「1. 需求来源与背景」「2. 问题价值分析」章节 | 仅有功能描述，无需求场景；受益方不明确；价值阐述模糊 |
| **技术 ROI** | 技术方案是否能确定实现需求目标？实现过程是否 ROI 合理（投入产出比）？ | 检查迭代目标与成功衡量是否合理 | 目标不可衡量；投入成本（人力/时间）远超价值 |

### 4.2 执行层评审

| 维度 | 评审要点 | 检查方法 | 常见不通过原因 |
|-----|---------|---------|---------------|
| **产品方案完整性** | 产品设计方案是否贴合场景需求？是否有完整对应的需求功能点拆解清单？产品流程链路、概念模型、界面设计、交互设计、数据处理、埋点采集、非功能性要求等不同维度是否均清晰阐述且逻辑自洽？ | 检查「5. 产品方案要点」章节是否覆盖各维度 | 缺少关键维度（如只有界面设计，无数据处理）；维度间逻辑不自洽 |
| **产品方案合理性** | 既不存在过度设计的情况，也不存在设计方案无法有效满足场景需求的情况？是否有明确的结果指标和过程衡量指标体系设计？ | 检查是否有过度设计（如为简单需求设计复杂流程）或设计不足 | 过度设计：简单需求复杂化；设计不足：关键场景未覆盖 |
| **技术方案匹配度** | 技术方案是否 100% 满足 PRD 期望实现需求和目标？总体架构设计是否合理、细节功能技术实现方案是否清晰、逻辑自洽？ | 检查「6. 与需求/PRD 的对应」章节；检查技术方案是否覆盖 PRD 所有功能点 | 技术方案遗漏 PRD 功能点；架构设计与需求不匹配 |
| **技术方案可行性** | 实现路径选型 ROI 最优？有落地可行性且可预见能稳定高效运行？ | 检查技术选型理由；检查是否有可行性分析 | 技术选型过于超前，团队无能力实现；无可行性分析 |
| **行业最佳实践** | 整体是否符合 PRD/技术方案撰写的行业最佳实践标准（无论前端产品 PRD、后端产品 PRD、数据产品 PRD，适用不同行业）？ | 对照本 REFERENCE「一、最小结构」和「二、评审纪要内容结构」检查 | 结构缺失（如只有功能列表，无价值分析）；场景描述缺失 |

## 五、常见问题与处理建议

### 5.1 PRD 过于简略

**现象**: PRD 只有功能列表，缺少价值分析、场景描述、验收标准等

**处理**:
- 判定：✗ 不通过
- 修改建议：按「一、最小结构」补全 8 类内容
- 引用依据：行业最佳实践 - PRD 必须回答「为什么做、给谁做、怎么做、怎么验收」

### 5.2 场景描述缺失

**现象**: 有功能列表，但没有「谁-在什么情况-为什么需要」场景描述

**处理**:
- 判定：✗ 不通过
- 修改建议：每个功能点补充 1～3 个典型场景
- 引用依据：`agentsystem/sys-root/lib/skills/request-analysis/REFERENCE/迭代需求说明-PRD最小结构与自检.md` 自检项 #5b2

### 5.3 验收标准模糊

**现象**: 验收标准写「功能可用」「体验良好」等不可衡量描述

**处理**:
- 判定：△ 有条件通过
- 修改建议：改为可验证、可衡量的标准，如「页面加载时间 < 2s」「支持 1000 并发」

### 5.4 与 Specs 不一致

**现象**: PRD 功能列表与 specs 中 Requirements/Scenario 不一致

**处理**:
- 判定：✗ 不通过（严重）/ △ 有条件通过（轻微差异）
- 修改建议：对齐 PRD 与 specs，确保「需求 → spec → 实现 → 验收」全链路可追溯

### 5.5 命名不规范

**现象**: PRD 文件命名为 `需求.md`、`新功能.md` 等无识别性名称

**处理**:
- 判定：△ 有条件通过
- 修改建议：按「二、PRD 文档命名规范」重命名为 `PRD-[change-id]-[关键词].md`

### 5.6 编码/测试落地不足（附录 C 为空）

**现象**: PRD 与附录 A 在产品层通过，但无 **页面路由、mock 字段、可执行验收步骤**，开发与测试无法唯一对齐「做什么、测什么」。

**处理**:
- 判定：△ 或 ✗（视是否完全无等效引用而定）
- 修改建议：在 PRD 增表、或补全 `design.md`（路由+数据）、`specs` Scenario，并在纪要 **附录 C** 引用；见 `prd-review/SKILL.md` **§3.8**
