# 技术方案评审技能调用规范

## 概述

本规范明确 `architecture-review` 和 `technical-design-review` 两个技能的职责边界、使用场景和调用顺序，确保技术方案评审工作完整、有序地进行。

## 两个技能的职责对比

| 对比维度 | architecture-review | technical-design-review |
|---------|---------------------|------------------------|
| **核心关注** | 产出物规范性、内容完整性 | 系统实现质量、生产就绪性 |
| **评审对象** | **两层文档**：design-brief.md（概述层）+ design.md（实现层） | **两层文档**：先看brief（架构），再看design（实现） |
| **检查重点** | brief: 架构图、产出物清单、可理解性；design: 6章节结构、代码示例、与brief一致性 | design: 代码实现是否与brief架构一致、接口协议、安全性、稳定性、可观测性 |
| **产出结果** | 评审纪要（通过/有条件通过/不通过） | 完成度评估（0-100%量化评分） |
| **使用阶段** | Step 4: 技术方案评审节点 | 技术方案自查、生产就绪验证 |

## 文档分层评审规范

### 概述层 (design-brief.md) - 先看

**由 architecture-review 主要负责检查**：
- 文档存在性（必须先产出brief）
- 架构图、流程图完整性
- 产出物清单、执行人映射、输入信息源
- 可理解性（非技术人员能读懂）
- 目录索引（指向design.md的快速定位）

**technical-design-review 也需关注**：
- brief中的架构设计是否合理
- brief中的流程设计是否清晰
- 作为评审design.md的基准

### 实现层 (design.md) - 后看

**由 technical-design-review 主要负责检查**：
- 代码示例是否完整
- 接口协议是否清晰
- 实现是否与brief中的架构一致
- 安全性、稳定性、可观测性模块

**architecture-review 也需关注**：
- 6章节结构是否完整
- 与PRD一致性
- 与brief一致性

## 详细职责划分

### architecture-review 技能职责

**负责回答的问题**:
- 技术方案文档结构是否完整？（6个章节）
- 与PRD是否一致？是否有遗漏需求？
- 与project.md是否一致？是否符合项目规范？
- 文档命名是否规范？
- 接口定义是否清晰？
- 关键流程是否有图示？

**不回答的问题**:
- 是否有安全性模块？（不归它管）
- 是否有熔断限流？（不归它管）
- 代码实现是否完整？（不归它管）

### technical-design-review 技能职责

**负责回答的问题**:
- 技术方案是否生产就绪？
- 是否有安全性模块（SecurityValidator/AuthChecker/AuditLogger）？
- 是否有稳定性模块（RateLimiter/CircuitBreaker/RetryPolicy）？
- 是否有可观测性模块（TraceContext/Logging/Metrics）？
- 是否完成P0-P3性能优化？
- 是否存在框架替代陷阱（TODO/空函数/伪实现）？
- 完成度是多少？为什么不是100%？

**不回答的问题**:
- 文档结构是否完整？（不归它管）
- 与PRD是否一致？（不归它管）
- 命名是否规范？（不归它管）

## 调用场景与顺序

### 场景一: Step 4 技术方案评审节点（标准流程）

```
Step 4: 技术方案评审节点
    ↓
【必须】调用 architecture-review
    - 检查文档结构（6章节）
    - 检查与PRD一致性
    - 检查命名规范
    ↓
若 architecture-review 不通过
    → 修改技术方案文档 → 重新评审
    ↓
若 architecture-review 通过
    ↓
【推荐】调用 technical-design-review
    - 检查生产就绪性（安全/稳定/观测）
    - 检查完成度（0-100%量化）
    - 检查框架替代陷阱
    ↓
若 technical-design-review < 90分
    → 补充缺失模块 → 重新评估
    ↓
若 technical-design-review ≥ 90分
    ↓
进入 Step 5: 编码实现
```

### 场景二: 技术方案深度自查

```
用户问: "技术方案是否还有错漏？"
    ↓
判断: 用户关心的是"实现质量"而非"文档结构"
    ↓
【调用】technical-design-review
    - 10维度评审
    - 生产就绪检查
    - 完成度评估
    ↓
若发现文档结构问题
    - 顺带提示"建议也检查文档规范性"
```

### 场景三: 生产就绪验证

```
用户问: "技术方案是否生产就绪？"
    ↓
【直接调用】technical-design-review
    - 检查安全性模块
    - 检查稳定性模块
    - 检查可观测性模块
    ↓
产出: 生产就绪评估报告
```

### 场景四: 快速区分调用

| 用户问题 | 调用技能 | 原因 |
|---------|---------|------|
| "文档结构是否完整？" | architecture-review | 文档规范性 |
| "与PRD是否一致？" | architecture-review | 一致性检查 |
| "命名是否规范？" | architecture-review | 文档规范性 |
| "是否有安全/稳定/观测模块？" | technical-design-review | 生产就绪性 |
| "代码实现是否完整？" | technical-design-review | 实现质量 |
| "是否生产就绪？" | technical-design-review | 生产就绪性 |
| "完成度是多少？" | technical-design-review | 量化评估 |
| "为什么不是100%？" | technical-design-review | 深度评估 |

## 协同工作模式

### 模式一: 顺序调用（推荐）

在 Step 4 技术方案评审节点，建议先执行 `architecture-review`，再执行 `technical-design-review`：

```
architecture-review（文档规范性）
    ↓
technical-design-review（系统实现质量）
    ↓
完整的技术方案评审
```

**优势**:
- 层次分明：先确保"有形"（文档规范），再确保"有质"（实现质量）
- 效率更高：文档问题先解决，避免在代码层面反复修改
- 职责清晰：各自专注自己的领域

### 模式二: 独立调用

根据用户明确的问题，独立调用相应技能：

```
用户问: "文档结构是否完整？"
→ 仅调用 architecture-review

用户问: "是否生产就绪？"
→ 仅调用 technical-design-review
```

### 模式三: 互补调用

当使用一个技能发现需要另一个技能时，进行互补：

```
执行 technical-design-review
    ↓
发现: "建议同时检查文档与PRD的一致性"
    ↓
提示用户: "是否需要同时检查文档规范性？"
    ↓
若用户确认 → 调用 architecture-review
```

## 常见错误与避免

### 错误一: 混淆调用

**错误场景**: 用户问"是否生产就绪"，却调用了 `architecture-review`

**后果**: 无法回答安全性、稳定性、可观测性等问题

**避免方法**: 根据关键词快速对照表判定

### 错误二: 重复调用

**错误场景**: 两个技能都检查"接口定义"

**实际情况**:
- `architecture-review`: 检查"接口是否定义清晰"（文档层面）
- `technical-design-review`: 检查"接口实现是否健壮"（代码层面，如限流熔断）

**不是重复，是不同层面的检查**

### 错误三: 遗漏调用

**错误场景**: 只调用 `architecture-review`，不调用 `technical-design-review`

**后果**: 技术方案文档规范，但代码实现缺少生产就绪模块

**避免方法**: 在 Step 4 节点，两个技能都建议调用
