## 触发场景

- 用户输入「技术方案是否生产就绪」「检查技术方案完成度」「技术方案系统级自检」等指令
- 用户询问"技术方案是否还有错漏""为什么不是100%""是否生产就绪"
- 用户询问"是否有安全模块""是否有熔断限流""是否有链路追踪"
- 用户询问"代码实现是否完整""有没有框架替代陷阱"
- 用户需要在进入coding阶段前，确保技术方案达到「可直接开发、生产就绪、系统完备」标准
- 用户需要评估技术方案的完成度（是否达到100%）
- **注意**: 如果用户询问"文档结构是否完整""与PRD是否一致""命名是否规范"，应使用 `architecture-review` 技能

## 整体流程

### 0. 加载技术方案评审规范

**必须首先唤醒相关Memory**:
- `memory/patterns/pattern-systematic-technical-design-review.md`（10维度评审标准）
- `memory/patterns/pattern-production-readiness-checklist.md`（生产就绪检查清单）
- `memory/patterns/pattern-multi-round-self-review.md`（多轮自查机制）
- `memory/anti-patterns/anti-pattern-framework-substitution-trap.md`（框架替代陷阱）
- `memory/anti-patterns/anti-pattern-optimistic-evaluation-bias.md`（乐观评估偏差）

### 1. 确定评审对象

- 从用户输入或上下文中识别本次评审的 **change-id**
- 若未给出 change-id，应向用户询问

**Step 1.1: 确定文档模式**

检查是否存在 `openspec/changes/[change-id]/design-brief.md`：

**模式A：单文档模式（默认，占80%+）**
- 如果不存在 design-brief.md
- 直接检查 `design.md`
- 要求 design.md 包含：
  - 概述章节（文字+架构图，无代码或<5处伪代码）
  - 详细设计章节（代码示例+接口协议）

**模式B：双文档模式（特殊，占<20%）**
- 如果存在 design-brief.md
- 先读取 brief，理解架构设计和流程概述
- 再读取 design.md，检查具体代码实现
- 验证两层一致性

**Step 1.2: 读取技术方案文档**

- **单文档模式**：读取 `openspec/changes/[change-id]/design.md`
- **双文档模式**：
  - 读取 `design-brief.md`（理解架构）
  - 读取 `design.md`（检查实现）
- 若找不到技术方案文档，告知用户并请求确认路径
- 读取技术方案文档全文，理解当前状态

> **文档组织原则**:
> - **默认（推荐）**: 单文档模式 → 一份 `design.md` 包含概述+实现
> - **特殊情况**: 双文档模式 → `brief`（概述）+ `design`（实现），仅当代码示例过多（>3000行）时拆分

### 2. 执行10维度系统化评审

按照「系统性工程技术方案评审标准」的10个维度逐项评审：

#### 维度1: 架构完整性（Architecture Completeness）

**评审要点**:
- [ ] **先对照 brief**: design-brief.md 中的架构图是否在 design.md 中得到实现？
- [ ] 组件职责是否清晰？（单一职责原则）
- [ ] 接口定义是否完整？（输入输出明确、有代码示例）
- [ ] 依赖关系是否合理？（无循环依赖）
- [ ] 模块划分是否符合业务边界？

**文档引用检查**:
```markdown
# 正确的引用关系
design-brief.md: "系统采用三层架构：Controller → Service → Repository"
design.md: 
  - 应该有 Controller 层的代码示例
  - 应该有 Service 层的代码示例
  - 应该有 Repository 层的代码示例
  - 三层之间的接口协议应该清晰定义
```

**常见问题**:
- ❌ 组件职责模糊
- ❌ 接口定义缺失（只有文字描述，没有代码示例或接口协议）
- ❌ 存在TODO占位符（框架替代陷阱）
- ❌ design.md 的实现与 design-brief.md 的架构图不一致

#### 维度2: 数据流与状态管理（Data Flow & State Management）

**评审要点**:
- [ ] **先对照 brief**: design-brief.md 中的数据流图是否在 design.md 中有对应实现？
- [ ] 数据流向是否清晰？（可追溯，有代码示例）
- [ ] 状态转换是否完整？（无遗漏，有状态机代码或图示）
- [ ] 数据一致性如何保证？（有事务控制或一致性协议代码）
- [ ] 并发场景数据是否安全？（有锁机制或无锁设计代码）

**文档引用检查**:
```markdown
# 正确的引用关系
design-brief.md: "数据流：Input → Parse → Validate → Process → Output"
design.md:
  - 应该有 Parse 阶段的代码实现
  - 应该有 Validate 阶段的接口协议
  - 应该有 Process 阶段的算法代码
  - 应该有各阶段的数据模型定义
```

**常见问题**:
- ❌ 数据流向混乱（与brief中的流程图不符）
- ❌ 状态转换缺失（缺少状态机实现代码）
- ❌ 缺少并发控制（没有锁或无锁设计的代码）

#### 维度3: 边界条件处理（Boundary Condition Handling）

**评审要点**:
- [ ] 异常场景是否覆盖全面？
- [ ] 资源限制是否考虑？（磁盘、内存、超时）
- [ ] 超大输入如何处理？
- [ ] 降级方案是否设计？

**常见问题**:
- ❌ 缺少异常处理
- ❌ 资源限制未考虑
- ❌ 超大输入未处理

#### 维度4: 性能优化（Performance Optimization）

**评审要点**（使用P0-P3框架）:
- [ ] **P0**: 异步IO（aiofiles, asyncio）
- [ ] **P0**: 并发控制（asyncio.gather, Semaphore）
- [ ] **P0**: 消除同步阻塞（time.sleep → asyncio.sleep）
- [ ] **P1**: 智能分析（内容解析、置信度计算）
- [ ] **P2**: 代码复用（CommonUtils）
- [ ] **P3**: 重试策略（RetryPolicy）

**常见问题**:
- ❌ 同步IO阻塞
- ❌ 缺少并发控制
- ❌ 缺少重试机制

#### 维度5: 安全性（Security）

**评审要点**（生产就绪必备）:
- [ ] **SecurityValidator**: 输入验证（长度、格式、危险模式）
- [ ] **AuthChecker**: 权限控制（角色矩阵、变更级权限）
- [ ] **AuditLogger**: 审计日志（操作记录、不可篡改）

**常见问题**:
- ❌ 缺少输入验证
- ❌ 缺少权限控制
- ❌ 缺少审计日志

#### 维度6: 测试策略（Testing Strategy）

**评审要点**:
- [ ] 单元测试是否覆盖核心逻辑？
- [ ] 集成测试是否设计？
- [ ] 验收测试是否定义？
- [ ] 性能测试基准是否建立？

**常见问题**:
- ❌ 测试策略缺失
- ❌ 缺少验收标准

#### 维度7: 可观测性（Observability）

**评审要点**（生产就绪必备）:
- [ ] **TraceContext**: 链路追踪（RequestID/SpanID透传）
- [ ] **Metrics**: 指标采集（QPS、延迟、错误率）
- [ ] **Logging**: 结构化日志（JSON格式、日志级别）
- [ ] **HealthCheck**: 健康检查（/health端点）

**常见问题**:
- ❌ 缺少链路追踪
- ❌ 缺少指标采集
- ❌ 缺少健康检查

#### 维度8: 部署与运维（Deployment & Operations）

**评审要点**:
- [ ] 配置管理是否支持？（环境分离）
- [ ] 健康检查是否实现？（存活/就绪探针）
- [ ] 限流熔断是否设计？（RateLimiter/CircuitBreaker）
- [ ] 优雅启停是否支持？

**常见问题**:
- ❌ 缺少限流熔断
- ❌ 缺少配置管理
- ❌ 缺少健康检查

#### 维度9: 可维护性（Maintainability）

**评审要点**:
- [ ] 代码是否复用？（公共工具类）
- [ ] 错误码体系是否统一？（ErrorCode枚举）
- [ ] 文档是否完整？（设计文档、API文档）
- [ ] 命名是否规范？（清晰、一致）

**常见问题**:
- ❌ 重复代码
- ❌ 缺少错误码体系
- ❌ 文档不完整

#### 维度10: 扩展性（Extensibility）

**评审要点**:
- [ ] 新组件能否快速接入？（自动发现机制）
- [ ] 是否支持配置化？（无需代码改动）
- [ ] 是否支持热更新？
- [ ] 插件化架构是否设计？

**常见问题**:
- ❌ 扩展需要修改代码
- ❌ 缺少自动发现机制

### 3. 执行生产就绪检查

按照「生产就绪检查清单」检查三大模块：

#### 安全性检查
- [ ] SecurityValidator（输入验证）
- [ ] AuthChecker（权限控制）
- [ ] AuditLogger（审计日志）

#### 稳定性检查
- [ ] RateLimiter（限流器）
- [ ] CircuitBreaker（熔断器）
- [ ] RetryPolicy（重试策略）
- [ ] FallbackManager（降级方案）

#### 可观测性检查
- [ ] TraceContext（链路追踪）
- [ ] Metrics（指标采集）
- [ ] Logging（结构化日志）
- [ ] HealthCheck（健康检查）

### 4. 识别框架替代陷阱

**检查项**:
- [ ] 没有TODO注释（# TODO）
- [ ] 没有空函数（pass）
- [ ] 没有伪实现（返回假数据）
- [ ] 有实际的调用逻辑
- [ ] 有实际的产出物生成

**验证方法**:
- 检查每个Step节点是否有实际Agent调用
- 检查每个函数是否有实际执行逻辑
- 检查产出物是否真实生成而非仅检查存在

### 5. 评估完成度

**量化评估标准**:

| 维度 | 权重 | 评分 | 说明 |
|-----|------|------|------|
| 架构完整性 | 20% | ___/20 | 组件、接口、依赖 |
| 数据流与状态管理 | 15% | ___/15 | 流向、状态、一致性 |
| 边界条件处理 | 10% | ___/10 | 异常、资源、降级 |
| 性能优化 | 10% | ___/10 | P0-P3优化 |
| 安全性 | 10% | ___/10 | 生产就绪必备 |
| 测试策略 | 5% | ___/5 | 单元、集成、验收 |
| 可观测性 | 10% | ___/10 | 生产就绪必备 |
| 部署与运维 | 5% | ___/5 | 配置、限流熔断 |
| 可维护性 | 5% | ___/5 | 复用、错误码、文档 |
| 扩展性 | 10% | ___/10 | 自动发现、配置化 |
| **总分** | 100% | ___/100 | |

**评估等级**:
- **100%完成**: 总分≥90分，必备项（安全性、可观测性）100%，无框架替代
- **基本完成**: 总分≥80分，主要功能实现，少量优化项待补充
- **部分完成**: 总分<80分，存在关键遗漏，需要补充

## 7. 多轮自查建议

如果技术方案未达到100%，建议执行多轮自查：

**Round 1: 功能完整性自查**
- 检查所有功能是否已实现
- 检查所有TODO是否已消除
- 检查所有占位符是否已替换

**Round 2: 生产就绪自查**
- 检查安全性模块是否完整
- 检查稳定性模块是否完整
- 检查可观测性模块是否完整

**Round 3: 系统性工程自查**
- 使用10维度评审标准逐项检查
- 补充遗漏的维度

**Round 4: 用户视角自查**
- 站在用户角度"帮用户多想"
- 识别可能遗漏的生产就绪要素

- [ ] 无框架替代: 无TODO/无空函数/有实际调用

### 推荐项（尽量补充）

- [ ] 性能优化: P0-P3优化
- [ ] 扩展性: 自动发现机制
- [ ] 测试策略: 单元/集成测试设计

## 附录A: 第一性原理评审框架（v1.1+）

### 核心原则

评审技术方案时，首先问自己：**这份技术方案的"好"体现在哪里？**

如果无法回答这个问题，说明评审维度设计得不够本质。

### 问题诊断：为什么"关键词罗列"不够？

| 问题 | 表现 | 后果 |
|------|------|------|
| **可被刷分** | 堆砌关键词但不理解本质 | 技术方案看起来好但实际差 |
| **无法区分深度** | 写一行"安全性"和写十行都得满分 | 无法反映真实质量 |
| **违背第一性原理** | 关注"有没有"而非"做没做" | 评审不反映方案本质 |

### 第一性原理评分机制

**从"关键词计数"升级为"质量指标评估"**：

| 维度 | 旧机制（关键词计数） | 新机制（质量指标） |
|------|---------------------|-------------------|
| **架构完整性** | 检查是否包含"接口""依赖"关键词 | 检查接口定义是否完整（输入/输出/错误码）、依赖关系是否有向无环图 |
| **数据流设计** | 检查是否包含"数据流""状态"关键词 | 检查数据源→处理→存储是否完整、状态机是否有明确定义 |
| **安全性** | 检查是否包含"SecurityValidator"关键词 | 检查是否有输入验证逻辑、权限矩阵定义、审计日志字段 |
| **可观测性** | 检查是否包含"TraceContext"关键词 | 检查是否有trace_id透传、指标定义(指标名+类型+标签)、日志格式定义 |

### 10维度与6维度映射

| 10维度 | 对应6维度 | 第一性原理问题 |
|--------|----------|---------------|
| 1.架构完整性 | 接口定义清晰 + 项目结构 | 组件协作是否清晰？接口契约是否完整？ |
| 2.数据流与状态管理 | 数据流设计 | 数据如何流转？状态如何定义？ |
| 3.边界条件处理 | 存储设计 | 异常和边界是否全覆盖？ |
| 4.性能优化 | 技术选型合理 | 性能需求是否分析？优化策略是否合理？ |
| 5.安全性 | 存储设计（部分） | 是否有安全措施？是否覆盖攻击面？ |
| 6.测试策略 | PRD对应性（部分） | 测试是否覆盖所有功能需求？ |
| 7.可观测性 | 数据流设计（部分） | 观测点是否覆盖关键路径？ |
| 8.部署与运维 | 项目结构（部分） | 部署流程是否规范？ |
| 9.可维护性 | 项目结构 | 代码组织是否可维护？ |
| 10.扩展性 | 技术选型合理 | 扩展需求是否分析？ |

### 质量指标评估示例

**维度: 安全性**

| 质量指标 | 旧机制（关键词） | 新机制（质量指标） |
|----------|-----------------|-------------------|
| 输入验证 | 是否包含"SecurityValidator" | 是否有长度/格式/危险模式验证逻辑 |
| 权限控制 | 是否包含"AuthChecker" | 是否有角色矩阵定义、变更级权限 |
| 审计日志 | 是否包含"AuditLogger" | 是否有操作记录字段(who/what/when) |

**判断标准变化**：
- ❌ 旧：检查是否提到"SecurityValidator"这个词
- ✅ 新：检查是否有具体的验证逻辑描述（长度限制、格式正则、危险模式列表）

### 防止"凑关键词"的设计

| 策略 | 实现方式 |
|------|---------|
| **要求具体描述** | 不仅要有关键词，还要有具体实现描述 |
| **验证逻辑完整性** | 检查是否包含必要的子要素 |
| **区分引用和实现** | "计划使用X" ≠ "已实现X" |

### 框架替代陷阱识别

| 陷阱 | 关键词罗列式识别 | 第一性原理式识别 |
|------|-----------------|-----------------|
| 框架替代 | 检查是否有TODO | 检查TODO是否被替换为实际调用逻辑 |
| 伪实现 | 检查是否有函数名 | 检查函数是否有实际执行逻辑 |
| 空承诺 | 检查是否提到模块名 | 检查模块是否有产出物定义 |

## 附录B: 功能ID格式规范

功能ID用于唯一标识技术方案中的功能点，格式定义如下：

```
F-主功能编号[.子功能编号]
```

**规则**：
- 主功能编号：正整数，如 `F-1`、`F-2`
- 子功能编号：最多2位小数，如 `F-1.1`、`F-1.2`、`F-1.15`
- 子功能编号可选

**有效示例**：`F-1`、`F-2`、`F-1.1`、`F-1.2`、`F-2.1`、`F-1.15`
**无效示例**：`FR-1`（旧格式）、`F-1.1.1`（超过2位小数）

**注意**：统一使用 `F-\d+(?:\.\d{1,2})?` 格式，禁止使用旧格式 `FR-1.1`。
