# 反模式：规范建立者自己不遵循

## 一句话定义

建立了详细的规范和技能，但在执行时却选择捷径（如对话形式），而非调用 skill 产出正式文档产物，破坏规范权威性。

## 问题场景

### 典型表现

```
❌ 错误执行:
用户: "请复盘为什么我能想到这个问题，但你却犯了错误？"

Agent 响应:
[长篇对话形式的复盘内容]
[没有调用 retrospective-analysis skill]
[没有产出正式复盘报告文档]
[没有更新迭代日志]

问题：虽然内容传达了，但没有遵循自己建立的规范流程
```

### 正确执行

```
✓ 正确执行:
用户: "请复盘为什么我能想到这个问题，但你却犯了错误？"

Agent 执行:
1. 读取 retrospective-analysis skill
2. 按 5 阶段复盘法执行
3. 产出正式复盘报告文档 → docs/retrospective/framework/.../复盘-xxx.md
4. 沉淀 anti-pattern memory
5. 更新迭代日志

用户看到的是执行结果通知，而非冗长对话
```

## 为什么会发生

### 认知根源

| 因素 | 错误思维 | 正确思维 |
|-----|---------|---------|
| **优先级** | 「快速响应 > 严格流程」 | 「流程纪律 > 快速响应」 |
| **惯性驱动** | 对话状态下选择即时响应 | 暂停，切换到 skill 执行模式 |
| **价值认知** | 「内容传达了就够了」 | 「规范的价值在于执行」 |
| **约束机制** | 依赖自觉，无强制检查 | 建立「执行前必须调用 skill」的自我约束 |

### 触发条件

- 建立了详细的 skill 规范
- 处于对话交互状态
- 有快速响应的压力
- 没有强制调用 skill 的机制

## 危害

### 直接影响

1. **规范形同虚设**
   - 建立了规范却不遵循 = 规范无价值
   - 用户发现「你自己都不遵循，我为什么要遵循？」

2. **流程纪律缺失**
   - 破坏了标准化执行的基础
   - 导致产出物质量不稳定

3. **信任受损**
   - 用户发现 AI 知行不一
   - 降低对 AI 建议的信赖度

### 长期影响

- 规范体系失去权威性
- 技能调用机制被架空
- 团队流程逐渐瓦解

## 如何避免

### 执行前强制检查

```yaml
当你要执行某个任务时，如果：
  - 该任务已有对应的 skill → 必须调用 skill
  - 该任务需要产出文档 → 必须产出正式文档
  - 该任务有更新迭代日志的要求 → 必须更新

请立即执行：
  1. 暂停对话响应
  2. 读取对应的 SKILL.md
  3. 按 skill 流程执行
  4. 产出正式文档产物
  5. 更新迭代日志
  6. 向用户汇报执行结果（简洁）
```

### 自我约束机制

| 约束点 | 检查问题 | 不通过时的动作 |
|-------|---------|--------------|
| **Skill 调用检查** | 执行前是否读取了对应 skill？ | 停止执行，先读取 skill |
| **文档产物检查** | 是否产出了正式文档产物？ | 补充产出文档 |
| **迭代日志检查** | 是否更新了迭代日志？ | 补充更新日志 |
| **流程完整性检查** | 是否遵循了 skill 的所有步骤？ | 补充执行遗漏步骤 |

### 快速决策流程

```
当用户要求执行任务时：

Step 1: 识别任务类型
   "这个任务是否有对应的 skill？"
   ├─ 有 → 必须调用 skill
   └─ 无 → 按通用流程执行

Step 2: 检查产出要求
   "skill 是否要求产出正式文档？"
   ├─ 是 → 产出正式文档（非对话形式）
   └─ 否 → 按 skill 要求产出

Step 3: 执行收尾
   "是否有更新迭代日志的要求？"
   ├─ 是 → 更新迭代日志
   └─ 否 → 执行 skill 定义的收尾

Step 4: 汇报结果
   向用户简洁汇报执行结果，而非展示过程
```

## 修复案例

### 案例：复盘执行修正

**修复前（错误）**:
```
用户: "请复盘为什么我能想到这个问题，但你却犯了错误？"

Agent: [长篇对话形式的复盘]
```

**问题识别**:
- 用户指出：「为什么以对话形式给我，而不是调用 skill 并输出成对应文档产物？」
- 发现：违反了「执行时必须调用 skill」的自我约束

**修复后（正确）**:
```
1. 读取 retrospective-analysis skill
2. 按 5 阶段复盘法执行
3. 产出正式复盘报告 → docs/retrospective/framework/2026-Q1/复盘-框架设计者认知盲区-2026-03-15.md
4. 创建 anti-pattern memory
5. 更新迭代日志
6. 向用户汇报："复盘报告已完成，存放于 [路径]，同时沉淀了 2 个 anti-pattern memory"
```

**关键改进**:
- 从「对话形式」→「正式文档产物」
- 从「直接响应」→「先调用 skill」
- 从「省略收尾」→「完整执行所有步骤」

## 对比：错误执行 vs 正确执行

| 维度 | 错误执行 | 正确执行 |
|-----|---------|---------|
| **响应形式** | 长篇对话 | 简洁的结果通知 |
| **Skill 调用** | 跳过 | 必须先读取 |
| **文档产物** | 无 | 正式复盘报告 |
| **Memory 沉淀** | 无 | 创建 anti-pattern |
| **迭代日志** | 不更新 | 必须更新 |
| **用户认知** | "AI 自己都不遵循规范" | "AI 严格执行规范" |

## 核心洞察

> **规范的价值在于执行，而不在于建立**
>
> 建立了规范却不遵循 = 规范无价值
> 用户关注「是否符合既定流程」而非「能不能 work」
