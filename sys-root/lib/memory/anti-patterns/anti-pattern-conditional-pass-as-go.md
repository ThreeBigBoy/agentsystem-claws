# 有条件通过即放行反模式（Conditional-Pass-as-Go Anti-Pattern）

## 一句话定义

将「有条件通过」误解为「可以进入下一阶段」，忽略修复和重新评审环节，导致问题流入后续工作。

---

## 症状与表现

### 典型症状

| 症状 | 示例 |
|------|------|
| 评审纪要有「有条件通过」但无后续修复记录 | PRD 评审为「有条件通过」，但迭代日志直接记录「进入 Step 3」 |
| 评审后直接「并行进入下一阶段」 | 技术方案评审为「有条件通过」，同时开始编码实现 |
| 评审发现的问题在后续文档中未提及 | 评审发现「需补充竞品调研」，但后续 PRD 版本未更新 |
| 「有条件通过」的结论长期未关闭 | 评审纪要在「有条件通过」状态停留数周 |
| 下一阶段发现问题追溯到上一阶段 | 编码时发现 PRD 不清晰，追溯发现 PRD 评审为「有条件通过」 |

### 代码/文档层面的表现

```markdown
# ❌ 错误示例：迭代日志
2026-03-14: PRD 评审完成，综合判定：有条件通过（2 项 minor 建议）
2026-03-14: 进入 Step 3，技术方案设计开始   ← 错误！未修复即进入下一阶段

# ✅ 正确示例：迭代日志
2026-03-14: PRD 评审完成，综合判定：有条件通过（2 项 minor 建议）
2026-03-15: 修复 PRD minor 问题（补充竞品调研、明确 Non-Goals）
2026-03-15: PRD 重新评审完成，综合判定：通过
2026-03-16: 进入 Step 3，技术方案设计开始   ← 正确！修复并重新评审后才进入
```

## 危害与影响

### 直接危害

| 危害 | 影响 | 严重程度 |
|------|------|---------|
| **质量问题流入下一阶段** | 基于不完善的文档工作，产生更多问题 | 🔴 高 |
| **技术债务积累** | Minor 问题积累，后期修复成本指数级增长 | 🔴 高 |
| **返工成本增加** | 后续阶段发现问题需回溯修改，浪费工作量 | 🔴 高 |
| **质量门禁失效** | 评审流于形式，失去质量保障作用 | 🔴 高 |

### 长期影响

- **团队信任下降**: 评审机制失去公信力，大家不再重视评审
- **质量标准模糊**: 「有条件通过」到底能不能进入下一阶段？标准不一
- **知识传递失效**: 评审发现的问题未修复，新加入的成员继承有问题的文档
- **流程改进困难**: 无法识别流程中的真实问题，因为问题都被「有条件通过」掩盖

## 根因分析

### 为什么会发生？

#### 原因 1：对「有条件通过」的理解偏差

**错误理解**:
> 「有条件通过」= 「基本可用，可以进入下一阶段，小问题后续优化」

**正确理解**:
> 「有条件通过」= 「还有问题需要修复，修复后重新评审通过才能进入下一阶段」

#### 原因 2：流程设计缺陷

- 流程缺少「评审→修复→重新评审」的循环节点
- 默认了「单向通过」的线性模型，而非「闭环修复」的循环模型
- 质量门禁标准不清晰：未明确区分「100% 通过」vs「有条件通过的后续处理」

#### 原因 3：效率优先思维

- 为了流程的「顺畅」，牺牲了「严谨」
- 「并行开发」被误用为「并行进入下一阶段」，忽略了质量依赖关系
- 短期效率（快速进入下一阶段）vs 长期质量（问题清零）的权衡失误

#### 原因 4：缺乏检查机制

- 没有自动化或人工的检查点：「前一阶段评审结论是否为通过？」
- 迭代日志记录不规范，无法追溯评审修复循环
- 缺乏强制阻断机制：即使发现「有条件通过」，也可强行进入下一阶段

---

## 避免方法

### 方法 1：建立评审修复循环（Pattern）

使用 `pattern-review-fix-loop` 模式：
- 「有条件通过」必须经过「修复→重新评审」循环
- 只有「100% 通过」才能进入下一阶段
- 在迭代日志中明确记录「重新评审」动作

### 方法 2：升级质量门禁标准

| 评审结论 | 是否可以进入下一阶段 |
|---------|-------------------|
| ✓ 通过 | ✅ **可以** |
| △ 有条件通过 | ❌ **不可以**（必须修复并重新评审） |
| ✗ 不通过 | ❌ **不可以**（必须修复并重新评审） |

### 方法 3：建立检查机制

**进入下一阶段前的强制检查**:
```markdown
检查项：
1. 前一阶段评审是否完成？ → 必须完成
2. 前一阶段评审结论是什么？ → 必须是「通过」（✓）
3. 是否有「有条件通过」或「不通过」？ → 如有，必须已完成重新评审并转为「通过」
4. 迭代日志是否有「重新评审」记录？ → 如有有条件通过/不通过，必须有

检查失败时的处理：
- 暂停进入下一阶段
- 先完成评审修复循环
- 更新迭代日志
```

### 方法 4：规范迭代日志记录

**必须记录的关键节点**:
1. 首次评审（有条件通过）+ 问题清单
2. 修复完成
3. 重新评审（通过）
4. 进入下一阶段

## 修复方案

### 发现后的修复步骤

**步骤 1**: 识别所有「有条件通过即放行」的案例
- 扫描迭代日志，找出「有条件通过」后直接进入下一阶段的记录
- 列出所有需要「回退修复」的案例

**步骤 2**: 评估影响
- 哪些问题已经流入后续阶段？
- 是否产生了新的问题？
- 返工成本有多大？

**步骤 3**: 回退修复（如必要）
- 如果问题严重，可能需要回退到上一阶段重新修复
- 如果问题轻微，可在当前阶段修复并记录

**步骤 4**: 完善流程
- 更新技能文档（prd-review, architecture-review）
- 更新 pattern（pattern-complete-quality-closed-loop）
- 建立检查机制

**步骤 5**: 培训与宣导
- 向所有 Agent 明确「有条件通过」的正确处理方式
- 在典型案例中复盘，强化记忆

### 自动化检测（建议实现）

```python
# 伪代码：质量门禁检查脚本
def check_quality_gate(change_id, from_step, to_step):
    review_log = get_iteration_log(change_id, f"Step {from_step} 评审")
    
    if review_log.conclusion == "通过":
        return True, "质量门禁通过"
    
    elif review_log.conclusion in ["有条件通过", "不通过"]:
        # 检查是否有重新评审记录
        re_review_log = get_iteration_log(change_id, f"Step {from_step} 重新评审")
        
        if re_review_log and re_review_log.conclusion == "通过":
            return True, "质量门禁通过（经重新评审）"
        else:
            return False, f"质量门禁阻塞：Step {from_step} 评审为「{review_log.conclusion}」，需修复并重新评审"
    
    else:
        return False, "评审结论异常"
```

## 总结

> **「有条件通过即放行」是质量流程中最隐蔽也最危险的反模式。**

> 
> **唯一的解药是严格的评审修复循环：只有「100% 通过」才是真正的通过。**
