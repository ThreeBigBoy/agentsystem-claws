# 反模式：惯性思维陷阱（Inertia Trap）

## 一句话定义

**依赖历史执行模式而不查阅当前规范，导致规范升级后仍执行旧模式，产生执行偏差。**

---

## 问题表现

### 典型症状

| 症状 | 说明 |
|-----|------|
| **"之前这么做"** | 执行理由基于历史而非规范 |
| **规范升级无感** | 不知道或不关心规范已升级 |
| **重复旧错误** | 在相同或类似场景重复之前的不规范执行 |
| **用户多次纠正** | 用户需多次指出才能按规范执行 |

### 诊断信号

- 思考过程出现"上次是这么做的"
- 执行前未查阅相关 skill 或规范文档
- 用户指出后才发现规范已升级
- 无法说明当前执行与规范定义的对应关系

---

## 根本原因

### 思维路径

```
遇到任务 X
    ↓
回忆："上次任务 X 是怎么做的？"
    ↓
找到历史执行模式 M
    ↓
执行模式 M（未检查规范是否已升级）
    ↓
结果：符合模式 M，但可能不符合当前规范
```

### 对比正确思维路径

```
遇到任务 X
    ↓
查阅："任务 X 的当前规范定义是什么？"
    ↓
找到规范定义 S（可能已升级）
    ↓
执行规范 S
    ↓
结果：符合当前规范
```

---

## 危害

| 危害 | 说明 |
|-----|------|
| **规范失效** | 规范定义与执行脱节，规范流于形式 |
| **质量倒退** | 规范升级本应提升质量，惯性思维抵消升级效果 |
| **用户挫败** | 用户需反复纠正同样的问题 |
| **无法持续改进** | 规范升级无法落地，改进循环断裂 |

---

## 避免方法

### 方法 1: 假设规范已升级

**执行前 mindset**:
- 默认假设"规范可能已升级"
- 每次执行前查阅当前规范版本
- 不问"之前怎么做"，问"规范现在怎么说"

### 方法 2: 版本检查

**操作前检查**:
- 查阅相关 skill 的版本号
- 对比上次执行时的版本
- 如有升级，仔细阅读升级内容

### 方法 3: 规范查阅强制化

**将"查阅规范"设为执行前置条件**:
```
执行任何操作前：
1. 查阅相关 skill 或规范文档
2. 确认当前执行步骤
3. 执行
4. 对照规范检查执行结果
```

---

## 与术语定义漂移的关系

| 反模式 | 侧重点 | 关系 |
|-------|--------|------|
| **术语定义漂移** | 对术语的理解偏差 | 原因：不理解规范术语 |
| **惯性思维陷阱** | 对执行方式的依赖 | 原因：不查阅当前规范 |
| **两者结合** | 既不查术语，也不查规范 | 最严重的执行偏差 |

---

## 案例

### 案例：OpenSpec 归档执行缺陷（2026-03-16）

**惯性思维表现**:
- "上次归档标记 tasks.md 完成就可以了"
- 未查阅 OpenSpec.md 归档章节的当前定义
- 不知道规范已将归档定义为"合并+移动"的完整操作

**后果**:
- 遗漏真正的归档操作
- 用户两次追问
- 需要启动复盘分析
