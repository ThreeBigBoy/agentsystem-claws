# 浅层复盘反模式

## 定义

浅层复盘是指在分析问题时仅停留在表面现象，未能深入探究根本原因，导致同样问题反复发生的无效复盘行为。

## 常见表现形式

### 反模式 1: 症状替换型

**表现**：将问题的表现形式当作原因

```
❌ 浅层复盘示例：
问题：系统经常崩溃
结论：服务器负载太高了
措施：增加服务器配置

✅ 正确做法：
继续追问：为什么负载会高？
→ 发现是某个接口被频繁调用且无缓存
→ 追问：为什么没有缓存？
→ 发现是设计时未考虑该场景的高频访问
→ 根本原因：需求分析阶段缺少流量预估环节
```

**警示信号**：
- 复盘结论都是「XX不够」「XX太多了」
- 改进措施都是增加资源或人力
- 同样的问题在不同地方重复出现

### 反模式 2: 指责归罪型

**表现**：将问题归因于个人失误，忽视系统性因素

```
❌ 浅层复盘示例：
问题：代码上线后出现严重bug
结论：开发人员不够细心
措施：要求开发人员更加仔细

✅ 正确做法：
分析：为什么测试环节没发现？
→ 发现测试用例未覆盖该场景
→ 追问：为什么测试用例不完整？
→ 发现需求文档缺少边界条件说明
→ 根本原因：需求分析和测试设计流程存在缺口
```

**警示信号**：
- 复盘会议变成「追责大会」
- 改进措施都是对人的要求（更仔细、更认真）
- 团队成员对复盘产生恐惧和抵触

### 反模式 3: 数据盲区型

**表现**：仅凭主观印象判断，缺乏数据支撑

```
❌ 浅层复盘示例：
问题：项目延期了
结论：开发效率太低
措施：要求团队提高效率

✅ 正确做法：
数据分析：
- 需求变更次数：15次（原计划5次）
- 阻塞等待时长：占总工期30%
- 实际编码时间：仅占总工期40%
→ 根本原因：需求管理和跨团队协作流程问题
```

**警示信号**：
- 结论都是「我感觉」「我觉得」
- 无法提供具体数据支持观点
- 不同人对同一问题的描述差异很大

### 反模式 4: 快速闭环型

**表现**：为了完成复盘任务而仓促结束，追求形式上的闭环

```
❌ 浅层复盘示例：
问题：Agent执行结果不符合预期
结论：提示词不够清晰
措施：优化提示词
（一周后同样问题再次发生）

✅ 正确做法：
深度分析：
- 查看原始提示词和执行输出
- 对比预期输出与实际输出的差异
- 分析模型理解偏差的原因
- 验证改进后的提示词是否真正解决问题
```

**警示信号**：
- 复盘会议时间很短（<30分钟）
- 没有记录详细分析过程
- 改进措施无法验证效果

### 反模式 5: 单一视角型

**表现**：仅从技术或仅从流程角度分析，缺乏多维视角

```
❌ 浅层复盘示例：
（纯技术视角）
问题：服务响应慢
结论：数据库查询效率低
措施：优化SQL，加索引

（遗漏）
- 为什么查询会设计得低效？（设计阶段问题）
- 为什么监控没及时发现？（运维阶段问题）
- 为什么架构不支持水平扩展？（架构阶段问题）
```

**警示信号**：
- 复盘参与者单一角色（全是开发或全是产品）
- 改进措施只涉及一个维度
- 问题在不同维度反复出现

## 如何避免

### 检测清单

在复盘结束时，使用以下清单自检：

- [ ] 是否能从当前结论继续追问「为什么」超过3次？
- [ ] 是否有数据或日志支撑每个关键结论？
- [ ] 改进措施是否针对流程/系统而非个人？
- [ ] 是否分析了问题在不同阶段的表现？
- [ ] 改进后如何验证效果？（有明确指标吗？）
- [ ] 类似问题在其他地方是否可能发生？

### 使用正确的模式替代

| 浅层表现 | 应使用的模式 |
|---------|-------------|
| 症状替换 | `pattern-deep-retrospective-five-whys.md` - 五层穿透分析 |
| 指责归罪 | 系统性视角 + 流程改进 |
| 数据盲区 | `pattern-data-driven-retrospective.md` - 数据驱动复盘 |
| 快速闭环 | `pattern-proactive-retrospective-trigger.md` - 主动触发机制 |
| 单一视角 | 多角色参与 + 全生命周期分析 |

## 典型案例

### 案例：迭代日志记录不完整

**浅层复盘（错误）**
```
问题：很多迭代日志缺少关键字段
结论：大家没有养成记录习惯
措施：在群里提醒，要求以后必须完整记录
结果：一周后仍有遗漏
```

**深度复盘（正确）**
```
第一层：日志字段缺失
第二层：为什么缺失？→ Agent执行时没有读取SKILL.md中关于日志格式的要求
第三层：为什么没读取？→ SKILL.md路径未在规则层明确索引
第四层：为什么路径不明确？→ 规则层只说了「要记录」没说「怎么看格式」
第五层：为什么规则层不完整？→ 规则层过度瘦身，结论级HOW不足

根本原因：规则层与SKILL层的职责边界不清
改进措施：
1. 在projects-rules-for-agent.md中明确指向日志格式的SKILL.md路径
2. 在Agent执行完成前增加格式校验环节
3. 在memory中沉淀日志格式的模式文档
```
