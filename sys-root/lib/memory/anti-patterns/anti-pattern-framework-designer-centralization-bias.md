# 反模式：框架设计者集中化管理偏见

## 一句话定义

设计框架级规范时，默认所有数据由框架集中管理，忽略业务项目的自主权和数据所有权，违反「数据就近存储」原则。

## 问题场景

### 典型表现

```
❌ 错误设计:
agentsystem/
└── docs/project-prd-changes/
    └── retrospectives/
        ├── framework/          ← 正确：框架级复盘
        └── projects/         ← 错误：业务项目复盘不应在这里
            └── shopify-theme/
                └── 复盘-商品详情页优化.md

问题：Shopify 团队想看自己的复盘，需要跨仓库访问，违背直觉
```

### 正确设计

```
✓ 正确设计:
agentsystem/              Proj01ShopifyTheme/
└── docs/project-prd-changes/               └── docs/project-prd-changes/
    └── retrospectives/                 └── retrospectives/
        └── framework/                      └── 2026-03/
            └── 复盘-能力成长.md              └── 复盘-商品详情页优化.md

优势：数据就近，Shopify 团队在自己的项目内就能找到复盘
```

## 为什么会发生

### 认知根源

| 因素 | 框架设计者视角 | 业务项目 owner 视角 |
|-----|--------------|-------------------|
| **默认假设** | 「框架应该提供完整解决方案」 | 「我的数据应该在我手里」 |
| **关注点** | 集中管理便于统一规范 | 数据就近存储符合直觉 |
| **视野范围** | 单一仓库（框架仓库） | 多仓库（框架 + 业务项目） |
| **问题定义** | 「如何设计统一规范？」 | 「数据应该存储在哪？」 |

### 触发条件

- 设计跨项目的目录结构
- 涉及多个仓库的数据管理
- 站在框架设计者视角思考

## 危害

### 直接影响

1. **跨仓库访问负担**
   - 业务团队需要切换到框架仓库才能访问自己的数据
   - 增加认知负担，违背直觉

2. **项目独立性受损**
   - 业务项目不是自包含的，依赖框架仓库存储其数据
   - 项目迁移或归档时，数据分散在不同仓库

3. **数据所有权模糊**
   - 不清楚数据应该归谁管理
   - 维护责任不明确

### 长期影响

- 框架仓库膨胀，包含大量非框架数据
- 业务项目团队难以找到自己的数据
- 数据管理混乱，难以追溯

## 如何避免

### 检查清单（设计前必须回答）

| # | 检查项 | 问题 | 判定标准 |
|---|-------|------|---------|
| 1 | 数据所有权 | 这个数据应该归谁所有？ | 明确回答：框架 or 业务项目 |
| 2 | 用户访问路径 | 用户会如何实际访问这个数据？ | 验证是否直观、就近 |
| 3 | 分布式方案 | 是否存在分布式管理的替代方案？ | 优先考虑分布式 |
| 4 | 多仓库验证 | 是否考虑了其他仓库也有类似目录结构？ | 确认跨仓库视角 |
| 5 | 视角切换 | 是否切换到业务项目视角验证？ | 强制切换视角 |

### 决策流程

```
设计跨项目数据管理时：

Step 1: 识别数据类型
   ├─ 框架能力相关（patterns、skills）→ 集中式（框架仓库）
   ├─ 项目特定数据（复盘、需求文档）→ 分布式（业务项目）
   └─ 索引/元信息 → 可选集中式（仅索引，不存实际数据）

Step 2: 验证访问路径
   「作为业务项目开发者，我会如何访问这个数据？」
   如果答案是「跨仓库」，则设计可能有误

Step 3: 选择存储位置
   数据应该存储在「数据产生的地方」
   遵循「谁产生，谁管理」原则
```

### 快速检查表

```yaml
当你设计目录结构时，如果：
  - 把所有数据都放在框架仓库 → 可能触发本反模式
  - 未验证业务项目的独立存储需求 → 可能触发本反模式
  - 默认「集中管理 = 便于维护」 → 可能触发本反模式

请立即执行：
  1. 问自己：数据应该归谁所有？
  2. 切换到业务项目视角验证
  3. 优先考虑分布式方案
```

## 修复案例

### 案例：复盘目录结构修正

**修复前（V1 错误）**:
```
agentsystem/docs/retrospective/projects/shopify-theme/
```

**问题识别**:
- 用户指出：「本仓和业务项目根都有 `docs/`（按 usr-rules OpenSpec 第一节），复盘应该存在哪个仓库？」
- 发现：违反了「数据就近存储」原则

**修复后（V2 正确）**:
```
Proj01ShopifyTheme/docs/retrospective/
```

**修复动作**:
1. 删除错误目录 `agentsystem/.../projects/`
2. 在 Proj01ShopifyTheme 中创建正确的复盘目录
3. 创建 V2 规范文档，明确分布式管理架构
4. 更新所有路径引用
