# 三层穿透分析法（3-Layer Problem Analysis）

## 一句话定义

问题分析时穿透「症状层 → 根因层 → 范式层」，在范式层寻找根本性解决方案。

## 为什么需要这个方法

### 常见陷阱

```
┌─────────────────────────────────────────────────────────────┐
│                    问题解决中的常见陷阱                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  陷阱 1: 症状缓解（Symptom Relief）                           │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 问题：timeout 错误                                   │    │
│  │ 解决：调大 timeout 参数                              │    │
│  │ 结果：这次不报错了，下次可能还报错                    │    │
│  │ 本质：治标不治本，问题还在                            │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  陷阱 2: 局部优化（Local Optimization）                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 问题：timeout 错误                                   │    │
│  │ 解决：减少任务数量，让执行变快                       │    │
│  │ 结果：这次通过了，但约束了功能范围                    │    │
│  │ 本质：在原有框架内优化，不解决根本问题                │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
│  陷阱 3: 假设忽略（Hidden Assumption）                      │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 隐含假设：「一次调用必须完成所有任务」               │    │
│  │ 从未质疑这个假设是否合理                              │    │
│  │ 结果：在假设的牢笼里寻找解决方案                      │    │
│  │ 本质：框架限制了思维                                  │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 真实案例

**案例：check-langgraph-backend timeout 问题**

| 分析层 | 观察 | 如果停留于此层的解决方案 | 问题 |
|-------|------|------------------------|------|
| **症状层** | timeout 错误 | 调大 timeout | 只能解决这次，下次可能还报 |
| **根因层** | 执行时间 600s > timeout 120s | 让任务变轻或减少任务 | 约束了功能，不扩展 |
| **范式层** | 「大事务」模式与「可观测需求」矛盾 | **阶段化执行模式** | ✅ 根本性解决，可复用 |

## 三层穿透分析法

### 方法框架

```
┌─────────────────────────────────────────────────────────────┐
│                     三层穿透模型                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   第一层：症状层（What）                                      │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  用户/系统观察到的表象问题                            │   │
│   │  例：timeout 错误、页面加载慢、功能不可用             │   │
│   │                                                     │   │
│   │  🔍 关键问题：这是真问题还是表象？                  │   │
│   └─────────────────────────────────────────────────────┘   │
│                         ↓ 穿透表象                         │
│   第二层：根因层（Why）                                       │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  导致症状的直接原因                                  │   │
│   │  例：执行时间 > timeout 时间、网络延迟、资源不足    │   │
│   │                                                     │   │
│   │  🔍 关键问题：为什么会这样？还有更深的原因吗？      │   │
│   └─────────────────────────────────────────────────────┘   │
│                        ↓ 透过直接原因                       │
│   第三层：范式层（How）                                       │
│   ┌─────────────────────────────────────────────────────┐   │
│   │  导致根因的「设计范式/隐含假设」                     │   │
│   │  例：「一次调用完成所有任务」是设计选择，而非必然     │   │
│   │                                                     │   │
│   │  🔍 关键问题：有没有隐含假设被忽略了？推翻会怎样？ │   │
│   │                                                     │   │
│   │  ✅ 真正有效的解决方案必须在范式层动手               │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 每层的关键问题

#### 第一层 → 第二层：穿透表象

**核心问题**：这是真问题还是表象？

```
症状：timeout 错误
  ├─ 直接原因：执行时间 > timeout 时间 ✓ 找到根因层
  └─ 表面缓解：调大 timeout（停留在症状层，治标不治本）
```

**检查清单**:
- [ ] 这个错误是真实原因，还是其他问题的症状？
- [ ] 如果解决这个表象，问题会彻底解决吗？
- [ ] 有没有其他可能的解释？

#### 第二层 → 第三层：透过直接原因

**核心问题**：为什么会这样？这是唯一的选择吗？

```
根因：执行时间 600s > timeout 120s
  ├─ 直接方案：减少任务或调大 timeout（停留在根因层，局部优化）
  └─ 范式追问：
       ├─ 为什么必须一次调用完成所有任务？
       ├─ 这是设计选择还是必然要求？
       ├─ 如果拆成多次调用会怎样？
       └─ 其他领域有类似解决方案吗？
```

**检查清单**:
- [ ] 这是技术约束还是设计选择？
- [ ] 有没有隐含假设我从未质疑过？
- [ ] 如果推翻这个假设，会怎样？
- [ ] 其他领域/项目是如何解决类似问题的？

#### 第三层：范式突破

**核心问题**：在范式层寻找根本性解决方案

```
范式问题：「大事务」模式与「可观测需求」的矛盾

根本性解决方案：
├─ 引入「阶段化执行」模式
├─ 改为「多次短调用 + 聚合」
└─ 支持 checkpoint + resume

结果：不是「解决这个问题」，而是「消除这类问题」
```

## 实际应用案例

### 案例 1：timeout 问题（本复盘）

```
症状层：timeout 错误
  ↓ 穿透
根因层：执行 600s > timeout 120s
  ↓ 穿透
范式层：「大事务」模式的问题
  ↓ 解决
方案：阶段化执行（pattern-observable-small-steps）
  ↓ 结果
价值：这类问题以后都不会有了（不是解决单次问题，是消除问题类型）
```

### 案例 2：页面加载慢

```
症状层：页面加载 > 5s，用户投诉
  ↓ 穿透
根因层：接口返回慢（3s）+ 渲染慢（2s）
  ↓ 穿透
范式层：「同步阻塞加载」模式的问题
  ↓ 解决
方案：
  ├─ 骨架屏 + 渐进式加载（先显示框架，再填数据）
  ├─ 接口分页/分片（先加载首屏，懒加载其余）
  └─ 缓存 + 预加载（提前准备数据）
  ↓ 结果
价值：用户感知时间从 5s → 0.5s（骨架屏立即显示）
```

### 案例 3：数据库查询慢

```
症状层：查询 timeout
  ↓ 穿透
根因层：单表数据量 1000w+，无索引
  ↓ 穿透（到这里可能足够，继续穿透）
范式层：「单库单表」存储模式的问题
  ↓ 解决
方案：
  ├─ 分库分表（sharding）
  ├─ 读写分离
  └─ 引入 Elasticsearch 等专用查询引擎
  ↓ 结果
价值：从根本解决大数据量查询问题
```

## 三层 vs 解决方案层次

```
┌─────────────────────────────────────────────────────────────┐
│                   解决方案的层次                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Level 1: 症状缓解（Symptom Relief）                          │
│  └── 对应：症状层                                            │
│  └── 例：调大 timeout                                       │
│  └── 特点：简单、立即生效、治标不治本                         │
│                                                              │
│  Level 2: 局部优化（Local Optimization）                    │
│  └── 对应：根因层                                            │
│  └── 例：优化 SQL、加索引、减少任务                           │
│  └── 特点：在现有框架内优化，不解决根本问题                   │
│                                                              │
│  Level 3: 模式创新（Pattern Innovation）★                   │
│  └── 对应：范式层                                            │
│  └── 例：阶段化执行、架构重构、范式转移                        │
│  └── 特点：范式转移，根本性解决，可复用                       │
│                                                              │
│  三层穿透分析法的目标是推动解决方案到达 Level 3                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 使用场景

### 何时使用

✅ **推荐使用**：
- 问题反复出现， symptom relief 无效
- 解决方案的可扩展性差，每次都需要 custom fix
- 多个类似问题，需要统一框架解决
- 技术债务积累，需要根本性重构

### 使用步骤

```
步骤 1：症状记录
├─ 客观记录观察到的表象（不解释、不判断）
└─ 例：「脚本报 timeout 错误」

步骤 2：根因分析
├─ 问「为什么会这样？」直到找到直接原因
├─ 使用 5 Whys 方法
└─ 例：「timeout 是因为执行 600s > timeout 120s」

步骤 3：范式识别
├─ 问「为什么会这样？这是唯一选择吗？」
├─ 寻找隐含假设
└─ 例：「为什么必须一次调用完成？这是设计选择」

步骤 4：方案设计
├─ 在范式层寻找根本性解决方案
├─ 问「如果推翻假设，会怎样？」
└─ 例：「拆成阶段化执行」

步骤 5：价值评估
├─ 评估方案是「解决单次问题」还是「消除问题类型」
└─ 例：阶段化执行是「消除问题类型」
```

## 常见误区

### 误区 1：过度穿透

**现象**：在简单问题上过度分析，追求「范式级解决方案」

**判断标准**：
- 问题是否反复出现？
- 解决方案是否可复用？
- 投入产出比是否合理？

**对策**：不是所有问题都需要穿透到范式层，局部优化有时足够。

### 误区 2：穿透不足

**现象**：停留在症状层或根因层，未到达范式层

**判断标准**：
- 方案是否能彻底解决这类问题？
- 是否需要反复应用类似方案？

**对策**：问自己「这是设计选择还是必然约束？」

### 误区 3：范式层方案过于激进

**现象**：提出的范式层方案成本过高，不切实际

**判断标准**：
- 实现成本 vs 收益
- 团队能力 vs 方案复杂度
- 时间约束

**对策**：范式层方案也需要评估 ROI，有时 Level 2 足够。
