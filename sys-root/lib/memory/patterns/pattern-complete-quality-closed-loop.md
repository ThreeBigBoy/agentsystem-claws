# 完整质量闭环流程（10步）

> **版本**: v1.4  
> **适用**: agentsystem 及所有采用 OpenSpec 治理的项目

---

## 一句话定义

覆盖从需求分析到全局联动的 **10 步完整质量闭环流程**，确保每个阶段都有质量保障、评审留痕、技能映射和系统一致性验证。

---

## 问题背景

### 传统工作流的质量缺口

```
┌─────────────────────────────────────────────────────────────┐
│                    传统工作流的质量缺口                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  传统流程：                                                   │
│                                                              │
│  PRD 产出 ──→ 技术方案 ──→ 编码实现 ──→ 测试 ──→ 发布          │
│      │          │          │         │                        │
│      ▼          ▼          ▼         ▼                        │
│   缺口1      缺口2       缺口3     缺口4                      │
│                                                              │
│  缺口1: PRD 缺乏系统评审                                       │
│  • 简单打勾或简单会议纪要                                       │
│  • 无详细评审过程、决策逻辑、修改建议                            │
│  • 无法追溯「为什么这样设计」                                    │
│                                                              │
│  缺口2: 技术方案缺乏架构评审                                    │
│  • 直接开始编码，方案质量依赖个人经验                             │
│  • 实现与需求脱节、与架构不符                                    │
│                                                              │
│  缺口3: 实现与方案脱节                                         │
│  • 代码实现与 PRD/技术方案不一致                                  │
│  • 代码评审流于形式，未对照方案                                   │
│                                                              │
│  缺口4: 验收标准模糊                                           │
│  • 无明确验收 Checklist                                         │
│  • 无法对照 specs 验证                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### AI 工作流的改进需求


## 模式解决方案

### 核心思想

```
┌─────────────────────────────────────────────────────────────┐
│                  10 步完整质量闭环 v1.3                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step1   Step2   Step3   Step4   Step5   Step6   Step7  Step8  │
│    │       │       │       │       │       │       │       │ │
│    ▼       ▼       ▼       ▼       ▼       ▼       ▼       ▼ │
│  需求 → PRD → 方案 → 编码 → 评审 → 验收 → 归档 → 复盘 → 检查 │
│         │      │              │                      │      │
│         └──评审修复循环───────┘                      │      │
│                                                      │      │
│                              Step 10: 全局检查与联动更新   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ 每步都有：                                            │  │
│  │ • 准入条件 → 执行方 → Skill → 产出物 → 准出条件      │  │
│  │ • 质量门禁：不通过则退回修改                         │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Agent 执行策略（与 usr-rules OpenSpec §2.5 对齐）

针对某一 **change-id**，Agent **必须**遵守 **`agentsystem/usr-devclaw/usr-rules/OpenSpec.md` §2.5**，摘要如下（权威条文以该文件为准）：

1. **标准答复**：每次回复须含 **当前步（Step 1～10）**、**下一步**、**人工门禁状态**（是否停在 PRD 终稿 / 技术方案终稿 / 归档前确认）。
2. **自主推进**：在未到上述人工门禁且准入满足时，**应主动编排并连续推进多步**，避免逐步空等；到达须人类书面确认或拍板的节点时**明确停下并提示**。
3. **Skill + Memory 强制**：进入任一步前，先读 **`claw-config/SKILLS.md`** 映射的 **`sys-root/lib/skills/<技能名>/SKILL.md`**；并加载 **`claw-config/MEMORY.md`** 索引及与本步相关的 memory（至少覆盖门禁与评审时 `pattern-quality-gate-checkpoint`、`pattern-review-fix-loop` 等）。

### 10 步详细流程

#### Step 1: 需求分析 (request-analysis)

```yaml
名称: 需求分析
Agent: 产品经理 Agent
触发 Skill: request-analysis

准入条件:
  - 用户提出需求或变更请求
  - 或识别到新的 change-id 需要启动

执行内容:
  - 产出 PRD: docs/project-prd-changes/[change-id]/PRD-[change-id]-[关键词].md
  - 必须符合 8 类内容结构:
    1. 需求来源与背景
    2. 问题价值分析与优先级
    3. 竞品/平台调研
    4. 本迭代目标与成功标准
    5. 产品方案要点
    6. 功能列表与验收标准
    7. 需求验收 Checklist
    8. 与 OpenSpec 的对应关系

产出物:
  - PRD 文档
  - 可选: 市场研究与产品方案.md
  - 可选: 功能需求说明书.md
  - 可选: 需求验收 Checklist.md

准出条件:
  - PRD 文档已产出
  - 自检清单通过（可自检）
  - 准备进入 prd-review

质量门禁:
  - PRD 结构不完整 → 补充完整后再进入评审
```

#### Step 2: PRD 评审 (prd-review)

```yaml
名称: PRD 评审
Agent: 产品经理 Agent / 主 Agent
触发 Skill: prd-review

准入条件:
  - Step 1 完成，PRD 已产出
  - PRD 文档存在于设计目录

执行内容:
  - 按 9 项自检清单逐项评审:
    1. 价值分析
    2. 竞品/调研
    3. 迭代目标
    4. 成功衡量
    5. 产品方案
    6. toC/toB 可落地
    7. 场景描述
    8. 设计产出物
    9. 文档命名规范
  - 产出评审纪要: docs/project-prd-changes/[change-id]/records/PRD-[change-id]-评审纪要.md

产出物:
  - 评审纪要（必须包含 5 章节）:
    1. 评审基本信息
    2. 评审过程记录
    3. 问题发现与处理
    4. 整体评审结论
    5. 附录

准出条件（综合判定）:
  - 通过: 所有自检项均通过（或仅 minor 建议，无阻塞问题）
  - 有条件通过: 存在有条件通过项，但无阻塞性问题（记录优化项）
  - 不通过: 存在不通过项（必须修改 PRD，重新评审）

质量门禁:
  - 不通过 → 退回 Step 1，修改 PRD
  - 有条件通过 → **不得**进入 Step 3；须按 `pattern-review-fix-loop` **修复 → 再评审 → 通过**
  - 评审「通过」后 → 仍须 **人类**完成 **Gate-PRD（PRD 终稿门禁）** 书面确认（见 `agentsystem/usr-devclaw/usr-rules/OpenSpec.md` **§2.2.1**），再进入 Step 3

**与 OpenSpec §2.2.1 对齐**：Step 2 在流程上拆为 **PRD-R（Agent 评审与纪要）→ PRD-F（修回再评直至通过）→ Gate-PRD（人工终稿）**；主步骤编号仍为 Step 2。

迭代日志记录:
  - "- YYYY-MM-DD | [change-id] | prd-review | PRD 评审完成，综合判定：[通过/有条件通过/不通过]，评审纪要路径：[路径]"
```

#### Step 3: 工程结构分析 (project-analysis)

```yaml
名称: 工程结构分析
Agent: 架构 Agent
触发 Skill: project-analysis

准入条件:
  - Step 2 内 PRD 评审已综合判定 **通过**（PRD-F 已收敛），且 **Gate-PRD** 已书面确认（见 `OpenSpec.md` §2.2.1）
  - PRD 评审纪要已产出

执行内容:
  - 分析工程结构与技术实现方案
  - 产出技术方案: openspec/changes/[change-id]/design.md
  - 必须符合 6 章节结构:
    1. 变更目标与范围
    2. 架构与模块
    3. 接口与数据
    4. 关键流程与执行逻辑
    5. 异常、安全与性能
    6. 与需求/PRD 的对应

产出物:
  - 技术方案文档（design.md）
  - 可选: 技术架构图、执行逻辑图、数据流图
  - 可选: project-rules/ 下补充约束文档

准出条件:
  - 技术方案已产出
  - 与 PRD 对应关系明确
  - 准备进入 architecture-review

质量门禁:
  - 技术方案结构不完整 → 补充完整后再进入评审
  - 与 PRD 不一致 → 对齐后再进入评审
```

#### Step 4: 技术方案评审 (architecture-review)

```yaml
名称: 技术方案评审
Agent: 架构 Agent / 主 Agent
触发 Skill: architecture-review

准入条件:
  - Step 3 完成，技术方案已产出
  - 技术方案文档存在于 openspec/changes/[change-id]/

执行内容:
  - 对照 PRD 评审技术方案
  - 按 9 项自检清单逐项评审:
    1. 变更目标与范围
    2. 与 project.md 一致
    3. 与需求/PRD 可追溯
    4. 接口与数据
    5. 关键流程与图示
    6. 异常与边界
    7. project-rules 从属关系
    8. 与 openspec 一致
    9. 文档命名规范
  - 产出评审纪要: docs/project-prd-changes/[change-id]/records/技术方案-[change-id]-评审纪要.md

产出物:
  - 评审纪要（必须包含 5 章节）:
    1. 评审基本信息
    2. 评审过程记录
    3. 问题发现与处理
    4. 整体评审结论
    5. 附录

准出条件（综合判定）:
  - 通过: 所有自检项均通过（或仅 minor 建议，无阻塞问题）
  - 有条件通过: 存在有条件通过项，但无阻塞性问题（记录优化项）
  - 不通过: 存在不通过项（必须修改技术方案，重新评审）

质量门禁:
  - 不通过 → 退回 Step 3，修改技术方案
  - 有条件通过 → **不得**进入 Step 5；须 **修复 → 再评审 → 通过**
  - 评审「通过」后 → 仍须 **人类**完成 **Gate-DESIGN（技术方案终稿门禁）**（见 `OpenSpec.md` **§2.2.1**），再进入 Step 5
  - 技术方案与 PRD 不一致 → 对齐后再进入编码

**与 OpenSpec §2.2.1 对齐**：Step 4 拆为 **DESIGN-R → DESIGN-F → Gate-DESIGN**；主步骤编号仍为 Step 4。

迭代日志记录:
  - "- YYYY-MM-DD | [change-id] | architecture-review | 技术方案评审完成，综合判定：[通过/有条件通过/不通过]，评审纪要路径：[路径]"
```

#### Step 5: 编码实现 (coding-implement)

```yaml
名称: 编码实现
Agent: 前端 Agent / 后端 Agent
触发 Skill: coding-implement

准入条件:
  - Step 4 内技术方案评审已综合判定 **通过**（DESIGN-F 已收敛），且 **Gate-DESIGN** 已书面确认（见 `OpenSpec.md` §2.2.1）
  - 技术方案评审纪要已产出
  - 技术方案与 PRD 对齐

执行内容:
  - 按技术方案实现代码
  - 遵循 project.md 和 project-rules/ 的工程约束
  - 维护相关元信息文档:
    - info-database/（如涉及数据库变更）
    - info-service-interface/（如涉及接口变更）

产出物:
  - 代码文件
  - 可选: 数据库表结构说明与 DDL 脚本
  - 可选: 接口说明文档

准出条件:
  - 代码实现完成
  - 自检通过（实现完成自检）
  - 准备进入 code-review

质量门禁:
  - 实现与技术方案不符 → 调整实现或更新技术方案
  - 自检不通过 → 修复问题后再进入评审
```

#### Step 6: 代码评审 (code-review)

```yaml
名称: 代码评审
Agent: 架构 Agent
触发 Skill: code-review

准入条件:
  - Step 5 完成，代码已实现
  - 实现自检通过

执行内容:
  - 对照技术方案评审代码
  - 检查维度:
    - 需求符合性（是否实现 PRD 所有功能点）
    - 架构分层（是否符合架构设计）
    - 代码质量（可读性、可维护性）
    - 安全/性能（是否满足非功能性要求）
    - 日志与监控（是否可观测）
    - 测试覆盖（是否充分测试）
  - 产出评审记录: docs/project-prd-changes/[change-id]/records/[change-id]-code-review.md

产出物:
  - 代码评审记录（包含问题清单与后续行动）

准出条件:
  - 无 Blocking/Major 级问题
  - 或问题已转化为 tasks.md 中的任务并计划修复

质量门禁:
  - 存在 Blocking 问题 → 修复后才能进入验收
  - Major 问题 → 评估后决定是否修复或记录为技术债务

协同关系:
  - 与 coding-implement 形成「实现 → 评审 → 迭代」闭环
  - 发现问题可回溯到需求/架构层，建议更新相关文档
```

#### Step 7: 功能验收 (func-test)

```yaml
名称: 功能验收
Agent: 测试 Agent
触发 Skill: func-test

准入条件:
  - Step 6 完成，代码评审通过
  - 代码已合并或准备合并

执行内容:
  - 对照 specs 中 Requirements + Scenarios 执行测试
  - 对照需求验收 Checklist 逐项验证
  - 执行两轮 OpenSpec validate:
    - 第一轮: openspec validate [change-id]（验证一致性）
    - 第二轮: openspec validate --strict（严格模式，验收通过后执行）
  - 产出验收记录: docs/project-prd-changes/[change-id]/records/[change-id]-func-test.md

产出物:
  - 验收记录（包含测试用例、测试结果、validate 结果）

准出条件:
  - 所有验收 Checklist 项通过
  - 或问题已修复并重新验证
  - openspec validate --strict 通过

质量门禁:
  - 关键功能未通过 → 修复后重新验收
  - validate 不通过 → 对齐文档与实现

协同关系:
  - 与 coding-implement、code-review 形成「实现 → 评审 → 验收」质量闭环
  - 测试中发现的需求/架构问题，反向推动文档与规范修订
```

#### Step 8: 归档完成（v1.1 升级：明确 OpenSpec 归档操作步骤）

```yaml
名称: 归档完成
Agent: 主 Agent / 自动流程
触发: 验收通过后自动触发

执行内容:
  1. 术语规范检查（新增）:
     - 查阅 OpenSpec.md "阶段 3: 归档变更" 章节
     - 确认归档操作定义：合并 specs/ 规范增量 + 移动 changes/ 目录
     - 唤醒 preference-archive-operation-checklist 进行检查

  2. 文档状态更新:
     - 更新 tasks.md，勾选所有已完成任务
     - 更新 design.md（如适用），添加变更完成摘要

  3. OpenSpec 归档核心操作（v1.1 明确）:
     - 阶段 A: 合并规范增量
       * 确认 changes/[change-id]/specs/[capability]/spec.md 存在
       * 阅读 ADDED/MODIFIED/REMOVED 内容
       * 合并到 specs/[capability]/spec.md
       * 验证合并结果
     - 阶段 B: 移动变更目录
       * 确认 changes/[change-id]/ 完整
       * 移动到 changes/archive/[change-id]-YYYY-MM-DD/
       * 验证移动结果

  4. 更新迭代日志:
     - 追加 OpenSpec 归档完成记录
     - 包含：合并规范、移动目录、归档验证

  5. 触发 Step 9（复盘）判断

产出物:
  - 已更新的 specs/[capability]/spec.md（包含本次变更增量）
  - 已移动的 changes/archive/[change-id]-YYYY-MM-DD/（完整变更历史）
  - 完整的 records/ 目录（评审纪要、验收记录等）
  - 归档完成声明（变更状态更新）
  - 触发复盘判断标记

准出条件（v1.1 完善）:
  - 所有任务已标记完成（tasks.md）
  - 验收已通过（func-test 结论为「通过」）
```

## 关键质量门禁

| 步骤 | 准入条件 | 准出条件 | 质量门禁（不通过时） |
|-----|---------|---------|-------------------|
| Step 1 | 需求提出 | PRD 产出 | 结构不完整 → 补充 |
| Step 2 | PRD 产出 | 评审通过 + Gate-PRD | 不通过/有条件通过 → PRD-F 修回再评；终稿须 Gate-PRD |
| Step 3 | PRD 评审通过 | 技术方案产出 | 与 PRD 不一致 → 对齐 |
| Step 4 | 技术方案产出 | 评审通过 + Gate-DESIGN | 不通过/有条件通过 → DESIGN-F 修回再评；终稿须 Gate-DESIGN |
| Step 5 | 技术方案评审通过 | 代码实现 | 与方案不符 → 调整 |
| Step 6 | 代码实现 | 评审通过 | Blocking 问题 → 修复 |
| Step 7 | 代码评审通过 | 验收通过 | 不通过 → 修复后重验 |
| Step 8 | 验收通过 | 归档完成 | - |
| Step 9 | 满足复盘触发条件 | 复盘完成 | 质量不通过 → 补充分析 |
| Step 10 | 复盘完成/用户指令 | 全局检查通过 | 发现遗漏 → 立即修复 |

## Step 9 复盘触发条件

复盘不是「可选」步骤，而是满足以下条件**必须触发**：

```yaml
触发条件（满足任一即触发）:
  条件 1: 完成重要里程碑后
    判定标准: change-id 完成归档（Step 8 结束）
    触发动作: 主 Agent 自动建议「是否进行复盘？」
    用户响应: 用户可确认「复盘」或「跳过」（需说明理由）
    
  条件 2: 问题反复出现时
    判定标准: 同类问题在 ≥2 个不同 change-id 中出现
    触发动作: Agent 主动提示「检测到同类问题重复出现，建议进行复盘分析根因」
    必须执行: 问题型复盘必须执行，不可跳过
    
  条件 3: 用户明确指令
    判定标准: 用户输入「复盘」「总结」「回顾」「反思」等关键词
    触发动作: 立即执行 retrospective-analysis skill
    必须执行: 用户指令型复盘必须执行
    
  条件 4: 定期复盘
    判定标准: 每周/每月/每季度周期到达
    触发动作: Agent 主动发起「定期复盘时间到了，是否进行复盘？」
    用户响应: 用户可确认「复盘」或「延期」（需设置新时间）

跳过复盘的记录:
  如果用户选择跳过复盘，必须在迭代日志中记录:
  "- YYYY-MM-DD | [change-id] | retrospective-analysis | 用户选择跳过复盘，理由：[理由]"
  
  问题型复盘不允许跳过，必须执行。
```

## 适用场景

### 必须使用

| 场景 | 说明 |
|-----|------|
| **新项目启动** | 首个 change-id 必须按完整闭环执行 |
| **重大功能迭代** | 涉及架构调整、新增能力、重大变更 |
| **跨团队协作** | 需要异步协作、文档驱动、质量可追溯 |
| **质量敏感项目** | 对质量要求高，需要详细审计记录 |

### 推荐使用

| 场景 | 说明 |
|-----|------|
| **小型功能迭代** | 可简化部分步骤，但核心评审不可少 |
| **技术债务偿还** | 需要评估、方案、实现、验收完整闭环 |
| **重构改造** | 必须评审技术方案，确保架构合理性 |
| **新人培训** | 通过完整闭环让新人理解规范 |

### 可简化使用

| 场景 | 简化方式 |
|-----|---------|
| **文档微调** | 跳过编码、代码评审、功能验收 |
| **Hotfix** | 紧急修复，可简化 PRD/技术方案评审，但需事后补录 |
| **实验性探索** | 可简化部分评审，但需明确实验范围 |

## 收益

| 维度 | 改进前 | 改进后 |
|-----|-------|--------|
| **质量可追溯** | 无法追溯设计决策 | 评审纪要详细记录决策逻辑 |
| **质量可审计** | 审计时无据可查 | 评审纪要作为审计依据 |
| **质量一致性** | 依赖个人经验，波动大 | 标准化评审清单，质量可控 |
| **知识传承** | 经验随人走 | 评审纪要沉淀为记忆 |
| **协作效率** | 会议驱动，成本高 | 文档驱动，异步协作 |
| **问题定位** | 难以定位问题根因 | 分阶段评审，精准定位 |

## 附录：评审修复循环（2026-03-16 升级新增）

### 升级背景

在 check-langgraph-backend 第二阶段推进过程中，发现 8+1 质量闭环存在重大缺陷：**缺乏对「有条件通过」评审结论的后续处理机制**。

详见复盘报告：`docs/retrospective/framework/2026-Q1/复盘-8+1质量闭环-有条件通过处理机制-2026-03-16.md`

### 升级内容：增加评审修复循环

#### 核心原则

> **只有「100% 通过」才是真正的通过。「有条件通过」必须经过修复和重新评审，转化为「通过」后才能进入下一阶段。**

#### 评审结论判定标准（升级后）

| 评审结论 | 判定标准 | 后续动作 | 是否可以进入下一阶段 |
|---------|---------|---------|-------------------|
| **✓ 通过** | 所有自检项通过，无 minor 建议 | 直接进入下一阶段 | ✅ **可以** |
| **△ 有条件通过** | 基本可用，但有 minor 问题 | **必须**修复并重新评审 | ❌ **不可以**（直到转为「通过」） |
| **✗ 不通过** | 有阻塞性问题，不满足落地要求 | **必须**修复并重新评审 | ❌ **不可以**（直到转为「通过」） |

## 规范执行衰减防护机制（v1.2 新增）

### 问题：规范执行衰减效应

**现象**：
- 新规范刚升级时执行最好（如 PRD/技术方案评审严格执行修复循环）
- 随着时间推移，执行力度逐渐衰减（如代码评审/验收沿用旧习惯）
- 惯性思维导致规范流于形式

**根因**：
- 过度依赖历史执行模式
- 未在每次执行前查阅当前规范版本
- 缺少显性检查机制

---

### 防护措施

#### 1. 执行前强制检查（Execution Pre-check）

**每次执行 10 步闭环任何阶段前，必须完成**：

| # | 检查项 | 检查内容 | 防止的反模式 |
|---|-------|---------|------------|
| P.1 | Skill 版本确认 | 确认使用的 skill 为最新版本（v1.1+） | anti-pattern-inertia-trap |
| P.2 | 术语定义查阅 | 查阅本阶段关键术语的规范定义 | anti-pattern-terminology-drift |
| P.3 | 检查清单唤醒 | 唤醒 `preference-quality-gate-checklist` 本阶段检查项 | 规范执行衰减 |
| P.4 | 前置阶段确认 | 确认前置阶段质量门禁已通过 | 阶段跳过 |

**执行声明**：
```markdown
**执行前强制检查声明**

我确认已执行以下检查：
- [ ] P.1 已确认 skill 版本为最新（[skill-name] v[x.x]）
- [ ] P.2 已查阅本阶段关键术语的规范定义
- [ ] P.3 已唤醒并准备执行质量门禁检查清单
- [ ] P.4 已确认前置阶段 [Step N-1] 质量门禁已通过

**签名**: [Agent 角色]  
**日期**: YYYY-MM-DD
```

#### 2. 阶段出口强制检查（Stage Exit Check）

**每个阶段出口必须执行**（来自 `preference-quality-gate-checklist`）：

- Step 1: 需求分析质量门禁（7项检查）
- Step 2: PRD 评审质量门禁（6项检查，含评审修复循环）
- Step 3: 工程结构分析质量门禁（4项检查）
- Step 4: 技术方案评审质量门禁（6项检查，含评审修复循环）
- Step 5: 编码实现质量门禁（5项检查）
- Step 6: 代码评审质量门禁（6项检查，含评审修复循环）
- Step 7: 功能验收质量门禁（8项检查，含验收修复循环）
- Step 8: 归档质量门禁（11项检查，含术语检查+4阶段归档操作）

**出口检查规则**：
- 只有全部检查项通过，才能进入下一阶段
- 评审/验收类阶段判定必须为「通过」（100% 通过），「有条件通过」不可进入下一阶段
- 未通过检查必须修复后重新执行该阶段

#### 3. 定期回顾检查（Periodic Review）

**触发条件**：
- 每周一次（轻量回顾）
- 每完成一个 change-id（详细回顾）
- 发现执行偏差时（立即回顾）

**回顾内容**：
| 回顾项 | 检查内容 | 防止的问题 |
|-------|---------|-----------|
| Skill 版本一致性 | 所有相关 skill 是否统一版本 | 执行标准不一致 |
| 术语使用准确性 | 最近执行中是否存在术语理解偏差 | 术语定义漂移 |
| 检查清单执行率 | 是否每次都严格执行了质量门禁检查清单 | 规范执行衰减 |
| 用户追问次数 | 最近是否有用户追问规范执行问题 | 执行偏差 |

#### 4. 自动化防护（Automation Shield）

**工具支持**（长期演进）：
- **validate 工具**：自动检查归档完整性（#8 开发 validate 工具）
- **skill 版本检查器**：自动检查使用的 skill 是否为最新版本
- **术语使用检查器**：自动检查术语使用是否符合规范定义
- **门禁检查清单自动化**：自动提醒并记录检查清单执行

### 规范执行检查融入 8+1 闭环

```
Step N 执行流程（v1.2 含规范执行检查）：

1. 执行前强制检查（新增）
   ├── P.1 确认 skill 版本
   ├── P.2 查阅术语定义
   ├── P.3 唤醒检查清单
   └── P.4 确认前置阶段
   
2. 执行阶段内容
   └── 按 skill 规范执行
   
3. 阶段出口强制检查（强化）
   ├── 执行本阶段质量门禁检查清单
   ├── 判定：通过 / 有条件通过 / 不通过
   └── 有条件通过/不通过 → 修复循环
   
4. 进入下一阶段（仅当全部通过）
```
