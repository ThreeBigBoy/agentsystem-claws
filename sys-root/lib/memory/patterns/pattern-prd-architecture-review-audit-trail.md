# PRD/技术方案评审留痕机制

## 一句话定义

建立独立于迭代日志的评审纪要机制，详细记录评审过程、决策逻辑、修改建议、引用依据，实现质量可追溯、可审计、可学习。

## 问题背景

### 改进前的问题

```
┌─────────────────────────────────────────────────────────────┐
│                    改进前的问题                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  问题 1: 迭代日志 ≠ 评审纪要                                  │
│                                                              │
│  迭代日志内容:                                                │
│  - 2026-03-14 | check-langgraph-backend | 产品经理 Agent |   │
│    完成 PRD 产出                                             │
│                                                              │
│  问题：                                                      │
│  • 只有「做了什么」，没有「评审了什么、结论是什么、为什么」      │
│  • 粒度太粗，无法记录评审过程、决策逻辑、修改建议               │
│  • 无法追溯「为什么这样设计」                                  │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│  问题 2: 自检走过场，无留痕                                    │
│                                                              │
│  自检方式：                                                   │
│  • PRD 产出后，简单打勾「自检通过」                            │
│  • 无详细评审记录                                            │
│  • 无修改建议、无引用依据                                     │
│                                                              │
│  问题：                                                      │
│  • 自检质量不可控                                            │
│  • 发现的问题无法追踪是否解决                                  │
│  • 自检经验无法沉淀复用                                       │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│  问题 3: 知识传承困难                                        │
│                                                              │
│  场景：                                                      │
│  • 新人问：「为什么这个 PRD 要这样写？」                      │
│  • 答：「当时评审过，觉得可以」                                │
│  • 再问：「评审时发现了什么问题？怎么解决的？」                │
│  • 答：「记不清了」                                          │
│                                                              │
│  问题：                                                      │
│  • 评审经验随人走，无法沉淀                                   │
│  • 新人无法从历史评审中学习                                    │
│  • 评审质量依赖个人经验，波动大                                │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 为什么必须独立留痕

```
┌─────────────────────────────────────────────────────────────┐
│                迭代日志 vs 评审纪要 对比                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  维度          迭代日志              评审纪要                 │
│  ─────────────────────────────────────────────────────────  │
│  定位         日志型               评审型                    │
│  ─────        ─────                ─────                    │
│  内容         谁在什么时间          评审了什么               │
│              做了什么事           结论是什么                │
│                                  为什么是这个结论            │
│                                  如何改进                   │
│                                                              │
│  粒度         一行记录             多章节详细文档            │
│  ─────        ─────                ─────────────            │
│  示例         - 2026-03-14 |       ## 评审过程记录          │
│              check-langgraph-     ### 自检项 #1: 价值分析   │
│              backend | 产品经      - 评审发现: ...          │
│             理 Agent | 完成      - 判定结论: ✗ 不通过      │
│              PRD 产出            - 判定理由: ...           │
│                                  - 修改建议: ...           │
│                                  - 引用依据: ...           │
│                                  - 修改后复核: ✓ 通过      │
│                                                              │
│  用途         追溯执行历史        追溯评审决策               │
│  ─────        ───────────        ─────────────             │
│              知道「做了什么」     理解「为什么这样设计」      │
│                                                              │
│  存放         docs/project-prd-changes/    docs/project-prd-changes/         │
│  ─────        迭代日志.md         [change-id]/records/       │
│              （项目级单一文件）   PRD-xxx-评审纪要.md        │
│                                  （按变更归档）              │
│                                                              │
│  关系         互补，非替代                                     │
│  ─────        迭代日志记录「执行」                            │
│              评审纪要记录「评审质量」                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 模式解决方案

### 核心机制

```
┌─────────────────────────────────────────────────────────────┐
│                  评审留痕机制核心设计                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 独立文档类型                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ • 评审纪要是独立于迭代日志的文档类型                   │   │
│  │ • 有独立的内容结构、存放路径、命名规范                 │   │
│  │ • 与迭代日志互补，非替代                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  2. 标准化内容结构                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 评审纪要必须包含 5 个章节：                             │   │
│  │                                                     │   │
│  │ 1. 评审基本信息                                      │   │
│  │    - 评审日期、评审人（Agent 角色）、版本、change-id │   │
│  │                                                     │   │
│  │ 2. 评审过程记录                                      │   │
│  │    - 逐条过自检清单                                  │   │
│  │    - 每项包含：评审发现、判定结论、判定理由、          │   │
│  │                修改建议、引用依据、修改后复核          │   │
│  │                                                     │   │
│  │ 3. 问题发现与处理                                    │   │
│  │    - 汇总表：序号、自检项、原始判定、问题类型、        │   │
│  │               修改建议、修改后判定、状态               │   │
│  │                                                     │   │
│  │ 4. 整体评审结论                                      │   │
│  │    - 综合判定（通过/有条件通过/不通过）                │   │
│  │    - 判定理由、后续行动项                              │   │
│  │                                                     │   │
│  │ 5. 附录                                              │   │
│  │    - 参考文档、相关记忆、评审过程元信息                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  3. 标准化评审判定                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 单项判定：                                           │   │
│  │ • ✓ 通过：内容完整、逻辑自洽、符合行业最佳实践        │   │
│  │ • △ 有条件通过：基本可用，但存在 minor 问题           │   │
│  │ • ✗ 不通过：缺失、逻辑不自洽、不满足落地要求          │   │
│  │                                                     │   │
│  │ 综合判定：                                           │   │
│  │ • 通过：所有自检项均通过（或仅 minor 建议）           │   │
│  │ • 有条件通过：存在有条件通过项，但无阻塞问题          │   │
│  │ • 不通过：存在不通过项                                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  4. 配套 Skills                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ prd-review（PRD 评审技能）                           │   │
│  │ • 触发：「评审 PRD」「检查 PRD 质量」「PRD 自检」      │   │
│  │ • 职责：按 9 项自检清单评审 PRD，产出评审纪要          │   │
│  │ • 产出：docs/project-prd-changes/[change-id]/records/        │   │
│  │          PRD-[change-id]-评审纪要.md                  │   │
│  │                                                     │   │
│  │ architecture-review（技术方案评审技能）               │   │
│  │ • 触发：「评审技术方案」「技术方案自检」               │   │
│  │ • 职责：按 9 项自检清单评审技术方案，产出评审纪要      │   │
│  │ • 产出：docs/project-prd-changes/[change-id]/records/          │   │
│  │          技术方案-[change-id]-评审纪要.md               │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 适用场景

### 必须使用

| 场景 | 说明 |
|-----|------|
| **PRD 评审** | 任何 PRD 产出后，进入 proposal/tasks/specs 阶段前 |
| **技术方案评审** | 任何技术方案产出后，进入 coding-implement 阶段前 |
| **回溯补录** | 基于历史对话和已产出文档，补录评审纪要 |

### 推荐使用

| 场景 | 说明 |
|-----|------|
| **需求变更** | 需求发生重大变更，重新评审时 |
| **架构调整** | 技术方案发生架构级调整，重新评审时 |
| **质量审计** | 需要追溯设计决策时 |
| **新人培训** | 让新人学习历史评审经验时 |

## 收益

| 维度 | 改进前 | 改进后 |
|-----|-------|--------|
| **可追溯性** | 无法追溯「为什么这样设计」 | 评审纪要详细记录决策逻辑 |
| **可审计性** | 质量审计时无据可查 | 评审纪要作为审计依据 |
| **可学习性** | 评审经验随人走 | 新人可通过历史评审纪要学习 |
| **质量一致性** | 依赖个人经验，波动大 | 标准化评审清单，质量可控 |
| **问题追踪** | 发现的问题无法追踪 | 问题汇总表跟踪到解决 |
| **知识传承** | 无法沉淀 | 评审纪要沉淀为记忆 |
