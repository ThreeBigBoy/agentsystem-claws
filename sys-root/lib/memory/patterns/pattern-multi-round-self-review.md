# 多轮自查改进机制（从85%到100%）

## 一句话定义

通过"自查→评估→改进→再评估"的多轮循环，主动识别遗漏，确保交付物从"基本完成"到"真正完成"的质量跃升。

## 问题背景

### 常见的交付质量问题

```
┌─────────────────────────────────────────────────────────────┐
│                    常见的交付质量问题                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  问题 1: 首轮评估乐观偏差                                       │
│  • 自评95%完成 → 用户识别大量缺漏                               │
│  • 评估不准确，需要多轮修正                                      │
│  • 影响用户信任                                                  │
│                                                              │
│  问题 2: 被动完善                                                │
│  • 等待用户追问才补充                                            │
│  • 用户体验差，效率低                                             │
│  • 反复沟通成本高                                                 │
│                                                              │
│  问题 3: "完成"标准模糊                                          │
│  • 什么算"完成"？                                                │
│  • 缺乏量化标准                                                   │
│  • 用户和AI理解不一致                                             │
│                                                              │
│  问题 4: 缺少自查机制                                            │
│  • 做完即结束                                                     │
│  • 没有系统性自查                                                 │
│  • 遗漏难以避免                                                   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 模式解决方案

### 评估与改进流程

```
完成交付物
    ↓
Round 1自查 → 评估
    ↓
若<90% → 补充功能 → 重新评估
    ↓
Round 2自查 → 评估
    ↓
若<100% → 补充生产就绪模块 → 重新评估
    ↓
Round 3自查 → 评估
    ↓
若<85分 → 补充遗漏维度 → 重新评估
    ↓
Round 4自查 → 评估
    ↓
若P0/P3未完成 → 执行优化 → 重新评估
    ↓
Round 5自查（用户视角）
    ↓
用户确认100%
    ↓
真正完成
```

## 实施建议

### 适用场景

| 场景 | 适用性 | 轮次 | 说明 |
|-----|-------|------|------|
| 技术方案设计 | 高 | 5轮 | 确保方案完备性 |
| 代码开发 | 高 | 4轮 | 确保代码质量 |
| PRD编写 | 中 | 3轮 | 确保需求完整 |
| 文档编写 | 中 | 3轮 | 确保文档质量 |
| 测试用例 | 中 | 3轮 | 确保覆盖完整 |

### 关键原则

1. **逐轮递进**: 先功能、再生产、再系统、再优化、最后用户确认
2. **量化评估**: 每轮都有明确的评估标准和目标
3. **主动完善**: 不等用户追问，主动识别遗漏
4. **专家视角**: Round 5要站在专家角度"帮用户多想"

### 常见陷阱

| 陷阱 | 表现 | 避免方法 |
|-----|------|---------|
| 跳过自查 | 直接交付，不检查 | 强制每轮自查 |
| 走过场 | 自查不认真 | 量化评估，未达标不通过 |
| 首轮即结束 | 完成Round 1就交付 | 必须完成5轮 |
| 用户确认形式化 | 不问用户就确认 | 必须获得用户明确确认 |

## 成功案例

### 案例: deepen-langgraph-v2-11-1技术方案评审

**背景**: 技术方案设计完成后的质量保障

**应用多轮自查**:

| 轮次 | 自查内容 | 评估结果 | 改进内容 | 最终达成 |
|-----|---------|---------|---------|---------|
| Round 1 | 功能完整性 | 85% | 补充Agent调用基础设施 | 100% |
| Round 2 | 生产就绪 | 60% | 补充安全性、稳定性模块 | 100% |
| Round 3 | 系统工程 | 75% | 补充限流熔断、链路追踪 | 100% |
| Round 4 | 性能优雅 | 80% | 补充P3健壮性优化 | 100% |
| Round 5 | 用户视角 | 95% | 响应"请帮我多想想" | 100% |

**成果**:
- 技术方案从85%提升到100%
- 主动识别并补充了大量遗漏
- 减少了用户追问次数
- 建立了可复用的自查机制
