# 突破性思维模式 checklist（Paradigm Shift Checklist）

## 一句话定义

推动从「症状驱动、局部优化」升级到「范式驱动、根本性解决」的思维 checklist。

## 为什么需要这个 checklist

### Agent 的思维局限

在 `check-langgraph-backend` 复盘过程中，我（Agent）展现了典型的「工程师思维」局限：

```
┌─────────────────────────────────────────────────────────────┐
│                   Agent 的思维路径                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  问题输入 → 诊断 → 找直接原因 → 给解决方案                   │
│     ↑___________________________________________↓            │
│                                                              │
│  这是一个「高效、直接、可执行」的闭环，但：                   │
│  • 容易陷入「治标不治本」                                     │
│  • 缺乏「为什么这样做」的深层追问                             │
│  • 容易忽略隐含假设                                          │
│  • 不主动质疑框架                                            │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

而用户展现了「突破性思维」：

```
┌─────────────────────────────────────────────────────────────┐
│                   用户的思维路径                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  问题输入 → 穿透表象 → 质疑假设 → 范式转移 → 根本性解决      │
│                                                              │
│  例：timeout 问题                                            │
│    ├─ 不是问「怎么让 timeout 不报错」                         │
│    ├─ 而是问「为什么需要等这么久才能知道结果」                │
│    ├─ 质疑「一次调用完成所有任务」这个假设                   │
│    └─ 提出「阶段化执行」的新范式                              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

这个 checklist 帮助 Agent 主动触发「突破性思维」。

## Checklist 内容

### 第一层追问：穿透表象

**核心问题**：这是真问题还是表象？

```markdown
□ 1.1 症状识别
    客观记录观察到的表象（不解释、不判断）
    例：「脚本报 timeout 错误」「用户投诉页面慢」

□ 1.2 根因定位
    问「为什么会这样？」找到直接原因
    使用 5 Whys 方法
    例：「timeout 是因为执行时间 > 超时阈值」

□ 1.3 价值确认
    解决这个直接原因，问题会彻底解决吗？
    例：调大 timeout，下次还报吗？
    如果否 → 需要继续穿透
```

### 第二层追问：质疑假设

**核心问题**：有没有隐含假设被忽略了？

```markdown
□ 2.1 假设识别
    列出解决问题的隐含前提：
    • 「一次调用必须完成所有任务」
    • 「用户愿意等待完整结果」
    • 「失败时必须重跑全部」
    • ...

□ 2.2 假设质疑
    每个假设问自己：
    • 这是技术约束还是设计选择？
    • 这是必然要求还是习惯做法？
    • 如果推翻这个假设，会怎样？
    • 其他领域/项目是如何做的？

□ 2.3 框架突破
    找到最值得质疑的假设，尝试推翻：
    • 原假设：「一次调用完成所有任务」
    • 推翻后：「多次短调用 + 聚合」
    • 新可能性：阶段化执行、可观测、可控
```

### 第三层追问：范式转移

**核心问题**：如果改变框架，根本性解决方案是什么？

```markdown
□ 3.1 范式识别
    当前使用的是什么范式/模式？
    • 「大事务」模式：单次调用，同步等待，整体成功/失败
    • 「同步阻塞」模式：请求 → 等待 → 响应
    • 「单体架构」模式：所有功能在一个单元

□ 3.2 范式评估
    当前范式的适用性：
    • 这个范式适合当前场景吗？
    • 这个范式的局限在哪里？
    • 有没有更适合的范式？

□ 3.3 范式转移
    寻找替代范式：
    • 「大事务」→「阶段化执行」
    • 「同步阻塞」→「异步可观测」
    • 「单体架构」→「微服务 + 编排」

□ 3.4 根本性方案
    基于新范式设计解决方案：
    • 不是「解决这个 timeout」
    • 而是「让这类 timeout 不再发生」
```

### 第四层追问：价值验证

**核心问题**：这个方案是「解决单次问题」还是「消除问题类型」？

```markdown
□ 4.1 方案层次评估
    Level 1: 症状缓解（调大 timeout）
    Level 2: 局部优化（减少任务）
    Level 3: 模式创新（阶段化执行）★

    目标：推动方案到达 Level 3

□ 4.2 可复用性评估
    • 这个方案能解决一类问题，还是只能解决这次？
    • 类似场景能否复用这个方案？
    • 是否沉淀为可复用的模式/规范？

□ 4.3 ROI 评估
    • 投入：实现成本、改造成本
    • 收益：解决的问题范围、长期价值
    • 判断：是否值得做范式转移？

□ 4.4 演进路径
    如果 Level 3 成本过高：
    • 短期：Level 1/2 workaround（标注技术债务）
    • 长期：Level 3 架构演进（纳入路线图）
```

### 第五层追问：用户价值

**核心问题**：用户真正关心的是什么？

```markdown
□ 5.1 用户视角转换
    从「技术实现视角」切换到「用户体验视角」：
    • 不是「timeout 参数怎么调」
    • 而是「用户能看到进度吗」
    • 不是「任务怎么跑更快」
    • 而是「失败了用户知道怎么重试吗」

□ 5.2 痛点识别
    用户真正的痛点：
    • 显性痛点：timeout 报错
    • 隐性痛点：不知道进度、不知道哪错了、失败后茫然

□ 5.3 价值排序
    对用户价值的优先级：
    1. 能完成任务（基本）
    2. 知道进度（可观测）
    3. 失败可定位（可调试）
    4. 失败可恢复（韧性）
    5. 速度快（性能）
```

## 使用场景

### 何时使用这个 checklist

✅ **推荐使用**：
- 问题反复出现， symptom relief 无效
- 解决方案的可扩展性差
- 多个类似问题需要统一框架
- 技术债务积累需要根本性重构
- 需要创新突破而非渐进优化

❌ **不需要使用**：
- 简单明确的 bug fix
- 一次性问题，无需复用
- 时间和资源不允许深度思考

## 实际应用案例

### 案例：check-langgraph-backend timeout 问题

```
□ 1.1 症状识别
    「脚本报 POST /run: timed out」

□ 1.2 根因定位
    「执行 600s > timeout 120s」

□ 1.3 价值确认
    调大 timeout 能解决这次，但 7 分钟对用户不友好，下次还可能有更重的任务
    → 需要继续穿透 ✓

□ 2.1 假设识别
    • 「一次调用必须完成所有任务」
    • 「用户愿意等待完整结果」

□ 2.2 假设质疑
    • 这是设计选择还是必然？→ 设计选择
    • 其他项目怎么做？→ CI/CD 分阶段、大数据分批次

□ 2.3 框架突破
    推翻「一次调用完成所有任务」
    新可能：阶段化执行

□ 3.1 范式识别
    当前：「大事务」模式（单次调用，同步等待）

□ 3.2 范式评估
    局限：不可观测、超时不可控、失败代价高

□ 3.3 范式转移
    新范式：「阶段化执行」模式

□ 3.4 根本性方案
    /run(phase="env-check") → /run(phase="mcp-check") → 聚合

□ 4.1 方案层次评估
    Level 3: 模式创新 ✓

□ 4.2 可复用性评估
    可复用于所有 LangGraph change-id，甚至其他长事务场景

□ 4.3 ROI 评估
    投入：框架改造、协议设计、工具更新
    收益：根本性解决一类问题，长期价值高

□ 4.4 演进路径
    短期：增加 --timeout 参数 workaround
    长期：阶段化执行架构演进

□ 5.1 用户视角转换
    不是「timeout 怎么调」，而是「我能看到进度吗」

□ 5.2 痛点识别
    隐性痛点：不知道执行到哪一步、失败后茫然

□ 5.3 价值排序
    可观测性 > 速度 > 其他
```

## 学习要点

### 对 Agent 的学习要求

1. **不要急于给解决方案**
   - 先完整走完 checklist
   - 特别是 2.2「假设质疑」和 3.3「范式转移」

2. **主动质疑隐含假设**
   - 问自己「这是必然还是习惯？」
   - 问「其他领域怎么做？」

3. **追求根本性解决**
   - 不只问「怎么解决这个问题」
   - 要问「怎么让这类问题不再发生」

4. **切换用户视角**
   - 技术实现视角 → 用户体验视角
   - 关注「用户能感知什么」而非「我怎么实现」
