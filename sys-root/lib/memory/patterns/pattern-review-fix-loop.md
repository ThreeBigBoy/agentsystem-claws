# 评审修复循环模式（Review-Fix-Loop Pattern）

## 一句话定义

评审不是单向的「通过/不通过」，而是可能包含「修复→重新评审」循环的完整闭环，直至达到「100% 通过」。

---

## 问题背景

### 传统流程的缺陷

```
┌──────────────────────────────────────────────────────────────┐
│                    传统流程（缺陷版）                          │
├──────────────────────────────────────────────────────────────┤
│                                                               │
│  评审 ──→ 通过 ──→ 进入下一阶段                               │
│    │                                                         │
│    ├─→ 有条件通过 ──→ 「可并行进入下一阶段，迭代中优化」      │
│    │              ──→ 进入下一阶段（问题未修复！）            │
│    │                                                         │
│    └─→ 不通过 ──→ 必须修复 ──→ 重新评审 ──→ 进入下一阶段    │
│                                                               │
│  ❌ 问题：有条件通过的问题未修复即流入下一阶段，质量门禁失效  │
│                                                               │
└──────────────────────────────────────────────────────────────┘
```

### 缺陷的本质

| 缺陷 | 影响 |
|------|------|
| **「有条件通过」理解偏差** | 被误解为「差不多可以了」，而非「还有问题需要修复」 |
| **流程闭环缺失** | 缺少「修复→重新评审」的循环节点，流程变成单向线性 |
| **质量门禁失效** | 有条件通过即可进入下一阶段，门禁形同虚设 |
| **技术债务积累** | Minor 问题未修复，积累到后续阶段产生更大问题 |

---

## 模式解决方案

### 核心思想

> **只有「100% 通过」才是真正的通过。「有条件通过」必须经过修复和重新评审，转化为「通过」后才能进入下一阶段。**

### 评审修复循环流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                      评审修复循环（闭环版）                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   评审 ───────────────────────────────────────────┬──→ 通过        │
│    │                                               │                │
│    │                                               └──→ 进入下一阶段  │
│    │                                                                  │
│    ├─→ 通过 ───────────────────────────────────────────────┐          │
│    │                                                       │          │
│    ├─→ 有条件通过 ──→ 识别问题清单 ──→ 修复 ──→ 重新评审 ──┤          │
│    │                                     ↑                │          │
│    │                                     │                │          │
│    │                                     └── 未通过 ──────┤          │
│    │                                                      │          │
│    └─→ 不通过 ──→ 必须修复 ──→ 重新评审 ─────────────────┘          │
│                                                                      │
│  ✅ 关键规则：                                                         │
│  1. 有条件通过 ≠ 通过                                                 │
│  2. 有条件通过必须经过「修复→重新评审」循环                           │
│  3. 只有「100% 通过」才能进入下一阶段                                 │
│  4. 重新评审通过后，评审结论更新为「通过」                            │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 评审结论判定标准（3 级制 + 动作映射）

| 评审结论 | 判定标准 | 后续动作 | 是否可以进入下一阶段 |
|---------|---------|---------|-------------------|
| **✓ 通过** | 所有自检项通过，无 minor 建议 | 直接进入下一阶段 | ✅ **可以** |
| **△ 有条件通过** | 基本可用，但有 minor 问题 | **必须**修复并重新评审 | ❌ **不可以**（直到转为「通过」） |
| **✗ 不通过** | 有阻塞性问题，不满足落地要求 | **必须**修复并重新评审 | ❌ **不可以**（直到转为「通过」） |

### 重要澄清

| 理解类型 | 描述 | 正确性 |
|---------|------|--------|
| **旧理解（错误）** | 「有条件通过」= 「可并行进入后续阶段，但须完成优化项」 | ❌ 错误 |
| **新理解（正确）** | 「有条件通过」= 「必须修复问题，重新评审通过后，才能进入下一阶段」 | ✅ 正确 |

## 在 8+1 质量闭环中的应用

### 升级后的 8+1 闭环（含评审修复循环）

```
Step 1: 需求分析 (request-analysis)
    │
    ▼
Step 2: PRD 评审 (prd-review)
    │
    ├──→ 通过 ──→ Step 3
    │
    ├──→ 有条件通过 ──→ 修复问题 ──→ 重新评审 ──→ 通过？──→ 否 → 继续修复
    │                                              └──────→ 是 → Step 3
    │
    └──→ 不通过 ──→ 修复问题 ──→ 重新评审 ──→ 通过？──→ 否 → 继续修复
                                               └──────→ 是 → Step 3

Step 3: 工程结构分析 (project-analysis)
    │
    ├──→ 通过 ──→ Step 4
    │
    ├──→ 有条件通过 ──→ 修复 ──→ 重新评审 ──→ 通过 ──→ Step 4
    │
    └──→ 不通过 ──→ 修复 ──→ 重新评审 ──→ 通过 ──→ Step 4

Step 4: 技术方案评审 (architecture-review)
    │
    ├──→ 通过 ──→ Step 5
    │
    ├──→ 有条件通过 ──→ 修复 ──→ 重新评审 ──→ 通过 ──→ Step 5
    │
    └──→ 不通过 ──→ 修复 ──→ 重新评审 ──→ 通过 ──→ Step 5

Step 5-9: 编码实现 → 代码评审 → 功能验收 → 归档 → 复盘
    （同样适用评审修复循环）
```

### 每个 Step 的准出条件（升级后）

| Step | 准出条件（旧） | 准出条件（新，升级后） |
|------|---------------|---------------------|
| Step 1 | PRD 已产出 | PRD 已产出 **且评审结论为「通过」** |
| Step 2 | PRD 评审完成 | PRD 评审完成 **且结论为「通过」（非「有条件通过」）** |
| Step 3 | 技术方案已产出 | 技术方案已产出 **且评审结论为「通过」** |
| Step 4 | 技术方案评审完成 | 技术方案评审完成 **且结论为「通过」（非「有条件通过」）** |
| Step 5-8 | 按原规范 | 代码评审、功能验收同样适用评审修复循环 |

## 重新评审的规范

### 重新评审的触发条件

| 场景 | 触发动作 | 重新评审范围 |
|------|---------|-------------|
| 首次评审为「有条件通过」 | 修复所有 minor 问题后 | 仅评审修改的部分 + 关键关联内容 |
| 首次评审为「不通过」 | 修复所有问题后 | 全量重新评审 |
| 修复后仍不通过 | 继续修复后 | 仅评审剩余问题 |

### 重新评审的记录格式

**迭代日志记录**:
```markdown
# 首次评审（有条件通过）
- YYYY-MM-DD | [change-id] | prd-review | PRD 评审完成，综合判定：有条件通过，问题清单：[路径]，评审纪要：[路径]

# 修复后重新评审（通过）
- YYYY-MM-DD | [change-id] | prd-review | PRD 重新评审完成，修复项：[N 项]，综合判定：通过，评审纪要：[路径]

# 进入下一阶段
- YYYY-MM-DD | [change-id] | project-analysis | 技术方案产出完成，design.md 路径：[路径]
```

**评审纪要命名**:
- 首次评审：`PRD-[change-id]-评审纪要.md`
- 重新评审：`PRD-[change-id]-重新评审纪要.md`

## 使用示例

### 示例：PRD 评审修复循环

**Step 1**: PRD 首次评审（2026-03-14）
- 评审结论：△ 有条件通过
- 发现问题：2 项 minor 建议
  1. 竞品调研章节可补充 CI/CD、Saga 模式引用
  2. Non-Goals 界定可更明确
- 评审纪要：`PRD-check-langgraph-backend-v2.0-评审纪要.md`

**循环节点**: 进入评审修复循环，不进入 Step 3

**Step 2**: 修复问题（2026-03-16）
- 修复项 1：在 PRD 9.2 节补充相关模式引用
- 修复项 2：在 9.4.3 节补充第二阶段特有的 Non-Goals
- 迭代日志记录修复完成

**Step 3**: PRD 重新评审（2026-03-16）
- 评审范围：仅评审修改的部分
- 评审结论：✓ 通过
- 评审纪要：`PRD-check-langgraph-backend-v2.0-重新评审纪要.md`

**Step 4**: 进入 Step 3（工程结构分析）
- 准出条件满足：PRD 评审结论为「通过」

## 质量门禁检查清单

在准备进入下一阶段前，必须检查：

| 检查项 | 检查内容 | 判定标准 |
|-------|---------|---------|
| 1 | 前一阶段评审是否完成？ | 必须完成 |
| 2 | 前一阶段评审结论是什么？ | 必须是「通过」（✓） |
| 3 | 是否有「有条件通过」或「不通过」？ | 如有，必须已完成重新评审并转为「通过」 |
| 4 | 迭代日志是否有「重新评审」记录？ | 如有有条件通过/不通过，必须有 |
| 5 | 评审纪要路径是否可访问？ | 必须能读取到评审纪要 |

**检查失败时的处理**:
- 如果检查 1-4 任一项失败 → **暂停进入下一阶段**，先完成评审修复循环
- 如果检查 5 失败 → 尝试重新生成评审纪要或标记为缺失
