# 5 阶段复盘法

## 一句话定义

系统化的复盘方法论，通过回顾目标→评估结果→分析原因→提炼经验→行动计划五个阶段，将经验转化为可复用的知识资产。

## 问题背景

### 常见的复盘误区

```
┌─────────────────────────────────────────────────────────────┐
│                    常见的复盘误区                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  误区 1: 流水账式复盘                                          │
│  • 做了什么 → 做了什么 → 做了什么                              │
│  • 问题: 只是记录，没有分析、没有学习                          │
│                                                              │
│  误区 2: 追责式复盘                                            │
│  • 谁的错 → 为什么错 → 怎么惩罚                                │
│  • 问题: 团队成员不敢说真话，隐瞒问题                          │
│                                                              │
│  误区 3: 表面分析                                              │
│  • 超时了 → 超时时间不够 → 调大超时                            │
│  • 问题: 没有找到根本原因，问题反复出现                          │
│                                                              │
│  误区 4: 无行动复盘                                            │
│  • 分析得头头是道，但没有行动计划                                │
│  • 问题: 复盘沦为空谈，无法落地改进                              │
│                                                              │
│  误区 5: 不复盘                                                │
│  • 做完就做完了，不复盘不总结                                    │
│  • 问题: 经验随时间流失，同样问题重复踩坑                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 各阶段详细执行指南

#### Stage 1: Review Goals（回顾目标）

**目的**: 明确当初的目标和预期，为后续评估提供基准

**执行要点**:
```
1. 列出初始目标（从 proposal.md、tasks.md 中提取）
2. 列出预期产出（从 PRD、技术方案中提取）
3. 列出关键成功指标（从验收标准中提取）
4. 列出实际完成的目标（客观记录）

关键原则:
• 基于文档，不依赖记忆
• 区分「目标」和「手段」
• 量化指标优先
```

**常见问题**:
```
❌ 没有明确目标就开始复盘
❌ 目标模糊，无法评估（如「做好这个项目」）
❌ 混淆目标和手段（如「写 PRD」是手段，不是目标）
```

#### Stage 2: Evaluate Results（评估结果）

**目的**: 对比实际结果与预期目标，识别亮点和问题

**执行要点**:
```
1. 对比表: 目标 vs 实际 vs 偏差
2. 亮点清单: 做得好的地方 + 为什么好
3. 问题清单: 做得不好的地方 + 影响
4. 意外发现: 计划外的收获或问题

关键原则:
• 客观事实，不回避问题
• 庆祝亮点，不只看问题
• 量化对比，不凭感觉
```

**对比表示例**:
```markdown
| 目标 | 预期 | 实际 | 偏差 | 说明 |
|-----|------|------|------|------|
| 功能完整性 | 5 项检查 | 5 项检查 | ✓ 达成 | 全部实现 |
| 易用性 | 5 分钟内 | 1-2 分钟 | ✓ 超额 | 执行效率高 |
| 错误诊断 | 5 类问题 | 6 类问题 | ✓ 超额 | 补充了权限问题 |
```

#### Stage 3: Analyze Causes（分析原因）

**目的**: 找到问题的根本原因，理解成功因素

**执行要点**:
```
1. 5 个 Why 分析:
   问题 → Why? → Why? → Why? → Why? → 根本原因

2. 问题分类:
   • 内因 vs 外因（自己可控 vs 外部因素）
   • 可控 vs 不可控（能改变 vs 不能改变）
   • 系统性 vs 偶然（会重复发生 vs 偶然发生）

3. 思维模式分析（如适用）:
   • 不同角色的思维差异
   • AI vs 用户的思维模式对比

关键原则:
• 找到根本原因，而不是表面原因
• 区分可控和不可控，聚焦可控因素
• 识别系统性问题，需要框架级解决
```

**5 个 Why 示例**:
```markdown
问题: 为什么自检脚本出现超时？

Why 1: 为什么超时？
→ 执行耗时 7~10 分钟，超过了脚本 120s 超时

Why 2: 为什么执行这么久？
→ 一次性执行完整的 change-id，包含 8 个任务

Why 3: 为什么必须一次性执行？
→ /run API 设计假设：单次调用完成完整 workflow

Why 4: 为什么这样设计？
→ 早期需求简单，没有考虑长时间运行场景

Why 5: 能否改变设计？
→ 可以，阶段化执行：/run(change_id, phase="env-check")

根本原因: 架构设计假设与实际场景不匹配
```

#### Stage 4: Extract Learnings（提炼经验）

**目的**: 将经验提炼为可复用的知识资产

**执行要点**:
```
1. 模式提炼:
   • 可复用的解决方案（pattern）
   • 最佳实践（preference）
   • 执行手册（playbook）

2. 反模式识别:
   • 应该避免的做法（anti-pattern）
   • 常见坑点

3. 流程改进:
   • 现有流程的问题
   • 改进建议

4. Memory 沉淀评估:
   • 是否有长期价值？
   • 是否会在 ≥2 个场景中出现？
   • 是否抽象出清晰的模式？

关键原则:
• 从个案抽象出通用模式
• 关注「为什么有效」，不只是「做了什么」
• 评估沉淀价值，避免过度沉淀
```

**经验提炼示例**:
```markdown
| 经验 | 类型 | 适用场景 | 沉淀价值 |
|-----|------|---------|---------|
| 突破性思维四步法 | pattern | 面对复杂问题 | 高 |
| 阶段化执行模式 | pattern | 长时间运行任务 | 高 |
| 评审留痕机制 | pattern | PRD/技术方案评审 | 高 |
| 头痛医头 | anti-pattern | 任何问题解决 | 高 |
```

#### Stage 5: Action Plan（行动计划）

**目的**: 将复盘结论转化为可执行的行动

**执行要点**:
```
1. 短期行动（1-3 天）:
   • 立即可以执行
   • 快速见效
   • 责任人明确

2. 中期改进（1-2 周）:
   • 下个迭代完成
   • 需要一定投入
   • 有明确验收标准

3. 长期演进（1-3 月）:
   • 架构/模式级改进
   • 需要较大投入
   • 长期价值高

关键原则:
• 行动必须可执行、可验证
• 明确责任人、时间节点、验收标准
• 优先级排序，聚焦高价值行动
```

**行动计划表示例**:
```markdown
| 行动 | 具体任务 | 责任人 | 完成标准 | 时间节点 | 优先级 |
|-----|---------|-------|---------|---------|-------|
| 修复拼写错误 | prrd→prd | 主 Agent | 全局无错误 | 2026-03-14 | P0 |
| 试点质量闭环 | 新 change-id 应用 | 主 Agent | 完成 1-2 个 | 2026-03-21 | P1 |
| LangGraph 阶段化 | 实现 phase 参数 | 后端 Agent | API 可用 | 2026-04-14 | P2 |
```

## 复盘类型与适配

### 不同复盘类型的侧重

| 复盘类型 | 侧重阶段 | 典型场景 | 报告侧重 |
|---------|---------|---------|---------|
| **问题诊断型** | Stage 3（分析原因） | 故障/问题后 | 根因分析、修复方案 |
| **经验沉淀型** | Stage 4（提炼经验） | 重要工作完成后 | 模式提炼、知识沉淀 |
| **流程优化型** | Stage 4、5（经验、行动） | 流程机制问题 | 流程改进、效率提升 |
| **学习成长型** | Stage 4（提炼经验） | 定期回顾 | 学习总结、能力提升 |
| **项目结项型** | 全阶段 | 项目/迭代结束 | 全面总结、经验库 |

### 复盘频率建议

| 复盘类型 | 频率 | 时长 | 参与人 |
|---------|------|------|-------|
| **每日站会** | 每天 | 15 分钟 | 核心团队 |
| **迭代复盘** | 每迭代 | 1-2 小时 | 全团队 |
| **项目复盘** | 项目结束 | 2-4 小时 | 核心成员 + 利益相关方 |
| **事故复盘** | 事故后 | 2-4 小时 | 核心团队 + 专家 |
| **季度复盘** | 每季度 | 半天 | 全团队 |

## 质量门禁与自检

### 复盘质量自检清单（9 项）

| # | 检查项 | 说明 | 权重 |
|---|-------|------|------|
| 1 | 目标回顾完整 | 清晰回顾初始目标和预期结果 | 必备 |
| 2 | 结果对比清晰 | 对比实际结果 vs 预期目标 | 必备 |
| 3 | 问题识别全面 | 识别所有关键问题 | 必备 |
| 4 | 根因分析深入 | 用 5 个 Why 找到根本原因 | 必备 |
| 5 | 思维模式分析 | 分析思维差异（如适用） | 推荐 |
| 6 | 经验提炼可复用 | 提炼的经验抽象、可复用 | 必备 |
| 7 | 行动计划可执行 | 行动具体、有责任人、有时间 | 必备 |
| 8 | memory 沉淀价值 | 沉淀的 memory 有长期价值 | 推荐 |
| 9 | 文档规范符合 | 符合复盘报告模板 | 必备 |

### 评审判定标准

- **通过**: 7 项及以上通过，无不通过项（必备项全部通过）
- **有条件通过**: 5-6 项通过，无不通过项
- **不通过**: 少于 5 项通过，或有不通过项

## 实施建议

### 个人层面

```
每日复盘（5 分钟）:
• 今天做了什么？
• 有什么收获？
• 明天改进什么？

每周复盘（30 分钟）:
• 本周目标完成度？
• 做得好的/不好的？
• 下周重点是什么？
```

### 团队层面

```
迭代复盘（1-2 小时）:
• 准备: 提前收集数据、整理时间线
• 进行: 按 5 阶段结构化讨论
• 产出: 复盘报告、行动计划、memory 沉淀
• 跟进: 检查行动计划执行情况
```

### 组织层面

```
建立复盘文化:
• 领导示范: 带头复盘，重视学习
• 时间保障: 预留复盘时间，不压缩
• 心理安全: 不追责，鼓励说真话
• 知识管理: 复盘报告归档，memory 沉淀
• 持续改进: 定期回顾复盘质量
```

## 常见陷阱与避免方法

| 陷阱 | 表现 | 避免方法 |
|-----|------|---------|
| **流水账** | 只记录过程，没有分析 | 强制按 5 阶段执行 |
| **追责** | 找谁的错，不找根因 | 强调学习而非追责 |
| **表面** | 只分析表面原因 | 强制 5 个 Why |
| **空谈** | 没有行动计划 | 阶段 5 必须有可执行行动 |
| **形式** | 为了复盘而复盘 | 与 memory 沉淀绑定 |
