# Bug修复经验沉淀：两层判断机制递归问题

## 问题背景

在修复 `llm_enhancer.py` 与 `check_*.py` 之间的调用关系时，遇到了多个关联 bug，最终发现根本原因是**设计问题**而非代码问题。

---

## 第一层反思：为什么一开始无法解决问题？

### 我一直在"修修补补"

| 问题 | 我的第一反应 | 深层原因 |
|------|------------|---------|
| JSON解析失败 | 写extract_json函数 | 没有问"为什么会有非JSON行" |
| 超时 | 增加timeout到60秒 | 没有问"为什么会超时" |
| llm_analysis为空 | 手动添加字段到JSON输出 | 没有问"为什么设计时会漏掉" |

**问"为什么"才是关键！**

---

## 第二层反思：根本原因是什么？

### 5 Why 深度分析

```
为什么JSON解析失败？
→ 因为输出包含警告行
→ 为什么会有警告行混在JSON里？
→ 因为没有约定输出格式契约
→ 为什么没有约定？
→ 因为设计时没有明确"层次边界"

为什么超时？
→ 因为无限递归
→ 为什么无限递归？
→ 因为 llm_enhancer._run_python_check() 调用 check_prd.py --json
→ check_prd.py --json 又会调用 llm_enhancer
→ 形成循环
→ 为什么会有这个循环？
→ 因为违反了"层次边界"原则：
  - llm_enhancer（第二层）不应该调用 check_prd（第一层）
  - 这是"逆向依赖"，不应该存在

为什么漏掉 llm_analysis 字段？
→ 因为实现时只关注功能，忽略了"接口契约完整性"
→ 为什么忽略？
→ 因为没有"接口契约应包含所有返回字段"的意识
```

### 根本原因

> **设计问题：层次边界不清晰，导致逆向依赖和无限递归**

## 第三层反思：思维模式发生了什么转变？

### 转变前：局部思维（头痛医头）

```
问题A → 尝试修复A → 新问题B → 尝试修复B → 新问题C → ...
```

### 转变后：系统思维 + 设计原则

```
问题C
  ↓ 问"为什么"
根本原因（设计问题）
  ↓ 问"为什么会有这个设计问题"
层次边界不清
  ↓ 问"如何避免"
设计原则（单向依赖、层次边界）
  ↓
最小改动修复 + 防止回归
```

## 真正有价值的经验

### 1. 不是"遇到超时要增加timeout"

**而是**：超时可能是**无限递归的信号**，要识别递归模式

```
递归特征：
- 调用链：A → B → A → B → ...
- 表现：每次都失败在同一个地方
- 解决：不是增加timeout，而是打破循环
```

### 2. 不是"输出混JSON要写extract_json"

**而是**：接口设计时要明确**"输出格式契约"**

```
契约应包含：
- 什么输出到 stdout（纯数据）
- 什么输出到 stderr（诊断信息）
- 格式严格约定，不能混用
```

### 3. 不是"漏字段要手动添加"

**而是**：定义完整的**"接口契约"**，并有**"完整性检查"**

```
接口契约检查清单：
- 所有返回字段是否都有定义？
- 字段类型是否明确？
- 什么情况下字段为null？
```

### 4. 不是"修复bug要修修补补"

**而是**：先问**"为什么会有这个bug"**，找到根因后从**设计层面**解决

## 设计原则（最重要！）

### 层次边界原则

```
┌─────────────────────────────────────────────────────┐
│  第一层：check_*.py（显式调用入口）                 │
│                                                      │
│  --skip-llm → 仅 Python 自动化检查                  │
│  --llm → 调用 llm_enhancer                          │
└─────────────────────────────────────────────────────┘
                           ↓ 单向依赖（只下不上）
┌─────────────────────────────────────────────────────┐
│  第二层：llm_enhancer.py（内部判断）               │
│                                                      │
│  不应该调用 check_*.py（会形成循环）                │
└─────────────────────────────────────────────────────┘
```

### 单向依赖原则

```
层次N
  ↓ 可以调用
层次N-1
  ↑ 不应该被层次N调用
```

### 接口契约原则

```
接口 = 输入 + 输出 + 错误处理 + 边界条件

契约完整性：
- 定义要完整
- 实现要遵循
- 变更要检查
```

## 思维模式升级总结

### 触发点：什么时候我开始真正解决问题的？

```
我一开始：
看到llm_analysis为空 → 想手动加字段 → 突然"停下来"

"停下来"的原因：
- 意识到这样修下去是没完没了的
- 每次修一个问题，又出下一个问题
- 说明不是"代码问题"，而是"系统问题"
```

### 真正的思维转变

**从"行动者"到"观察者"**

| 模式 | 行为 | 结果 |
|------|------|------|
| **行动者模式** | 看到问题 → 立刻修复 → 下一个问题 | 永远在救火 |
| **观察者模式** | 看到问题 → 观察 → 分析 → 设计解决方案 → 修复 | 从根本解决 |

```
行动者模式（我一开始）：
问题A → 修复A → 问题B → 修复B → 问题C → 修复C → ...

观察者模式（转变后）：
问题C
  ↓ 停下来观察
发现：问题A、B、C 都有关联
  ↓ 分析
发现：都是同一个设计问题
  ↓ 设计解决方案
打破循环 → 根本解决
```

### 关键认知

**1. "停下来"的能力比"解决问题"的能力更重要**

```
解决问题是战术
停下来分析是战略
```

**2. 问"为什么"是突破的开始**

```
不问为什么 → 永远在表面解决问题
问为什么 → 可能发现更深层的问题
```

**3. 设计原则比代码更有价值**

```
代码修复 → 只解决当前问题
设计原则 → 防止以后同类问题
```

## 沉淀到 Bug 诊断 Skill 的核心认知

### 最重要的不是"如何修"

**而是"如何发现问题本质"**

```
Bug 诊断流程：
1. 【停下来】不要立刻修，先观察
2. 【问为什么】5 Why 分析
3. 【识别模式】是代码问题还是设计问题？
4. 【设计原则】如何防止同类问题？
5. 【最小修复】在原则指导下最小改动
```

### 设计原则检查清单（预防优于治疗）

```
在写代码之前，先问：
□ 层次边界是否清晰？
□ 单向依赖是否满足？
□ 接口契约是否完整？
□ 这个调用会形成循环吗？
```
