# 从具体问题到框架级演进的复盘反思

## 反思背景

### 起点：一个具体的超时问题

```
问题: check-langgraph-backend 自检脚本 POST /run 超时 120s，
      但后端实际执行需要 400~600s

表面问题: 超时时间不够
初步方案: 调大超时时间（治标）
```

### 终点：框架级改进 + 5 个记忆沉淀

```
成果:
1. 架构改进: 阶段化执行模式（phase / stage / checkpoint）
2. 规范升级: PRD/技术方案评审留痕机制（5 章节评审纪要）
3. 命名规范: PRD-[change-id]-[关键词].md 新格式
4. 技能创建: prd-review、architecture-review 两个新 skills
5. 记忆沉淀: 5 个长期可复用的 memory
```

### 关键问题

**为什么能从具体问题演进为框架改进？**
**如何避免「头痛医头、脚痛医脚」的局部优化？**
**如何将一次性解决方案沉淀为可复用的知识？**

## 关键成功因素

### 因素 1: 深度问题诊断（不只是表面现象）

```
❌ 浅层诊断:
   超时 → 超时时间不够 → 调大超时

✅ 深度诊断:
   超时 → 为什么超时？→ 执行耗时 7~10 分钟
   → 为什么这么久？→ 一次性执行完整 change-id
   → 为什么必须一次性？→ 架构设计假设
   → 能否改变假设？→ 阶段化执行

关键动作:
• 收集足够的现象和数据（日志、耗时、task_range 测试结果）
• 问「为什么」直到触及架构/设计层面的假设
• 区分「本质约束」和「偶然约束」
```

### 因素 2: 思维模式突破（用户的突破性思维）

```
❌ AI 的局限:
   模式库依赖 → 只能在已有模式中找解决方案
   上下文窄化 → 只关注当前 change-id 的具体问题
   风险规避 → 倾向于保守的局部优化

✅ 用户的优势:
   系统思维 → 看到问题背后的结构性矛盾
   模式迁移 → 将其他领域的模式迁移到新场景
   长期视角 → 不追求最快短期修复，追求最健康的长期架构
   突破勇气 → 愿意接受架构级改动的复杂度

关键动作:
• 当用户提出突破性思路时，识别并学习
• 记录「用户能提出而 AI 不能」的思维差异
• 将突破性思维固化为可复用的模式
```

### 因素 3: 系统性规范升级（不只是单点修复）

```
❌ 单点修复:
   只解决 check-langgraph-backend 的超时问题
   → 其他 change-id 可能遇到同样问题

✅ 系统性升级:
   阶段化执行模式 → 所有 change-id 受益
   评审留痕机制 → 所有 PRD/技术方案受益
   命名规范 → 所有文档受益
   技能创建 → 所有项目受益

关键动作:
• 从单点问题抽象出通用模式
• 升级规范，让模式成为标准
• 创建技能，让规范可执行
```

### 因素 4: 显性映射建立（确保可执行）

```
❌ 隐式知识:
   流程定义了，但不知道如何触发
   Agent 与 Skill 的关系在文档里，但执行时不明确

✅ 显性映射:
   步骤 → Agent → Skill → Memory → 产出物
   触发指令 → 步骤识别 → Agent 指派 → Skill 加载 → Memory 唤醒

关键动作:
• 创建 playbook 建立显性映射
• 定义触发词和唤醒条件
• 提供可执行的手册
```

### 因素 5: 知识沉淀与记忆化（确保可复用）

```
❌ 一次性解决:
   问题解决后，经验随时间流失
   新团队遇到同样问题，重新踩坑

✅ 长期记忆:
   经验沉淀为 pattern、preference、playbook、reflection
   后续遇到类似问题，自动唤醒记忆
   知识在团队间传承

关键动作:
• 将经验抽象为可复用的 memory
• 定义唤醒条件和适用场景
• 建立 memory 间的关联关系
```

## 演进过程中的关键决策点

### 决策点 1: 是否接受架构级改动？

```
决策场景:
  用户提出「阶段化执行」架构改进方案
  vs
  AI 提出的「调大超时」参数调整方案

决策分析:
  • 短期成本: 架构改动 > 参数调整
  • 长期收益: 架构改动 >> 参数调整
  • 系统健康: 架构改动从根本上解决问题
  • ROI: 架构改动是一次性投入，长期受益

决策结果: 接受架构级改动，纳入 proposal 第二阶段
```

### 决策点 2: 是否建立评审留痕机制？

```
决策场景:
  识别到「迭代日志 ≠ 评审纪要」的缺口
  是否投入成本建立独立的评审留痕机制？

决策分析:
  • 成本: 需要创建 skills、REFERENCE、memory
  • 收益: 质量可追溯、可审计、可学习
  • 必要性: 没有评审留痕，质量闭环不完整
  • 可复制性: 机制建立后可复用于所有项目

决策结果: 建立完整的评审留痕机制
```

### 决策点 3: 是否创建显性映射手册？

```
决策场景:
  流程和技能已定义，但触发机制不明确
  是否创建 playbook 建立 Agent-Skill-Memory 映射？

决策分析:
  • 成本: 需要编写详细的手册
  • 收益: 流程可执行、可唤醒、可自动化
  • 必要性: 没有映射，流程可能流于形式
  • 长期价值: 映射是流程落地的关键

决策结果: 创建 playbook-agent-skill-memory-mapping-for-quality-closed-loop
```

## 可复用的方法论

### 方法论 1: 问题诊断的 5 个 Why

```
面对任何问题，连续问 5 个「为什么」：

问题: 超时
Why 1: 为什么超时？→ 执行耗时 7~10 分钟
Why 2: 为什么这么久？→ 一次性执行完整 change-id
Why 3: 为什么必须一次性？→ /run API 不支持分段执行
Why 4: 为什么不支持？→ 架构设计假设
Why 5: 能否改变假设？→ 可以，阶段化执行

结果: 从表面问题（超时）到根本原因（架构假设）
```

### 方法论 2: 突破性思维的 4 步思考法

```
面对任何约束，按以下顺序思考：

1. 先问「为什么」
   → 为什么必须有这个限制？
   → 是本质的还是偶然的？

2. 再问「如果不」
   → 如果不接受这个限制，世界会怎样？
   → 有什么新可能性？

3. 三问「如何设计」
   → 如何重新设计系统，让原有问题不再存在？
   → 新架构的关键要素是什么？

4. 最后「价值验证」
   → 新方案的 ROI 如何？
   → 短期成本 vs 长期收益
```

### 方法论 3: 从单点到框架的抽象过程

```
单点问题              通用模式              框架升级
   │                    │                    │
   ▼                    ▼                    ▼
check-langgraph-    阶段化执行模式      phase/stage/checkpoint
backend 超时          （pattern）          协议升级

具体文档              通用机制              规范升级
   │                    │                    │
   ▼                    ▼                    ▼
迭代需求说明.md     评审留痕机制         PRD/技术方案评审规范
评审走过场           （pattern）          （5 章节评审纪要）
```

### 方法论 4: 知识沉淀的 4 层结构

```
经验 → 模式 → 规范 → 技能 → 记忆
 │      │       │       │       │
 ▼      ▼       ▼       ▼       ▼
具体   抽象   标准化   可执行   可唤醒
案例   模式   规范     工具     记忆
```

## 反思中的反思

### 本次复盘的局限

1. **时间局限**: 复盘在问题解决后立即进行，可能缺乏长期视角
2. **范围局限**: 主要聚焦于 check-langgraph-backend 一个 change-id
3. **验证局限**: 新的规范、技能、记忆还需要在实际项目中验证

### 未来改进方向

1. **长期跟踪**: 跟踪 5 个 memory 在实际项目中的唤醒频率和效果
2. **持续优化**: 根据实际使用情况，优化评审清单和判定标准
3. **扩展应用**: 将方法论应用到其他 change-id，验证通用性
4. **团队培训**: 通过复盘文档和记忆，培训新成员理解规范

## 对其他项目的启示
