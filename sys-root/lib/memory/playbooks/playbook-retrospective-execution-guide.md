# 复盘执行操作手册

## 手册定位

本手册是 `pattern-five-stage-retrospective` 的**可执行版本**，提供复盘执行的**详细步骤、触发指令、检查清单、产出模板**，解决「知道要复盘但不知道怎么执行」的问题。

## 快速启动指南

### 触发复盘的 3 种方式

```
方式 1: 用户明确指令
• 用户说: "复盘一下最近的工作"
• 识别: 触发 retrospective-analysis skill
• 执行: 按本手册执行 5 阶段复盘

方式 2: Agent 自动触发
• 场景: 完成重要里程碑/出现问题后
• 识别: 主 Agent 自动判定需要复盘
• 执行: 建议用户「是否进行复盘？」

方式 3: 定期复盘
• 场景: 迭代结束/季度末
• 识别: 时间触发
• 执行: 主动发起复盘
```

### 复盘前的 3 个确认

```
确认 1: 复盘范围
• 时间范围: 最近 N 天？某个 change-id？某个项目？
• 关键事件: 是否有明确的触发事件？

确认 2: 复盘目标
• 问题诊断？经验沉淀？流程优化？
• 预期的复盘产出是什么？

确认 3: 复盘材料
• 迭代日志是否完整？
• 相关文档是否可以访问？
```

## 详细执行步骤

### Step 0: 准备阶段（10-15 分钟）

#### 0.1 确定复盘范围与目标

**执行指令**:
```
向用户确认:
1. "本次复盘的时间范围是什么？"
   - 最近 2 天？
   - 某个 change-id？
   - 某个时间段？

2. "是否有特定的触发事件？"
   - 出现了问题？
   - 完成了重要工作？
   - 定期复盘？

3. "希望通过复盘达成什么目标？"
   - 找出问题根因？
   - 提炼可复用经验？
   - 优化流程机制？
```

**检查清单**:
- [ ] 时间范围已明确
- [ ] 触发事件已识别
- [ ] 复盘目标已确定

#### 0.2 收集复盘材料

**执行指令**:
```
收集以下材料:
1. 迭代日志: docs/项目事件日志.md
   → 了解完整时间线

2. 相关文档:
   → openspec/changes/[change-id]/proposal.md
   → openspec/changes/[change-id]/tasks.md
   → docs/project-prd-changes/[change-id]/PRD-xxx.md
   → openspec/changes/[change-id]/design.md
   → docs/project-prd-changes/[change-id]/records/

3. 聊天记录/决策过程:
   → 如有对话记录，提取关键决策点

4. 产出物:
   → 代码变更
   → 配置变更
   → 文档产出
```

**检查清单**:
- [ ] 迭代日志已读取
- [ ] 关键文档已收集
- [ ] 产出物清单已整理

#### 0.3 制定复盘计划

**执行指令**:
```
制定计划:
1. 确定复盘时长: 建议 1-3 小时（根据复杂度）
2. 确定执行 Agent: 
   - 技术问题 → 架构 Agent
   - 产品问题 → 产品经理 Agent
   - 整体复盘 → 主 Agent
3. 确定复盘类型:
   - 问题诊断型
   - 经验沉淀型
   - 流程优化型
   - 学习成长型
   - 项目结项型
```

**产出**:
```markdown
## 复盘计划

**复盘主题**: [主题]
**时间范围**: YYYY-MM-DD 至 YYYY-MM-DD
**复盘目标**: [目标]
**复盘类型**: [类型]
**执行 Agent**: [Agent 角色]
**预计时长**: N 小时
**关键材料**:
- [ ] 迭代日志
- [ ] proposal.md
- [ ] tasks.md
- [ ] PRD
- [ ] 其他
```

### Step 1: 回顾目标（Stage 1: Review Goals）（15-20 分钟）

#### 1.1 提取初始目标

**执行指令**:
```
从以下文档提取目标:

1. proposal.md:
   → 查找 "Why" 章节（背景与目标）
   → 查找 "What Changes" 章节（预期产出）
   → 查找 "Impact" 章节（影响范围）

2. tasks.md:
   → 查找任务列表（目标拆解）
   → 查找验收标准（成功指标）

3. PRD（如存在）:
   → 查找 "迭代目标与成功标准" 章节
   → 查找 "需求验收 Checklist"
```

**产出模板**:
```markdown
### 2.1 初始目标

**核心目标**:
[从 proposal.md 提取的核心目标]

**具体目标**:
1. [具体目标 1]
2. [具体目标 2]
3. [具体目标 3]

**关键成功指标**:
| 指标 | 目标值 | 说明 |
|-----|-------|------|
| [指标1] | [目标值] | [说明] |
| [指标2] | [目标值] | [说明] |
```

#### 1.2 提取预期产出

**执行指令**:
```
从以下文档提取预期产出:

1. proposal.md → "Affected code" 章节
2. tasks.md → 任务列表
3. PRD → "功能列表与验收标准" 章节
```

**产出模板**:
```markdown
### 2.2 预期产出

**文档产出**:
- [ ] PRD
- [ ] 技术方案（design.md）
- [ ] 评审纪要
- [ ] 验收记录

**代码产出**:
- [ ] [代码文件 1]
- [ ] [代码文件 2]

**规范产出**:
- [ ] 符合 OpenSpec 规范
- [ ] 通过 func-test 验收
```

#### 1.3 检查清单

- [ ] 初始目标已提取
- [ ] 成功指标已识别
- [ ] 预期产出已列出
- [ ] 实际完成情况已记录

### Step 2: 评估结果（Stage 2: Evaluate Results）（20-30 分钟）

#### 2.1 对比实际 vs 预期

**执行指令**:
```
制作对比表:

对于每个目标/指标:
1. 预期值: [从 Step 1 提取]
2. 实际值: [从迭代日志、产出物评估]
3. 偏差: ✓ 达成 / ✓ 超额 / ✗ 未达成 / △ 部分达成
4. 说明: [具体说明]

评估维度:
• 功能完整性
• 质量（性能、稳定性、安全）
• 进度（是否按时）
• 成本（资源投入）
• 规范符合度
```

**产出模板**:
```markdown
### 3.1 实际结果 vs 预期目标

| 目标 | 预期 | 实际 | 偏差 | 说明 |
|-----|------|------|------|------|
| [目标1] | [预期] | [实际] | ✓/✗/△ | [说明] |
| [目标2] | [预期] | [实际] | ✓/✗/△ | [说明] |

**总体评估**:
[对整体完成情况的评估]
```

#### 2.2 识别亮点

**执行指令**:
```
识别亮点:

1. 回顾所有完成的工作
2. 找出超出预期的部分
3. 找出做得特别有创意的部分
4. 找出团队协作特别好的部分

对于每个亮点:
• 具体表现是什么？
• 为什么做得好？（关键因素）
• 是否可以复用？（可复用性）
```

**产出模板**:
```markdown
### 3.2 做得好的地方（亮点）

| 亮点 | 具体表现 | 成功因素 | 可复用性 |
|-----|---------|---------|---------|
| [亮点1] | [表现] | [因素] | [高/中/低] |
| [亮点2] | [表现] | [因素] | [高/中/低] |

**亮点总结**:
[对亮点的整体总结]
```

#### 2.3 识别问题

**执行指令**:
```
识别问题:

1. 回顾所有未完成/未达标的工作
2. 回顾过程中的阻碍和困难
3. 回顾意外发生的问题
4. 回顾质量不达标的部分

对于每个问题:
• 具体表现是什么？
• 影响范围有多大？
• 严重程度（高/中/低）
```

**产出模板**:
```markdown
### 3.3 做得不好的地方（问题）

| 问题 | 具体表现 | 影响 | 严重程度 |
|-----|---------|------|---------|
| [问题1] | [表现] | [影响] | [高/中/低] |
| [问题2] | [表现] | [影响] | [高/中/低] |

**问题总结**:
[对问题的整体总结]
```

#### 2.4 记录意外发现

**执行指令**:
```
记录意外:

1. 好的意外:
   • 哪些收获是计划外的？
   • 哪些能力提升是意外的？

2. 坏的意外:
   • 哪些问题是意料之外的？
   • 哪些风险是没有预见到的？
```

**产出模板**:
```markdown
### 3.4 意外发现

**好的意外**:
1. [意外发现 1]
2. [意外发现 2]

**坏的意外**:
1. [意外问题 1]
2. [意外问题 2]
```

#### 2.5 检查清单

- [ ] 对比表已完成
- [ ] 亮点已识别（至少 2-3 个）
- [ ] 问题已识别（不回避）
- [ ] 意外发现已记录

### Step 3: 分析原因（Stage 3: Analyze Causes）（30-45 分钟）

#### 3.1 5 个 Why 分析

**执行指令**:
```
对关键问题执行 5 个 Why:

问题: [选择最关键的问题，通常是影响最大的]

Why 1: 为什么 [问题]？
→ [答案 1]

Why 2: 为什么 [答案 1]？
→ [答案 2]

Why 3: 为什么 [答案 2]？
→ [答案 3]

Why 4: 为什么 [答案 3]？
→ [答案 4]

Why 5: 为什么 [答案 4]？
→ [根本原因]

**注意**:
• 不一定非要 5 个 Why，可能是 3-7 个
• 要问到「架构/设计/假设」层面，才是根本原因
• 避免停留在表面原因（如「超时时间不够」）
```

**产出模板**:
```markdown
### 4.1 关键问题根因分析（5 个 Why）

**问题**: [问题描述]

Why 1: 为什么 [问题]？
→ [答案 1]

Why 2: 为什么 [答案 1]？
→ [答案 2]

Why 3: 为什么 [答案 2]？
→ [答案 3]

Why 4: 为什么 [答案 3]？
→ [答案 4]

Why 5: 为什么 [答案 4]？
→ [根本原因]

**根本原因**:
[总结根本原因]

**解决方案**:
[基于根本原因的解决方案]
```

#### 3.2 问题分类

**执行指令**:
```
对识别的问题进行分类:

分类维度 1: 内因 vs 外因
• 内因: 自己可以控制的
• 外因: 外部环境/他人因素

分类维度 2: 可控 vs 不可控
• 可控: 可以改变的
• 不可控: 无法改变的

分类维度 3: 系统性 vs 偶然
• 系统性: 会重复发生的
• 偶然: 偶然发生的

处理策略:
• 内因 + 可控 + 系统性 → 重点解决
• 外因 + 不可控 → 接受或规避
• 偶然 → 监控，不投入过多资源
```

**产出模板**:
```markdown
### 4.2 问题分类分析

| 问题 | 内因/外因 | 可控/不可控 | 系统性/偶然 | 处理策略 |
|-----|----------|-----------|------------|---------|
| [问题1] | 内因 | 可控 | 系统性 | 重点解决 |
| [问题2] | 外因 | 不可控 | - | 接受/规避 |
| [问题3] | 内因 | 可控 | 偶然 | 监控 |

**分类总结**:
[对不同类别问题的处理策略]
```

#### 3.3 思维模式分析（如适用）

**执行指令**:
```
分析思维模式差异:

场景: AI vs 用户的思维差异

维度 1: 问题定义
• AI: [如何定义问题]
• 用户: [如何定义问题]
• 差异: [关键差异]

维度 2: 解决层次
• AI: [在哪个层次解决]
• 用户: [在哪个层次解决]
• 差异: [关键差异]

维度 3: 思考深度
• AI: [思考特点]
• 用户: [思考特点]
• 差异: [关键差异]

学习启示:
• AI 应该学习什么？
• 如何提升 AI 的思维能力？
```

**产出模板**:
```markdown
### 4.3 思维模式差异分析

| 维度 | AI 思维 | 用户思维 | 差异影响 |
|-----|--------|---------|---------|
| 问题定义 | [AI] | [用户] | [影响] |
| 解决层次 | [AI] | [用户] | [影响] |
| 思考深度 | [AI] | [用户] | [影响] |

**学习启示**:
• [启示 1]
• [启示 2]
```

#### 3.4 检查清单

- [ ] 至少 1 个关键问题执行了 5 个 Why
- [ ] 找到了根本原因（不是表面原因）
- [ ] 问题已分类（内因/外因、可控/不可控）
- [ ] 思维模式分析已完成（如适用）

### Step 4: 提炼经验（Stage 4: Extract Learnings）（20-30 分钟）

#### 4.1 提炼可复用模式

**执行指令**:
```
从本次经验中提炼模式:

1. 可复用的解决方案（pattern）:
   • 什么问题？
   • 怎么解决的？
   • 为什么有效？
   • 适用场景？

2. 最佳实践（preference）:
   • 什么场景？
   • 推荐做法？
   • 为什么推荐？

3. 执行手册（playbook）:
   • 什么任务？
   • 具体步骤？
   • 检查清单？

评估是否值得沉淀:
• 会在 ≥2 个场景中出现？
• 抽象出清晰的模式？
• 用户明确要求沉淀？
• 对团队有长期价值？
```

**产出模板**:
```markdown
### 5.1 可复用的模式/方法论

| 模式/方法论 | 类型 | 适用场景 | 核心要点 | 沉淀价值 |
|------------|------|---------|---------|---------|
| [模式1] | pattern | [场景] | [要点] | 高/中/低 |
| [模式2] | preference | [场景] | [要点] | 高/中/低 |
| [模式3] | playbook | [场景] | [要点] | 高/中/低 |

**模式总结**:
[对提炼的模式的整体总结]
```

#### 4.2 识别反模式

**执行指令**:
```
识别应该避免的做法:

1. 常见的坑:
   • 什么做法？
   • 有什么危害？
   • 如何避免？

2. 误区:
   • 什么误区？
   • 为什么是错误的？
   • 正确的做法是什么？
```

**产出模板**:
```markdown
### 5.2 避免踩坑的反模式

| 反模式 | 表现 | 危害 | 避免方法 |
|-------|------|------|---------|
| [反模式1] | [表现] | [危害] | [方法] |
| [反模式2] | [表现] | [危害] | [方法] |
```

#### 4.3 提出改进建议

**执行指令**:
```
提出流程/机制改进:

1. 当前流程的问题:
   • 什么问题？
   • 影响是什么？

2. 改进建议:
   • 建议什么？
   • 预期效果？
   • 实施成本？

3. 优先级:
   • P0: 必须立即改进
   • P1: 下个迭代改进
   • P2: 长期逐步改进
```

**产出模板**:
```markdown
### 5.3 流程/机制改进建议

| 改进项 | 当前问题 | 建议改进 | 预期效果 | 优先级 |
|-------|---------|---------|---------|-------|
| [改进1] | [问题] | [建议] | [效果] | P0/P1/P2 |
| [改进2] | [问题] | [建议] | [效果] | P0/P1/P2 |
```

#### 4.4 评估 Memory 沉淀

**执行指令**:
```
评估哪些经验值得沉淀为 memory:

对于每个提炼的经验:
1. 类型选择:
   • pattern（可复用模式）
   • anti-pattern（反模式）
   • preference（偏好）
   • playbook（执行手册）
   • reflection（深度反思）

2. 价值评估:
   • 复用频率: 高/中/低
   • 适用范围: 广/中/窄
   • 长期价值: 高/中/低

3. 沉淀决策:
   • 立即沉淀（价值高）
   • 观察后再沉淀（价值待验证）
   • 不沉淀（价值低或一次性）
```

**产出模板**:
```markdown
### 5.4 Memory 沉淀候选清单

| 经验 | 类型 | 适用场景 | 沉淀价值 | 决策 |
|-----|------|---------|---------|------|
| [经验1] | pattern | [场景] | 高 | 立即沉淀 |
| [经验2] | anti-pattern | [场景] | 高 | 立即沉淀 |
| [经验3] | preference | [场景] | 中 | 观察后沉淀 |
```

#### 4.5 检查清单

- [ ] 至少提炼 2-3 个可复用模式
- [ ] 至少识别 1-2 个反模式
- [ ] 改进建议已提出（含优先级）
- [ ] Memory 沉淀已评估

### Step 5: 行动计划（Stage 5: Action Plan）（15-20 分钟）

#### 5.1 制定短期行动（1-3 天）

**执行指令**:
```
制定立即执行的行动:

筛选标准:
• 可以快速完成（1-3 天）
• 不需要大量资源
• 效果明显

对于每个行动:
• 具体任务: [做什么]
• 责任人: [谁来做]
• 完成标准: [怎么算完成]
• 时间节点: [什么时候完成]
```

**产出模板**:
```markdown
### 6.1 短期行动（立即执行，1-3 天）

| 行动 | 具体任务 | 责任人 | 完成标准 | 时间节点 |
|-----|---------|-------|---------|---------|
| [行动1] | [任务] | [责任人] | [标准] | YYYY-MM-DD |
| [行动2] | [任务] | [责任人] | [标准] | YYYY-MM-DD |
```

#### 5.2 制定中期改进（1-2 周）

**执行指令**:
```
制定下个迭代的改进:

筛选标准:
• 需要一定投入（1-2 周）
• 有明确验收标准
```

执行完全部步骤后，按以下清单自检：

### 示例：5 个 Why 分析

```markdown
### 4.1 关键问题根因分析（5 个 Why）

**问题**: 为什么自检脚本出现超时？

Why 1: 为什么超时？
→ 执行耗时 7~10 分钟，超过了脚本 120s 超时设置

Why 2: 为什么执行这么久？
→ 一次性执行完整的 check-langgraph-backend change-id，包含 8 个任务

Why 3: 为什么必须一次性执行？
→ /run API 设计假设：单次调用完成完整 workflow

Why 4: 为什么这样设计？
→ 早期需求简单，没有考虑长时间运行的场景

Why 5: 能否改变设计？
→ 可以，引入阶段化执行：/run(change_id, phase="env-check")

**根本原因**: 架构设计假设（一次性全量执行）与实际场景（长时间运行）不匹配

**解决方案**: 重构为阶段化执行架构，将大事务拆成可观测的小步骤
```
