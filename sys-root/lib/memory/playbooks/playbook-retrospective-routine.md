# 复盘机制常态化执行手册

## 一句话定义

**将复盘分析融入日常工作的三种触发机制（定期、事件、问题）和完整执行流程，确保持续学习和改进。**

---

## 复盘触发机制（三种类型）

### 类型一：定期复盘（Periodic Retrospective）

**触发周期**:
- **轻量回顾**: 每周一次（周五下午，30分钟）
- **详细复盘**: 每完成一个 change-id（里程碑后，2-3小时）
- **深度复盘**: 每季度一次（季度末，半天）

**触发条件**:
```
时间条件满足
    ↓
检查本周/本迭代是否有值得复盘的内容
    ↓
有 → 执行定期复盘
无 → 记录"本次跳过"，下次继续检查
```

**轻量回顾检查清单（每周）**:
- [ ] 本周完成了哪些 change-id？
- [ ] 本周遇到了哪些执行偏差？
- [ ] 本周是否有用户追问/纠正？
- [ ] 本周是否发现了新的模式/反模式？
- [ ] 下周需要特别注意什么？

**详细复盘触发条件（每 change-id）**:
- [ ] change-id 已完成归档
- [ ] 该 change-id 涉及多个阶段（建议复盘）
- [ ] 该 change-id 发现了新问题（必须复盘）

---

### 类型二：事件触发复盘（Event-Driven Retrospective）

**触发事件**:
1. **重要里程碑完成**: change-id 完成归档后自动触发
2. **重大规范升级**: 如本次 8+1 闭环 v1.2 升级
3. **工具/流程发布**: 如 validate 工具 v2.0 发布
4. **团队扩张/变化**: 新成员加入、角色调整

**触发流程**:
```
事件完成
    ↓
主 Agent 自动建议："是否进行复盘分析？"
    ↓
用户确认：
    ├── "复盘" → 执行复盘分析
    └── "跳过" → 记录跳过理由
```

**事件触发复盘检查清单**:
- [ ] 事件目标是否达成？
- [ ] 执行过程中遇到了哪些问题？
- [ ] 有哪些值得沉淀的经验？
- [ ] 有哪些需要改进的地方？
- [ ] 是否需要升级文档/规范？

---

### 类型三：问题驱动复盘（Problem-Driven Retrospective）

**触发条件**（满足任一）:
1. **问题反复出现**: 同类问题 2+ 次出现
2. **用户多次追问**: 同一执行偏差被用户追问 2+ 次
3. **质量门禁未通过**: 关键阶段质量门禁检查失败
4. **规范执行偏差**: 发现重大术语定义漂移或惯性思维陷阱

**触发流程**:
```
发现问题
    ↓
问题严重程度评估
    ↓
高 → 立即启动复盘分析
中 → 记录问题，本周复盘时分析
低 → 记录问题，定期复盘时分析
```

**问题严重程度评估**:

| 严重程度 | 标准 | 复盘时机 |
|---------|------|---------|
| **高** | 影响交付质量、阻塞流程、用户强烈反馈 | 立即（2小时内） |
| **中** | 影响效率、有改进空间、值得学习 | 本周内 |
| **低** | 轻微偏差、偶发问题、影响有限 | 定期复盘时 |

## 复盘执行流程

### 阶段一：复盘准备（10-15分钟）

**1.1 确定复盘类型和范围**

| 复盘类型 | 确定方法 | 产出 |
|---------|---------|------|
| 定期复盘 | 检查日历/周期 | 复盘范围：本周/本迭代 |
| 事件触发 | 识别触发事件 | 复盘范围：事件全过程 |
| 问题驱动 | 识别问题根因 | 复盘范围：问题相关流程 |

**1.2 收集复盘材料**

必须收集:
- [ ] 迭代日志（`docs/项目事件日志.md`）
- [ ] 相关变更目录（`openspec/changes/[change-id]/`）
- [ ] 评审纪要（`docs/project-prd-changes/[change-id]/records/*-评审纪要.md`）
- [ ] 验收记录（`docs/project-prd-changes/[change-id]/records/*-func-test.md`）

可选收集:
- [ ] 代码变更（Git diff）
- [ ] 聊天记录/决策过程
- [ ] 用户反馈
- [ ] 性能指标/数据

**1.3 唤醒复盘相关 Memory**

强制唤醒:
- `skills/retrospective-analysis/SKILL.md`
- `pattern-five-stage-retrospective`
- `pattern-breakthrough-thinking-redefine-problem-space`

建议唤醒:
- `anti-pattern-terminology-drift`（如发现术语问题）
- `anti-pattern-inertia-trap`（如发现惯性思维）

**1.4 确定复盘目标**

```markdown
复盘目标模板:

复盘类型: [定期/事件触发/问题驱动]
复盘范围: [时间范围/change-id/事件]
复盘目标:
  1. [目标1: 如找出问题根因]
  2. [目标2: 如沉淀改进模式]
  3. [目标3: 如优化流程机制]

预期产出:
  - 复盘报告: `docs/retrospective/[框架/项目]/[时间]/复盘-[主题]-[日期].md`
  - Memory 沉淀: [pattern/anti-pattern/preference/playbook/reflection]
  - 改进行动: [行动计划清单]
```

---

### 阶段二：执行复盘分析（核心，按 retrospective-analysis skill）

**严格遵循 5 阶段复盘法**:

#### 阶段 1: 回顾目标（Review Goals）

- 当初的目标是什么？
- 预期的结果是什么？
- 关键成功指标有哪些？
- 实际完成了哪些？

**产出**: 目标清单、成功指标清单

#### 阶段 2: 评估结果（Evaluate Results）

- 实际结果 vs 预期目标（对比分析）
- 做得好的地方（亮点清单）
- 做得不好的地方（问题清单）
- 意外发现（好的/坏的）

**产出**: 结果对比表、亮点清单、问题清单

#### 阶段 3: 分析原因（Analyze Causes）

- 5 个 Why 找到根本原因
- 区分内因/外因、可控/不可控因素
- 识别系统性问题 vs 偶然问题
- 思维模式差异分析（AI vs 用户）

**产出**: 根因分析表、问题分类

#### 阶段 4: 提炼经验（Extract Learnings）

- 可复用的模式/方法论
- 避免踩坑的反模式
- 流程/机制改进建议
- 沉淀为 memory 的候选

**产出**: 模式清单、反模式清单、改进建议、Memory 候选

#### 阶段 5: 行动计划（Action Plan）

- 短期行动（立即执行，1-3天）
- 中期改进（下个迭代，1-2周）
- 长期演进（架构/模式级，1-3月）

**产出**: 行动计划表（含优先级、责任人、时间节点）

---

### 阶段三：产出复盘报告（按模板）

**复盘报告必须包含**（8 章节）:
1. 复盘背景与目标
2. 回顾目标
3. 评估结果
4. 分析原因
5. 提炼经验
6. 行动计划
7. 关键洞察与总结
8. 附录

**报告存放路径**:
- 框架级复盘: `docs/retrospective/framework/[YYYY-QN]/复盘-[主题]-YYYY-MM-DD.md`
- 项目级复盘: `[项目]/docs/retrospective/[YYYY-MM]/复盘-[主题]-YYYY-MM-DD.md`
- change-id 级复盘: `docs/project-prd-changes/[change-id]/records/复盘报告-YYYY-MM-DD-[主题].md`

---

### 阶段四：Memory 沉淀（如需要）

**评估是否值得沉淀**:
- [ ] 在 ≥2 个不同场景中出现过
- [ ] 抽象出清晰的模式/方法论
- [ ] 用户明确要求沉淀
- [ ] 对团队有长期价值

**创建 Memory 文档**:
- **pattern**: 可复用的解决方案 → `memory/patterns/pattern-xxx.md`
- **anti-pattern**: 应该避免的坑 → `memory/anti-patterns/anti-pattern-xxx.md`
- **preference**: 偏好/最佳实践 → `memory/preferences/preference-xxx.md`
- **playbook**: 执行手册 → `memory/playbooks/playbook-xxx.md`
- **reflection**: 深度反思 → `memory/reflections/reflection-xxx.md`

---

### 阶段五：更新迭代日志与通知

**更新迭代日志**:
```markdown
- YYYY-MM-DD | [change-id] | retrospective-analysis | 
  复盘完成，复盘报告路径：[路径]，
  沉淀 Memory：[memory1, memory2, ...]，
  改进行动：[行动1, 行动2, ...]
```

**通知相关方**（如复盘涉及重要改进）:
- 架构 Agent（如涉及架构改进）
- 产品经理 Agent（如涉及需求流程改进）
- 主 Agent（如涉及整体流程改进）

## 复盘质量自检（9 项）

| # | 检查项 | 说明 | 判定 |
|---|-------|------|------|
| 1 | 目标回顾完整 | 清晰回顾了初始目标和预期结果 | ✓/△/✗ |
| 2 | 结果对比清晰 | 对比了实际结果 vs 预期目标 | ✓/△/✗ |
| 3 | 问题识别全面 | 识别了所有关键问题 | ✓/△/✗ |
| 4 | 根因分析深入 | 用 5 个 Why 找到根本原因 | ✓/△/✗ |
| 5 | 思维模式分析 | 分析了 AI vs 用户的思维差异 | ✓/△/✗ |
| 6 | 经验提炼可复用 | 提炼的经验是否抽象、可复用 | ✓/△/✗ |
| 7 | 行动计划可执行 | 行动计划是否具体、有责任人、有时间 | ✓/△/✗ |
| 8 | memory 沉淀价值 | 沉淀的 memory 是否有长期价值 | ✓/△/✗ |
| 9 | 文档规范符合 | 是否符合复盘报告模板要求 | ✓/△/✗ |

**评审判定**:
- **通过**: 7 项及以上通过，无不通过项
- **有条件通过**: 5-6 项通过，无不通过项（需补充优化）
- **不通过**: 少于 5 项通过，或有不通过项（需重写）

## 与 8+1 质量闭环的关联

```
8+1 闭环中的 Step 9: 复盘与持续改进
    ↓
本手册定义的三种复盘触发机制
    ├── 定期复盘 → 每周/每迭代/每季度
    ├── 事件触发复盘 → 里程碑/规范升级
    └── 问题驱动复盘 → 问题反复出现
    ↓
使用 retrospective-analysis skill 执行
    ↓
产出复盘报告 + Memory 沉淀
    ↓
纳入下一次 8+1 闭环执行（持续改进）
```

## 执行检查清单

**启动复盘前**:
- [ ] 确定复盘类型（定期/事件/问题）
- [ ] 确定复盘范围
- [ ] 收集复盘材料
- [ ] 唤醒复盘相关 Memory

**执行复盘时**:
- [ ] 遵循 5 阶段复盘法
- [ ] 使用复盘报告模板
- [ ] 逐项执行质量自检（9 项）

**复盘完成后**:
- [ ] 产出复盘报告
- [ ] 沉淀 Memory（如需要）
- [ ] 更新迭代日志
- [ ] 通知相关方（如需要）
- [ ] 安排改进行动（按行动计划）
