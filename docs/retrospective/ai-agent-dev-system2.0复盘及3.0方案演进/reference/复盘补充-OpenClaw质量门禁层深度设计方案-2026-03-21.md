## 执行摘要

### 核心结论

基于 OpenClaw 架构研究，设计「质量门禁层（Quality Gate Layer）」可实现：

| 关键门禁 | 预期达成率 | 实现机制 | 严格度 |
|---------|-----------|---------|--------|
| **PRD质量门禁** | 80-85% | Skill Gating + Post-check | Blocking |
| **方案质量门禁** | 75-80% | Post-check + Template匹配 | Warning |
| **代码质量门禁** | 70-75% | exec工具运行测试/检查 | Blocking |
| **交付质量门禁** | 85-90% | Checklist验证 + 人工确认 | Blocking |
| **综合达成率** | **77.5-82.5%** | 4门禁加权平均 | Mixed |

**结论**: ✅ 通过精心设计的质量门禁层，可以达到**75-80%综合达成率**的目标

### 1.2 质量门禁层设计原则

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **关键少数** | 只关注4个关键门禁，非关键步骤灵活 | 4门禁设计，其他步骤不阻断 |
| **产出物中心** | 验证产出物质量，而非步骤执行 | Post-check验证模板符合度 |
| **可配置严格度** | 不同门禁可配置Blocking/Warning | metadata.gate.severity |
| **渐进增强** | 从Warning开始，逐步增强到Blocking | 配置化升级路径 |
| **失败可重试** | 门禁不通过可修改后重试 | 不销毁Session，允许重试 |

## 第二部分：4个关键质量门禁详细设计

### 2.1 门禁1：PRD质量门禁（PRD Quality Gate）

#### 门禁定位
- **触发时机**: 用户要求创建PRD或进入Step 2（PRD评审）前
- **严格度**: **Blocking**（不通过则无法进入方案设计）
- **预期达成率**: 80-85%

#### 验证检查清单

| 检查项 | 验证方法 | 通过标准 | 权重 |
|--------|---------|---------|------|
| **结构完整性** | Template匹配 | 包含8类结构（背景/需求/验收标准/功能点/非功能需求/技术约束/时间计划/资源预算） | 30% |
| **验收标准明确** | 关键词检测 | 包含"验收标准"、"可测试"、"可验证"等关键词 | 25% |
| **功能点可追踪** | 编号检测 | 功能点有明确编号（如F1, F2...） | 20% |
| **非空检查** | 字段长度 | 所有必填字段非空，长度>10字符 | 15% |
| **格式规范** | Markdown语法 | 正确Markdown格式，有标题层级 | 10% |

#### 通过阈值
```
总得分 ≥ 80分 → 通过（进入方案设计）
总得分 60-79分 → 警告（建议完善，但可人工确认继续）
总得分 < 60分 → 阻断（必须修改后重试）
```

#### 失败处理流程
```
阻断时行为:
1. 向用户展示详细检查报告（哪项未通过）
2. 提供修改建议（使用edit工具自动修复或人工修改）
3. 允许用户选择 "修改后重试" 或 "人工确认强制通过"
4. 记录门禁尝试历史到 Session JSONL
```

#### OpenClaw SKILL.md 配置示例
```yaml
---
metadata:
  openclaw:
    name: "PRD Quality Gate"
    description: "验证PRD文档质量是否符合规范"
    emoji: "📋"
    
    # 质量门禁配置
    gate:
      id: "prd-quality"
      trigger: "before:prd-review"
      severity: "blocking"  # blocking | warning | info
      required_score: 80    # 通过阈值
      
    # 检查项配置
    checks:
      - id: "structure"
        name: "结构完整性"
        weight: 30

      - id: "acceptance-criteria"
        name: "验收标准明确"
        weight: 25

      - id: "non-empty"

# PRD Quality Gate Skill

## Instructions

当用户准备进入PRD评审或创建PRD文档时，执行以下质量门禁检查:

1. **读取检查配置**
   - 从 metadata.gate.checks 获取检查项列表
   - 计算总分公式: sum(check_score * weight) / sum(weights) * 100

2. **执行结构完整性检查**
   - 使用 `read` 工具读取待检查的PRD文件
   - 对比 `template` 指定的模板结构
   - 检查8类结构是否存在（背景/需求/验收标准/功能点/非功能需求/技术约束/时间计划/资源预算）
   - 每项存在得满分，不存在得0分

3. **执行验收标准明确检查**
   - 搜索关键词: "验收标准", "可测试", "可验证", "checklist"
   - 每找到一个关键词得部分分
   - 关键词越多，得分越高

4. **执行功能点可追踪检查**
   - 使用正则表达式匹配功能点编号（如 F1, F2...）
   - 至少找到3个功能点得满分
   - 少于3个按比例扣分

5. **执行非空检查**
   - 检查必填字段长度
   - 长度≥10字符得满分

6. **执行格式规范检查**
   - 验证Markdown语法正确性
   - 检查标题层级（至少2级）

7. **计算总得分并决策**
   - 总分 ≥ required_score (80): 通过，继续执行
   - 总分 < required_score: 阻断，提示用户

8. **生成检查报告**
   ```markdown
   ## PRD质量门禁检查报告
   
   | 检查项 | 得分 | 权重 | 加权得分 | 状态 |
   |--------|------|------|---------|------|
   | 结构完整性 | 30/30 | 30% | 9.0 | ✅ |
   | 验收标准明确 | 20/25 | 25% | 5.0 | ⚠️ |
   | ... | ... | ... | ... | ... |
   
   **总得分**: 75/100
   **状态**: 警告（建议完善）
   
   **未通过项详情**:
   - 验收标准明确: 未找到"可测试"关键词
   
   **建议操作**:
   1. 使用 edit 工具补充验收标准
   2. 或选择 "人工确认强制通过"
   ```

## Tools

- `read`: 读取PRD文件
- `edit`: 自动修复（如果配置了auto_fix）
- `prompt_user`: 提示用户决策

## Post-check

检查通过后，必须验证:
1. PRD文件已保存到正确路径
2. 文件命名符合规范: `PRD-{change-id}-{description}.md`
3. 文件已添加到设计文档目录
```

### 2.2 门禁2：方案质量门禁（Technical Solution Quality Gate）

#### 门禁定位
- **触发时机**: 用户完成技术方案设计后，进入编码前
- **严格度**: **Warning**（不通过提示警告，但可继续）
- **预期达成率**: 75-80%

#### 验证检查清单

| 检查项 | 验证方法 | 通过标准 | 权重 |
|--------|---------|---------|------|
| **与PRD对应性** | 功能点映射 | 每个PRD功能点(F1, F2...)都有对应技术实现 | 35% |
| **接口定义清晰** | 字段检测 | 包含接口定义、输入/输出字段、数据类型 | 25% |
| **技术选型合理** | 关键词检测 | 技术栈与项目约束匹配（如用Python不用Java） | 20% |
| **风险识别** | 关键词检测 | 包含"风险"、"回滚方案"、"监控方案" | 15% |
| **可实施性** | 长度/完整性 | 方案详细程度足够开发团队执行 | 5% |

#### 通过阈值
```
总得分 ≥ 75分 → 通过
总得分 50-74分 → 警告（可继续，但建议完善）
总得分 < 50分 → 强警告（强烈建议完善）
```

#### 特殊设计：Warning模式
```
Warning模式行为:
1. 检查不通过时，向用户展示警告报告
2. 但不阻断流程，用户可选择 "继续" 或 "完善方案"
3. 记录警告到 Session 历史（供后续复盘）
4. 如果用户选择继续，在最终交付报告中标记 "方案质量警告"
```

#### OpenClaw SKILL.md 配置示例
```yaml
---
metadata:
  openclaw:
    name: "Technical Solution Quality Gate"
    description: "验证技术方案与PRD的对应性"
    emoji: "🔧"
    
    gate:
      id: "solution-quality"
      trigger: "before:implementation"
      severity: "warning"  # 注意: warning模式
      required_score: 75
      
    checks:
      - id: "prd-mapping"
        name: "PRD对应性"
        weight: 35
        method: "cross_reference"  # 交叉引用PRD功能点
      - id: "api-definition"
        name: "接口定义清晰"
        weight: 25
        keywords: ["接口", "输入", "输出", "字段", "类型"]
      - id: "tech-stack"
        name: "技术选型合理"
        weight: 20
        allowed_stack: ["python", "typescript", "javascript"]
      - id: "risk-awareness"
        name: "风险识别"
        weight: 15
        keywords: ["风险", "回滚", "监控", "降级"]
      - id: "implementability"
        name: "可实施性"
        weight: 5
        min_detail_level: "medium"
    
    on_failure:
      action: "warn_and_continue"  # 警告但允许继续
      show_report: true
      require_acknowledgment: true  # 需要用户确认
---

# Technical Solution Quality Gate Skill

## Instructions

执行技术方案质量门禁检查:

1. 读取当前技术方案文档
2. 读取对应的PRD文档（通过change-id关联）
3. 执行交叉引用检查:
   - 提取PRD中的所有功能点编号（F1, F2...）
   - 在技术方案中搜索每个功能点的对应实现描述
   - 计算对应覆盖率: 找到对应的功能点数 / 总功能点数
4. 执行其他检查项...
5. Warning模式: 不通过时提示但不阻断
```

### 2.3 门禁3：代码质量门禁（Code Quality Gate）

#### 门禁定位
- **触发时机**: 编码完成后，代码评审前
- **严格度**: **Blocking**（不通过则无法进入验收）
- **预期达成率**: 70-75%

#### 验证检查清单

| 检查项 | 验证方法 | 通过标准 | 权重 |
|--------|---------|---------|------|
| **功能实现完整** | 测试运行 | 所有PRD功能点有对应测试且通过 | 40% |
| **无明显安全漏洞** | 静态扫描 | 通过安全扫描（无SQL注入/XSS等高危漏洞） | 25% |
| **代码规范** | Linter检查 | 通过项目配置的Linter检查 | 15% |
| **单元测试覆盖** | 覆盖率检测 | 核心功能单元测试覆盖率≥60% | 15% |
| **性能基准** | 基准测试 | 关键路径性能不劣化（如果配置了基准） | 5% |

#### 通过阈值
```
总得分 ≥ 80分 → 通过（进入验收）
总得分 60-79分 → 有条件通过（需人工确认）
总得分 < 60分 → 阻断（必须修复）
```

#### 自动化实现重点
```
代码质量门禁高度依赖自动化工具:

1. **功能实现检查**:
   - 使用 exec 运行测试套件
   - 检查测试通过率
   - 验证所有PRD功能点有对应测试

2. **安全漏洞检查**:
   - 集成 bandit (Python) 或 eslint-security (JS)
   - 扫描高危漏洞模式

3. **代码规范检查**:
   - 使用项目配置的 linter (flake8, pylint, eslint)
   - 检查格式、命名、复杂度

4. **单元测试覆盖**:
   - 使用 coverage.py 或 jest --coverage
   - 检查核心功能覆盖率
```

#### OpenClaw SKILL.md 配置示例
```yaml
---
metadata:
  openclaw:
    name: "Code Quality Gate"
    description: "自动化代码质量检查"
    emoji: "🧪"
    
    gate:
      id: "code-quality"
      trigger: "after:implementation"
      severity: "blocking"
      required_score: 80
      
    checks:
      - id: "test-pass"
        name: "功能实现完整"
        weight: 40
        command: "pytest --tb=short -q"
        expect_exit_code: 0
      - id: "security-scan"
        name: "安全漏洞检查"
        weight: 25
        command: "bandit -r . -f json"
        max_severity: "medium"  # 只允许medium及以下漏洞
      - id: "lint"
        name: "代码规范"
        weight: 15
        command: "flake8 . --max-line-length=100"
      - id: "coverage"
        name: "单元测试覆盖"
        weight: 15
        command: "pytest --cov=src --cov-report=term-missing"
        min_coverage: 60
      - id: "performance"
        name: "性能基准"
        weight: 5
        command: "pytest benchmarks/"
        optional: true  # 可选检查
    
    on_failure:
      action: "prompt_user"
      show_report: true
      show_logs: true  # 显示详细的测试/扫描日志
      allow_override: true
      
    requires:
      bins: ["pytest", "bandit", "flake8"]
      env: ["PYTHONPATH"]
---

# Code Quality Gate Skill

## Instructions

执行自动化代码质量检查:

1. 运行测试套件
   - 使用 `exec` 执行 `pytest --tb=short -q`
   - 检查 exit_code == 0
   - 收集测试通过数/失败数

2. 运行安全扫描
   - 使用 `exec` 执行 `bandit -r . -f json`
   - 解析JSON结果
   - 检查是否有 HIGH/CRITICAL 级别漏洞

3. 运行代码规范检查
   - 使用 `exec` 执行 `flake8`
   - 统计错误数/警告数

4. 运行覆盖率检查
   - 使用 `exec` 执行 `pytest --cov`
   - 解析覆盖率报告
   - 验证是否 ≥ min_coverage (60%)

5. 汇总报告
   ```markdown
   ## 代码质量门禁报告
   
   ### 自动化检查结果
   | 检查项 | 命令 | 结果 | 得分 | 权重 |
   |--------|------|------|------|------|
   | 功能测试 | pytest | ✅ 45/45通过 | 40/40 | 40% |
   | 安全扫描 | bandit | ⚠️ 2个low级别 | 20/25 | 25% |
   | 代码规范 | flake8 | ❌ 3个错误 | 10/15 | 15% |
   | 单元覆盖 | pytest --cov | 65%覆盖 | 15/15 | 15% |
   
   **总得分**: 85/100
   **状态**: 通过 ✅
   
   ### 详细日志
   [附详细测试日志...]
   ```

## Tools

- `exec`: 运行测试、扫描、检查命令
- `read`: 读取覆盖率报告
- `prompt_user`: 提示用户决策

## Post-check

验证:
1. 所有自动化检查命令已执行
2. 检查报告已保存到 records/ 目录
3. 测试覆盖率数据已记录
```

### 2.4 门禁4：交付质量门禁（Delivery Quality Gate）

#### 门禁定位
- **触发时机**: 验收测试完成后，归档前
- **严格度**: **Blocking**（不通过则无法归档）
- **预期达成率**: 85-90%

#### 验证检查清单

| 检查项 | 验证方法 | 通过标准 | 权重 |
|--------|---------|---------|------|
| **验收Checklist通过** | 逐项验证 | 所有验收项标记为通过 | 40% |
| **可部署/可运行** | 部署验证 | 能成功部署并运行核心功能 | 30% |
| **文档完整** | 文件检查 | 必需文档齐全（README/部署文档/使用说明） | 15% |
| **变更记录完整** | 记录检查 | 迭代日志、复盘文档已记录 | 10% |
| **人工确认** | 用户确认 | 用户显式确认 "验收通过" | 5% |

#### 通过阈值
```
总得分 ≥ 85分 → 通过（进入归档）
总得分 70-84分 → 有条件通过（需人工确认）
总得分 < 70分 → 阻断（必须完善）
```

#### 特殊设计：人工确认环节
```
交付门禁包含显式的人工确认环节:

1. 自动化检查通过后，提示用户:
   "所有自动化检查已通过，请进行最终人工确认:
   - [ ] 我已验证核心功能正常运行
   - [ ] 我已阅读验收报告
   - [ ] 我确认本次变更可以归档
   
   请输入 '确认通过' 或 '需要修改'"

2. 记录人工确认结果到:
   - Session 历史
   - 验收纪要文档
   - 迭代日志

3. 如果用户选择 "需要修改"，返回修改流程
```

#### Skill Gating 决策流程

```python
# 伪代码: Skill Gating 决策逻辑
def should_load_skill(skill_metadata, context):
    requires = skill_metadata.get('requires', {})
    
    # 1. 检查文件存在性
    for file_req in requires.get('files', []):
        if not file_exists(file_req['path']):
            if gating_behavior['on_missing_file'] == 'prompt':
                return {
                    'load': False,
                    'reason': f"缺少必需文件: {file_req['path']}",
                    'action': 'prompt_user_to_provide'
                }
            elif gating_behavior['on_missing_file'] == 'error':
                raise GatingError(f"缺少必需文件: {file_req['path']}")
            # 'skip' 则忽略此检查继续
    
    # 2. 检查环境变量
    for env_req in requires.get('env', []):
        if not os.environ.get(env_req['name']):
            if gating_behavior['on_missing_env'] == 'prompt':
                return {
                    'load': False,
                    'reason': f"缺少环境变量: {env_req['name']}",
                    'action': 'prompt_user_to_set'
                }
    
    # 3. 检查前置门禁
    for gate_req in requires.get('gates_passed', []):
        if not check_gate_passed(gate_req['gate_id'], context['session']):
            if gating_behavior['on_missing_gate'] == 'block':
                return {
                    'load': False,
                    'reason': f"必须先通过门禁: {gate_req['gate_id']}",
                    'action': 'redirect_to_gate_skill'
                }
            elif gating_behavior['on_missing_gate'] == 'warn':
                # 记录警告但允许加载
                log_warning(f"门禁未通过但允许继续: {gate_req['gate_id']}")
    
    # 所有检查通过
    return {'load': True}
```

## 第六部分：预期达成率分析与验证计划

### 6.1 4个关键门禁达成率预测

| 门禁 | 权重 | 基础达成率 | 优化后达成率 | 置信度 |
|------|------|-----------|-------------|--------|
| **PRD质量门禁** | 25% | 70% | **80-85%** | 高（Template匹配成熟） |
| **方案质量门禁** | 25% | 65% | **75-80%** | 中（PRD对应性检查较复杂） |
| **代码质量门禁** | 25% | 60% | **70-75%** | 高（自动化工具成熟） |
| **交付质量门禁** | 25% | 80% | **85-90%** | 高（Checklist验证简单） |
| **加权平均** | 100% | 68.75% | **77.5-82.5%** | 中高 |

### 6.2 达成率计算公式

```
综合达成率 = Σ(门禁达成率 × 门禁权重)

示例计算:
= 82.5% × 25% (PRD)
+ 77.5% × 25% (方案)
+ 72.5% × 25% (代码)
+ 87.5% × 25% (交付)
= 20.625% + 19.375% + 18.125% + 21.875%
= 80%

结论: 预期综合达成率 77.5-82.5%，中位数 80%
```

### 6.3 风险与缓解措施

| 风险 | 影响 | 可能性 | 缓解措施 |
|------|------|--------|---------|
| **Template匹配不准确** | PRD门禁达成率下降 | 中 | 使用多模板+模糊匹配；允许人工覆盖 |
| **PRD-方案对应性检查复杂** | 方案门禁达成率下降 | 中 | 从简单关键词匹配开始；逐步增强 |
| **自动化测试不稳定** | 代码门禁达成率波动 | 中 | 增加重试机制；区分"测试失败"vs"门禁失败" |
| **用户不接受Warning模式** | 流程感受差 | 低 | 提供配置化严格度；从Warning开始逐步增强 |
| **集成复杂度超预期** | 实现延期 | 低 | 分Phase实现；先配置文件，再Skill，最后集成 |
