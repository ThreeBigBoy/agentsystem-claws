# OpenClaw 架构深度研究报告

> **文档类型**: 复盘研究补充  
> **研究日期**: 2026-03-21  
> **研究对象**: OpenClaw (https://github.com/openclaw/openclaw)  
> **GitHub Stars**: 327,489+ (高度活跃项目)  
> **关联复盘**: `复盘-二轮-突破性思维-商业化战略转型分析-2026-03-21.md`  
> **研究目的**: 回答三个核心问题：
> 1. OpenClaw 如何解决系统层、用户层、项目层的执行规范？
> 2. 多 Agent、多 Skill 协同机制如何实现？
> 3. 多 Agent 协同是否是伪命题？

---

## 执行摘要

### 核心发现

1. **OpenClaw 采用「单 Gateway + 多 Session」架构**，不是真正的多 Agent 架构，而是「单 Agent 运行时 + Session 隔离」

2. **Skill 系统高度模块化**：支持 Bundled/Managed/Workspace 三层加载，通过 `SKILL.md` + YAML frontmatter 定义

3. **Session 是核心抽象**：通过 `dmScope` 实现用户隔离，`per-channel-peer` 模式解决多用户隐私问题

4. **Agent-to-Agent 通过 Session 工具实现**：`sessions_list` / `sessions_history` / `sessions_send` / `sessions_spawn`，而非真正的多 Agent 并行

5. **关键洞察**：OpenClaw 证明「多 Agent 协同」确实是伪命题——单 Agent + 多 Skills + Session 隔离已足够

### 1.2 关键设计原则

| 原则 | 实现 | 说明 |
|------|------|------|
| **Single Gateway** | 单控制平面 | 一个主机只运行一个 Gateway，所有消息通道由它控制 |
| **Session 隔离** | 通过 sessionKey | 不同对话隔离状态，支持多用户场景 |
| **Tool 中心化** | 所有功能都是 Tool | Browser、Canvas、Nodes、Cron 等都是 Tool |
| **Skill 即文档** | `SKILL.md` 定义 | AgentSkills 兼容格式，YAML frontmatter + Markdown |
| **Local-first** | 本地运行优先 | Gateway 默认绑定 loopback，支持远程通过 Tailscale/SSH |

## 2. 系统层、用户层、项目层的执行规范

### 2.1 三层架构映射

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     OpenClaw 三层执行规范                                 │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  系统层（System Layer）- Gateway 控制                                    │
│  ├─ 配置: ~/.openclaw/openclaw.json                                      │
│  ├─ 安全: DM Policy (pairing/open), Sandbox mode                         │
│  ├─ 模型: Model refs (provider/model)                                    │
│  ├─ 通道: Channels config (WhatsApp, Telegram, Slack...)                 │
│  └─ 会话: Session defaults (dmScope, reset policy, maintenance)         │
│                                                                          │
│  用户层（User Layer）- Agent 会话控制                                    │
│  ├─ 会话隔离: session.dmScope (main/per-peer/per-channel-peer)           │
│  ├─ 身份链接: session.identityLinks (跨通道统一用户)                      │
│  ├─ 发送策略: session.sendPolicy (allow/deny by channel)               │
│  ├─ 重置策略: session.reset (daily/idle/byType/byChannel)               │
│  └─ 维护策略: session.maintenance (pruneAfter, maxEntries, enforce)      │
│                                                                          │
│  项目层（Project Layer）- Workspace 控制                                   │
│  ├─ 工作空间: agents.defaults.workspace (~/.openclaw/workspace)          │
│  ├─ 引导文件: AGENTS.md, SOUL.md, TOOLS.md, BOOTSTRAP.md...              │
│  ├─ 技能: /skills (Workspace level)                                      │
│  ├─ 会话存储: ~/.openclaw/agents/{agentId}/sessions/                    │
│  └─ 沙盒: agents.defaults.sandbox (Docker mode for non-main sessions)    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 系统层执行规范详解

#### Gateway 配置（~/.openclaw/openclaw.json）

```json5
{
  // 系统层: Agent 全局默认
  agent: {
    model: "anthropic/claude-opus-4-6",
    workspace: "~/.openclaw/workspace",
    sandbox: {
      mode: "non-main",  // 非主会话使用 Docker 沙盒
    },
  },
  
  // 系统层: 会话默认配置
  session: {
    dmScope: "per-channel-peer",  // 安全 DM 模式
    reset: { mode: "daily", atHour: 4, idleMinutes: 120 },
    maintenance: {
      mode: "enforce",
      pruneAfter: "30d",
      maxEntries: 500,
    },
  },
  
  // 系统层: 通道配置
  channels: {
    whatsapp: { allowFrom: ["sender1", "sender2"] },
    telegram: { botToken: "...", groups: [...] },
    slack: { botToken: "...", appToken: "..." },
  },
  
  // 系统层: 安全策略
  agents: {
    defaults: {
      sandbox: {
        mode: "non-main",
        allowlist: ["bash", "read", "write", "edit"],
        denylist: ["browser", "canvas", "nodes"],
      },
    },
  },
}
```

### 2.3 用户层执行规范详解

#### Session 隔离策略（解决多用户隐私问题）

```
问题场景:
- Alice (telegram:123) 私信 Agent 询问医疗预约
- Bob (telegram:456) 私信 Agent 问"我们在聊什么？"
- 默认共享同一个 main session → Bob 可能看到 Alice 的隐私信息

解决方案: dmScope

模式 1: main (默认，单用户)
├── 所有 DM 共享 main session
└── 适合: 单人使用

模式 2: per-peer (按发送者隔离)
├── Session key: agent::<agentId>::direct:<senderId>
└── 适合: 多用户，但同一人在不同通道还是分开

模式 3: per-channel-peer (推荐，多用户安全模式) ⭐
├── Session key: agent::<agentId>::<channel>:<senderId>:direct:<senderId>
├── 不同通道完全隔离
└── 适合: 多用户 inbox，最安全

模式 4: per-account-channel-peer (多账号 inbox)
├── Session key: agent::<agentId>::<accountId>:<channel>:<senderId>:direct:<senderId>
└── 适合: 同一通道多账号

跨通道统一用户: identityLinks
{
  session: {
    identityLinks: {
      alice: ["telegram:123456789", "discord:987654321"],
    },
  },
}
→ Alice 在 Telegram 和 Discord 的消息归于同一个会话
```

│       ├── profile.json

    ├── images/
    ├── documents/

```

**目录说明**:
- `user.json`: 用户核心配置，管理跨通道身份链接、默认隔离策略
- `identities/`: 多用户场景下的身份管理，每个子目录对应一个用户
- `sessions/`: 用户级别的会话存储，按活跃/归档分类
  - `index.json`: 快速查找会话元数据
  - `active/`: 当前活跃的会话，按通道隔离
  - `archived/`: 已归档的历史会话，按月分组
- `memory/`: 用户级别的长期记忆，Agent可以跨会话访问
- `uploads/`: 用户上传的文件缓存

**与系统层的关系**:
```
系统层（~/.openclaw/）: 全局配置、核心运行时、系统级日志
         ↓
用户层（~/.openclaw/user/）: 用户偏好、身份管理、个人会话历史
         ↓
项目层（~/.openclaw/workspace/）: 项目特定配置、工作区Skills、项目文档
```

**隔离策略**:
- 单用户场景: `~/.openclaw/user/` 只有一个用户配置
- 多用户场景: `~/.openclaw/user/identities/` 下有多个用户子目录
- 通过 `dmScope` 配置决定会话隔离级别（main/per-peer/per-channel-peer）

### 2.4 项目层执行规范详解

#### Workspace 结构

```
~/.openclaw/workspace/
├── AGENTS.md          # 操作指令 + "记忆"
├── SOUL.md            # 人设、边界、语气
├── TOOLS.md           # 用户维护的工具笔记（项目工具链配置）
├── BOOTSTRAP.md       # 首次运行仪式（完成后删除）
├── IDENTITY.md        # Agent 名称/风格/emoji
├── USER.md            # 用户资料 + 首选地址
└── skills/            # Workspace 级技能
    ├── skill-a/
    │   └── SKILL.md
    └── skill-b/
        └── SKILL.md
```

**设计说明：为什么Agent/Skill配置放在项目层？**

这涉及OpenClaw的核心设计哲学——**"Context-as-Code"（上下文即代码）**：

```
传统理解 vs OpenClaw设计:

传统项目层:
├── 源代码（src/）
├── 项目文档（docs/）
├── 构建配置（package.json, Makefile）
└── 部署配置（Dockerfile, k8s/）
→ 这些都是"工程产物"

OpenClaw的Workspace层:
├── AGENTS.md          # Agent行为定义（项目的"操作手册"）
├── SOUL.md            # Agent人设（项目的"人格定义"）
├── skills/            # 项目特定技能（项目的"能力清单"）
└── ...
→ 这些都是"上下文配置"，是项目的"运行依赖"
```

**关键设计理念**:

1. **Agent人设是项目特定的**
   ```
   项目A（金融系统）: SOUL.md定义Agent为"严谨、保守、合规优先"
   项目B（创意工具）: SOUL.md定义Agent为"创新、灵活、用户优先"
   → 同一个人在不同项目中需要不同的Agent人设
   ```

2. **Skill需求是项目特定的**
   ```
   项目A: 需要金融分析Skill、合规检查Skill
   项目B: 需要图像生成Skill、创意写作Skill
   → Skills随项目需求变化，不是用户通用能力
   ```

3. **上下文应该版本控制**
   ```
   AGENTS.md + SOUL.md + Skills/
   ├── 应该随项目代码一起版本控制
   ├── 新成员clone项目即获得完整Agent配置
   └── CI/CD可以复用相同的Agent上下文
   ```

4. **三层架构的清晰分工**
   ```
   系统层: 提供"运行时能力"（Agent能做什么）
          ↓
   用户层: 提供"用户身份"（谁在用Agent）
          ↓
   项目层: 提供"执行上下文"（在这个项目中Agent该怎么表现）
   ```

**对比表**:

| 配置项 | 放在项目层的原因 | 如果放在用户层的问题 |
|--------|-----------------|-------------------|
| **AGENTS.md** | 不同项目有不同操作规范 | 一个用户多个项目时冲突 |
| **SOUL.md** | 不同项目需要不同Agent人设 | 金融项目和创意项目人设矛盾 |
| **Skills/** | 不同项目需要不同技能集 | 加载无关Skill浪费资源 |
| **TOOLS.md** | 不同项目使用不同工具链 | 工具版本冲突 |

**迁移建议**:
如果习惯传统项目结构，可以建立软链接或符号链接：
```bash
# 项目根目录
my-project/
├── src/                    # 源代码
├── docs/                   # 文档
├── openclaw-config/        # OpenClaw配置（可版本控制）
│   ├── AGENTS.md
│   ├── SOUL.md
│   └── skills/
└── .openclaw-workspace -> openclaw-config/  # 软链接到~/.openclaw/workspace/
```

#### Bootstrap 文件详解

OpenClaw使用一组Bootstrap文件来定义项目的Agent上下文。这些文件在`~/.openclaw/workspace/{project}/`下：

**1. AGENTS.md - 操作指令 + "记忆"**

```markdown
# AGENTS.md 示例（Curiobuddy）

## 角色定位
你是Curiobuddy，一个面向K12教育的AI学习助手。

## 核心职责
1. 帮助学生理解复杂概念
2. 生成个性化练习题
3. 评估学习进度并提供反馈

## 操作规范
- 回答必须简洁明了，适合学生理解
- 使用鼓励性语言，避免打击学习积极性
- 遇到不确定的问题，引导学生思考而非直接给答案
- 使用表情符号增加亲和力 ✨📚

## 记忆（长期上下文）
- 学生姓名: {username}
- 年级: {grade}
- 薄弱知识点: {weak_points}
- 学习偏好: {learning_style}
```

**作用**：定义Agent的核心行为规范、操作指令和需要记住的长期信息。

---

**2. SOUL.md - 人设、边界、语气**

```markdown
# SOUL.md 示例（Curiobuddy）

## 人格设定
- **身份**: 耐心友善的学习伙伴，不是严厉的老师
- **性格**: 乐观、鼓励、富有同理心
- **语言风格**: 轻松活泼，避免过于学术化
- **emoji使用**: 适度使用 📚✨🌟 增加亲和力

## 边界定义
### ✅ 应该做的
- 解释概念时举生活化例子
- 鼓励学生尝试，允许犯错
- 根据学生水平调整难度

### ❌ 不应该做的
- 直接给出答案（要引导思考）
- 使用过于复杂的术语
- 批评或贬低学生的错误
- 涉及非教育相关话题

## 语气示例
```
学生: 我还是不懂这道数学题
❌ 不好: "这么简单你都不会？"
✅ 好: "没关系！这道题确实有难度，我们一步步来看~ 🌟"
```
```

**作用**：定义Agent的"灵魂"——人格特征、行为边界、语气风格。

**3. TOOLS.md - 用户维护的工具笔记（重点！）**

```markdown

#### 多项目工作区（Multi-Project Workspace）

**OpenClaw支持多项目模式**：`~/.openclaw/workspace/` 是多项目容器，每个项目是其中的子目录。

**目录结构**（以Curiobuddy为例）：

```
~/.openclaw/
├── ...                    # 系统层配置
├── user/                  # 用户层配置
│   └── ...
└── workspace/             # 项目层根目录（多项目容器）
    ├── curiobuddy/        # 项目1: Curiobuddy产品
    │   ├── AGENTS.md     # Curiobuddy专属Agent配置
    │   ├── SOUL.md       # Curiobuddy Agent人设（教育类助手）
    │   ├── TOOLS.md      # Curiobuddy工具链
    │   ├── skills/       # Curiobuddy专属技能
    │   │   ├── content-generation/   # 内容生成技能
    │   │   ├── student-assessment/   # 学生评估技能
    │   │   └── knowledge-base/       # 知识库技能
    │   └── src/          # Curiobuddy源代码（可选）
    │
    ├── another-product/   # 项目2: 另一个产品
    │   ├── AGENTS.md     # 不同产品的Agent配置
    │   ├── SOUL.md       # 不同人设（医疗类助手）
    │   └── skills/       # 不同技能集
    │
    └── my-tools/          # 项目3: 工具集合
        ├── AGENTS.md
        └── skills/
```

**为什么选择子目录模式（而非平行）**：

| 设计 | 子目录模式（推荐） | 平行模式（~/.openclaw-curiobuddy/） |
|------|-------------------|-----------------------------------|
| **集中管理** | ✅ 所有项目在一个workspace下 | ❌ 项目散落在各处 |
| **快速切换** | ✅ `openclaw workspace switch curiobuddy` | ❌ 需要改环境变量 |
| **资源共享** | ✅ 可共享系统层和用户层配置 | ✅ 同样可行 |
| **版本控制** | ✅ 每个子目录独立git仓库 | ✅ 同样可行 |
| **备份方便** | ✅ 备份整个workspace即可 | ❌ 需备份多个目录 |

**项目激活机制**：

```bash
# 查看所有项目
$ openclaw workspace list
- curiobuddy (active)
- another-product
- my-tools

# 切换项目
$ openclaw workspace switch another-product

# 创建新项目
$ openclaw workspace init new-project
✓ Created new workspace: ~/.openclaw/workspace/new-project
```

**当前项目标识**：

```

# 内容示例:
{

**与业务项目目录的关系**：

您可能还有一个问题：**Curiobuddy的源代码放哪里？**

```
方案A: 源代码在~/.openclaw/workspace/下（OpenClaw原生方式）
~/.openclaw/workspace/curiobuddy/
├── AGENTS.md           # Agent配置
├── SOUL.md             # Agent人设
├── skills/             # 技能
└── src/                # 源代码（与OpenClaw配置在一起）
    ├── frontend/
    └── backend/

方案B: 源代码独立，OpenClaw配置通过软链接（混合方式）
~/projects/                    # 业务项目目录
└── curiobuddy/               # 业务项目根目录
    ├── README.md
    ├── src/                  # 源代码
    └── openclaw-config/      # OpenClaw配置子目录

~/.openclaw/workspace/
└── curiobuddy -> ~/projects/curiobuddy/openclaw-config/  # 软链接

方案C: 完全独立（传统方式）
~/projects/curiobuddy/        # 业务项目（独立git仓库）
~/.openclaw/workspace/curiobuddy/  # OpenClaw配置（独立管理）
→ 需要手动同步或CI/CD集成
```

**版本控制适用性分析**：

| 方案 | 适合版本控制？ | 原因 | 推荐场景 |
|------|---------------|------|---------|
| **方案A** | ⚠️ **不太适合** | `~/.openclaw/`是系统目录，不应整体git管理；单个项目可以git init，但路径在隐藏目录下，不符合常规习惯 | 个人快速原型、临时项目 |
| **方案B** | ✅ **强烈推荐** | 源代码和Agent配置在同一目录，可以一起版本控制；路径在`~/projects/`下，符合常规项目习惯 | 团队协作、开源项目、长期维护产品 |
| **方案C** | ⚠️ **可以但麻烦** | 需要维护两个git仓库，通过CI/CD或手动同步；增加了复杂性 | 大型企业、严格权限分离场景 |

**为什么方案A不太适合版本控制**：

```bash
# 方案A的问题:
~/.openclaw/workspace/curiobuddy/
├── .git/                 # 如果在这里git init
├── AGENTS.md
├── SOUL.md
├── skills/
└── src/

问题1: 路径在~/.openclaw/隐藏目录下
  → 不符合常规项目路径习惯
  → 其他工具（IDE、文件管理器）访问不便
  
问题2: 如果忘记这是个git仓库
  → 可能误删~/.openclaw/导致代码丢失
  
问题3: 团队协作时
  → 其他成员需要手动复制到~/.openclaw/workspace/
  → 而不是常规的git clone
```

**为什么方案B最适合版本控制**：

```bash
# 方案B的优势:
~/projects/curiobuddy/          # 常规项目路径
├── .git/                      # 整体版本控制
├── README.md
├── src/                       # 源代码
├── docs/                      # 文档
└── openclaw-config/           # OpenClaw配置作为项目的一部分
    ├── AGENTS.md
    ├── SOUL.md
    └── skills/

优势1: 符合常规项目结构
  → git clone后立即获得完整项目（代码+Agent配置）
  → 新成员无需额外配置
  
优势2: Agent配置随代码演进
  → PRD变更 → 同步更新AGENTS.md
  → 代码重构 → 同步更新SOUL.md
  → 历史可追溯
  
优势3: CI/CD友好
  → 构建时自动加载项目Agent配置
  → 保持开发/测试/生产环境Agent行为一致
```

**最终推荐**：
- **个人学习/原型**：方案A（简单，快速开始）
- **团队协作/产品化**：✅ **方案B（强烈推荐）**
- **企业级/严格管控**：方案C（权限分离）

**团队 onboarding 体验对比**：

```
方案B的协作体验（推荐）:
新成员: git clone https://github.com/team/curiobuddy.git
        cd curiobuddy
        
        # 自动获得:

方案A的协作体验:

```

## 3. 多 Agent、多 Skill 协同机制

### 3.1 澄清：OpenClaw 没有真正的「多 Agent」

**关键发现**：OpenClaw 运行的是「单 Agent 运行时」，所谓的「多 Agent」是通过以下机制模拟的：

1. **Session 隔离**：不同对话隔离状态（看起来像多个 Agent）
2. **Session 工具**：`sessions_send`、`sessions_spawn` 实现 Agent-to-Agent 通信
3. **Subagent**：通过 `sessions_spawn` 在隔离 Session 中运行子任务

### 3.2 真正的架构：单 Agent + 多 Skills

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   OpenClaw: 单 Agent + 多 Skills                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  单 Agent Runtime (Pi Agent Core)                                        │
│  ├─ 一个运行时进程                                                        │
│  ├─ 加载所有可用的 Skills                                                │
│  ├─ 通过 Tool 调用执行功能                                                 │
│  └─ 通过 Session 隔离不同对话                                             │
│                                                                          │
│  Skill 加载层级（优先级从高到低）                                         │
│  ├─ Level 1: Workspace Skills (/skills)                                  │
│  │   └── ~/.openclaw/workspace/skills/                                   │
│  ├─ Level 2: Managed/Local Skills                                        │
│  │   └── ~/.openclaw/skills/                                              │
│  ├─ Level 3: Bundled Skills (npm 包内置)                                  │
│  └─ Level 4: Extra Dirs (skills.load.extraDirs)                           │
│                                                                          │
│  Skill 格式: AgentSkills 兼容                                             │
│  ├─ 文件: SKILL.md                                                        │
│  ├─ Frontmatter: YAML (name, description, metadata)                       │
│  └─ Content: Markdown (instructions)                                    │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.4 Agent-to-Agent 通信机制

**不是真正的多 Agent 并行，而是 Session 间通信**

```
工具集: Session Tools
├─ sessions_list      # 列出会话
├─ sessions_history   # 获取会话历史
├─ sessions_send      # 向另一个会话发送消息
└─ sessions_spawn     # 在隔离会话中生成子 Agent
```

#### sessions_spawn 详解（Subagent 机制）

```typescript
// 参数
{
  task: "分析代码库",           // 任务描述
  label?: "代码分析",           // 日志/UI 用标签
  agentId?: "sub-agent-1",      // 指定 Agent ID（可选）
  model?: "gpt-4",              // 覆盖模型
  thinking?: "high",            // 思考级别
  runTimeoutSeconds?: 300,      // 超时时间
  thread?: true,                // 线程绑定
  mode?: "run" | "session",     // run=单次, session=持久
  cleanup?: "keep",             // 完成后是否删除
  sandbox?: "inherit",          // 沙盒继承
}

// 行为
1. 创建新的 agent::<agentId>::subagent:: 会话
2. 子 Agent 默认拥有完整工具集（减去 session tools）
3. 子 Agent 不能再 spawn 子 Agent（防止递归）
4. 非阻塞：立即返回 { status: "accepted", runId, childSessionKey }
5. 完成后，OpenClaw 自动向请求者通道发送结果公告
6. 子 Agent 会话在 archiveAfterMinutes（默认60）后自动归档

// 安全限制
- 子 Agent allowlist: agents.list[].subagents.allowAgents
- 沙盒继承检查: 如果请求者已沙盒化，拒绝非沙盒目标
```

### 3.5 多 Skill 协同示例

```
用户请求: "帮我分析这个项目，生成一个报告"

执行流程:
1. Agent 加载所有可用 Skills
   ├─ browser (内置)
   ├─ read/write/edit (内置)
   ├─ summarize (Skill)
   ├─ image-lab (Skill)
   └─ code-review (Skill)

2. Agent 决策使用哪些 Skills
   ├─ 使用 read 工具读取项目文件
   ├─ 使用 summarize Skill 生成摘要
   ├─ 使用 image-lab Skill 生成架构图
   └─ 使用 write 工具写入报告

3. 如果任务复杂，调用 sessions_spawn
   ├─ 主 Agent: "分析代码架构"
   ├─ 子 Agent (spawned): 专门分析代码
   └─ 子 Agent 完成后，结果返回主 Agent

4. 最终结果汇总返回给用户
```

## 4. 关键洞察：多 Agent 协同是否是伪命题？

### 4.1 分析框架

| 定义 | 标准 | OpenClaw 实现 | 结论 |
|------|------|--------------|------|
| **严格定义** | 多个独立决策实体，并行运行，动态协调 | ❌ 单运行时，Session 隔离 | 是伪命题 |
| **实用定义** | 不同上下文隔离，看起来像是多个 Agent | ✅ Session 隔离 | 不是伪命题 |
| **工程定义** | 子进程/线程隔离，父 Agent 协调 | ✅ Subagent 机制 | 不是伪命题 |

### 4.2 OpenClaw 的设计哲学

```
设计选择: 单 Agent + Session 隔离 vs 多 Agent

选择单 Agent 的原因:
1. 资源效率: 一个运行时进程，复用模型连接
2. 一致性: 所有 Skills 共享同一个上下文管理系统
3. 简单性: 不需要 Agent 间复杂的协调协议
4. 可预测性: 线性执行，易于调试

何时需要 Subagent:
1. 任务隔离: 不想让子任务污染主会话上下文
2. 并发执行: 多个独立任务并行（通过 spawn）
3. 安全沙盒: 不信任的代码在隔离环境运行
```

### 4.3 与 LangGraph 的对比

| 维度 | OpenClaw | LangGraph |
|------|---------|-----------|
| **Agent 模型** | 单 Agent + Session 隔离 | 多 Executor（真多 Agent） |
| **协调机制** | Session Tools（父-子） | StateGraph（图流转） |
| **进程模型** | 单 Gateway 进程 | FastAPI + 独立后端 |
| **状态管理** | Session JSONL 文件 | Checkpoint 持久化 |
| **复杂度** | 中（本地优先） | 高（分布式架构） |
| **适用场景** | 个人助手、本地工具 | 企业级复杂工作流 |
| **生产就绪** | ✅ 已验证（327k+ stars） | ⚠️ init-mvp 中未触发 |

### 4.4 对「第三选择」的启示

**核心启示：不要追求「真正的多 Agent」，而要追求「可验证的单 Agent 执行」**

```
LangGraph 的问题:
- 追求真正的多 Agent 并行
- 导致架构复杂（独立后端、StateGraph、Checkpoint）
- 实战落地困难（init-mvp 中未触发）

OpenClaw 的成功:
- 接受单 Agent 现实
- 用 Session 隔离实现「看起来像多 Agent」
- 架构简单，生产就绪

「第三选择」应该学习 OpenClaw:
1. 单 Agent 运行时（利用 Cursor 内置 Agent）
2. Session/项目隔离（通过目录结构）
3. Skill 模块化（通过 SKILL.md 规范）
4. 轻量级验证（通过前置/后置检查）
```

## 5. OpenClaw 对 ai-agent-dev-system 的借鉴

### 5.1 可以借鉴的机制

| OpenClaw 机制 | 应用到 ai-agent-dev-system | 难度 |
|--------------|---------------------------|------|
| **三层 Skill 加载** | Bundled(rules) + Managed(global) + Workspace(project) | 低 |
| **SKILL.md 格式** | 已有 SKILL.md，增加 YAML frontmatter | 低 |
| **Skill Gating** | 通过 metadata 控制 Skill 加载条件 | 中 |
| **dmScope 隔离** | 用 change-id 实现类似 Session 隔离 | 低 |
| **Session Tools** | 用迭代日志 + records 实现类似机制 | 中 |
| **Subagent spawn** | 用 Task 工具或独立文件实现 | 中 |
| **Bootstrap 文件** | AGENTS.md → 已有；增加 SOUL.md, TOOLS.md | 低 |
| **Gateway 架构** | ❌ 不需要，Cursor 就是 Gateway | - |

## 6. 最终结论与建议

### 6.1 三个核心问题的回答

| 问题 | 回答 |
|------|------|
| **1. OpenClaw 如何解决三层执行规范？** | 系统层(Gateway config) + 用户层(Session dmScope/identityLinks) + 项目层(Workspace/Skills)。关键是「Session 隔离」而非「Agent 隔离」 |
| **2. 多 Agent、多 Skill 如何协同？** | **没有真正的多 Agent**，是「单 Agent + 多 Skills + Session 隔离」。协同通过 Session Tools（list/history/send/spawn）实现 |
| **3. 多 Agent 协同是否是伪命题？** | ✅ **是伪命题**。严格定义下，主流产品（OpenClaw、Cursor、Claude Code）都用「单 Agent + Skills」模式。实用定义下，Session/Subagent 机制已足够 |

### 6.2 对 ai-agent-dev-system 的战略建议

**不要再追求「真正的多 Agent」架构**，而是学习 OpenClaw：

1. **接受单 Agent 现实**：Cursor/Claude 内置的 Agent 已足够强大
2. **强化 Skill 系统**：借鉴 OpenClaw 的三层加载 + Skill Gating
3. **建立验证层**：这是关键缺失——用 Pre/Post Check 替代 LangGraph 的强制流转
4. **简化架构**：放弃独立后端，完全宿主原生

### 6.3 下一步行动（基于 OpenClaw 研究）

| 优先级 | 行动 | 预期产出 |
|-------|------|---------|
| **P0** | 设计「验证层」原型 | Pre/Post Check 机制，可以在 Cursor 内运行 |
| **P0** | 改造 SKILL.md 格式 | 增加 YAML frontmatter，支持 Skill Gating |
| **P1** | 实现三层 Skill 加载 | Global → Project → Change-specific |
| **P1** | 增加 Bootstrap 文件 | SOUL.md, TOOLS.md 等 |
| **P2** | 移除 LangGraph 后端依赖 | 简化部署，降低启动成本 |
| **P2** | 评估是否完全迁移到 OpenClaw 模式 | 或者直接使用 OpenClaw 作为基础？ |
