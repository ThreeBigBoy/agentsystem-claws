## 执行摘要（Executive Summary）

### 核心发现

1. **OpenAI Agents SDK 已生产就绪**：作为 Swarm 的继任者，20,000+ GitHub stars，提供五大核心原语（Agents、Tools、Handoffs、Guardrails、Tracing）

2. **Claude Code 的 Subagent 架构**：通过子进程隔离实现「伪多Agent」，实际是分层的单Agent+专项工具模式

3. **「多Agent协同」可能是伪命题**：主流产品（Cursor、Claude Code、GitHub Copilot）实际采用「单Agent + 多Tools/Skills」模式，而非真正的多Agent协同

4. **历史证据确认**：`.cursor/rules/agent.mdc` 和 `global-rules.mdc` 文件存在，证明「借船出海」确实尝试过

5. **「第三选择」方向**：轻量级验证层 + 宿主原生执行，可能是比 LangGraph 更可行的路径

## 1. 历史记录查证：「借船出海」确实尝试过

### 1.1 现存证据

```bash
# 在 ai-agent-dev-system 仓库中发现的历史文件
.cursor/rules/
├── agent.mdc      # Agent 角色约定 - 身份回答为主Agent
└── global-rules.mdc  # 全局规则入口
```

### 1.2 文件内容分析

**agent.mdc** 核心内容：
- 身份持久约定（主Agent）
- 入口职责与规则来源（指向 `agents/主Agent.md`）
- Cursor 宿主说明（薄入口）

**global-rules.mdc** 核心内容：
- 规则来源声明（OpenSpec.md、global-rules/*.md）
- 加载策略声明（simple/heavy 判定）
- 执行方与技能判定（skills-rules-for-agent.md）

### 1.3 推断的失败原因

基于文件内容分析，推测最初「借船出海」失败的原因：

| 问题 | 证据 | 影响 |
|------|------|------|
| **规则过于复杂** | 要求读取多个文件后才能执行 | 用户/Agent 难以遵循 |
| **缺乏强制机制** | 使用「须」「应」等建议性措辞 | 可以被忽略 |
| **没有验证层** | 只定义规则，不验证执行 | 无法确保遵守 |
| **薄入口设计** | 自称「薄入口」，实际依赖后端 | 后端未启动时失效 |

**关键失败点**：设计意图是「Cursor Rules 作为入口，触发 LangGraph 后端执行」，但两者未有效衔接，导致「既不强制，也不灵活」。

```
┌─────────────────────────────────────────────────────────────┐
│                 OpenAI Agents SDK 架构                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. Agents（代理）                                            │
│     ├─ LLM 配置（模型选择、指令、工具）                       │

│     └─ 轻量级封装，非持久化进程                               │
│                                                              │

│     ├─ 普通 Python 函数                                       │
│     ├─ 自动 JSON Schema 转换                                  │

│  3. Handoffs（交接）                                          │
│     ├─ 与 Tool Call 的区别：父Agent完全放弃控制权            │

│     └─ 实现路由模式                                          │
│                                                              │
│  4. Guardrails（护栏）                                        │
│     ├─ 输入/输出验证                                          │

│     └─ 策略执行                                               │
│                                                              │
│  5. Sessions & Tracing（会话与追踪）                         │
│     ├─ 自动对话历史管理                                       │
│     ├─ 可观测性（Logfire、AgentOps 集成）                     │
│     └─ 三种记忆策略：手动/SDK管理/服务端管理                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 与 LangGraph 的关键差异

| 维度 | OpenAI Agents SDK | LangGraph |
|------|-------------------|-----------|
| **架构重量** | 轻量级 | 重量级 |
| **状态管理** | 会话级（Session） | 图级（StateGraph） |
| **持久化** | 可选（三种策略） | 强制（Checkpointer） |
| **进程模型** | 无独立进程 | FastAPI 独立进程 |
| **启动复杂度** | 低（库调用） | 高（服务启动） |
| **宿主集成** | 原生支持 | HTTP 通信 |
| **调试难度** | 低 | 高（多进程） |
| **生产就绪** | ✅ 已验证 | ⚠️ 实战暴露问题 |

### 2.4 对「第三选择」的启示

**关键洞察**：OpenAI Agents SDK 的设计理念是「轻量级 + 宿主原生」，与 LangGraph 的「独立后端」形成对比。

**启示**：
1. **无需独立进程**：可以在 Cursor/Claude 宿主内直接调用
2. **可选持久化**：不需要强制 checkpoint 机制
3. **Guardrails 内置**：提供验证层（与「验证前置」模式契合）

### 3.2 两种「多Agent」机制对比

| 机制 | Subagents（Task Tool） | Agent Teams |
|------|----------------------|-------------|
| **执行模式** | 单次执行 | 持久会话 |
| **上下文** | 独立上下文窗口 | 共享任务队列 |
| **通信** | 无（直接返回结果） | 文件系统 inbox 消息 |
| **协调** | 无（父 Agent 协调） | 多轮协调 |
| **Token 成本** | 较低 | 高 2x |
| **使用场景** | 简单任务 | 复杂协作 |

**关键洞察**：Claude Code 的「多Agent」实际上是「单Agent + 子进程工具」，而非真正的多Agent协同。

## 4. 核心命题研究：「多Agent协同」是否是伪命题？

### 4.1 主流产品架构对比

| 产品 | 宣称架构 | 实际架构 | 协同机制 |
|------|---------|---------|---------|
| **Cursor** | 单 Agent | 单 Agent + Rules | 宿主内置 |
| **Claude Code** | Multi-Agent | 单 Agent + Subagents（工具） | 父-子进程 |
| **GitHub Copilot** | 单 Agent | 单 Agent + 上下文感知 | 宿主内置 |
| **OpenAI Agents SDK** | Multi-Agent | Agent + Tools + Handoffs | 函数调用/交接 |
| **LangGraph** | Multi-Agent | StateGraph + Executors | 独立后端 |

### 4.2 关键发现

**真正的「多Agent协同」定义**：
- 多个独立决策实体
- 相互通信协调
- 共享或分配目标
- 动态任务分配

**主流产品的实际实现**：
- **Cursor**: 单 Agent 根据 Rules 切换角色
- **Claude Code**: 单 Agent 调用 Subagent「工具」
- **GitHub Copilot**: 单 Agent 根据上下文调整

### 4.3 结论：「多Agent协同」是伪命题吗？

**回答：取决于定义**

| 定义层级 | 是否是伪命题 | 说明 |
|---------|-------------|------|
| **严格定义**（独立决策实体） | ✅ 是伪命题 | 主流产品未实现 |
| **实用定义**（角色分工） | ❌ 不是伪命题 | 单Agent+不同角色即可满足 |
| **工程定义**（进程隔离） | ❌ 不是伪命题 | Subagent/Handoffs 已实现 |

**关键洞察**：
- 商业化产品不需要「真正的多Agent协同」
- 需要的是「角色分工 + 可验证的执行」
- 单Agent+多Skills/Subagents 模式已足够

## 5. 「第三选择」架构设计建议

### 5.1 设计原则

基于研究成果，「第三选择」应遵循以下原则：

```yaml
原则 1: 宿主原生（Host-Native）
├── 不依赖独立后端进程
├── 在 Cursor/Claude 宿主内执行
└── 利用宿主内置 Agent 能力

原则 2: 轻量级验证（Lightweight Validation）
├── 不强制复杂状态图
├── 关键节点自动化验证
└── 产出物模板检查

原则 3: 可选强制（Optional Enforcement）
├── 默认辅助（建议性）
├── 可切换为强制（阻断性）
└── 用户控制严格程度

原则 4: 技能化封装（Skill-Based）
├── 不定义固定 Agent 角色
├── 将能力封装为可调用 Skills
└── 主Agent根据需要加载 Skills
```

### 5.2 推荐架构：「轻量级验证层 + 宿主原生执行」

```
┌─────────────────────────────────────────────────────────────┐
│              「第三选择」推荐架构                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Layer 1: Cursor/Claude 宿主层                                │
│  ├─ 内置 Agent（Claude 3.5/4、GPT-4o）                       │
│  ├─ MCP 工具集成                                            │
│  └─ Rules/CLAUDE.md 上下文                                   │
│                                                              │
│  Layer 2: 轻量级验证层（Validation Layer）                    │
│  ├─ Pre-execution Check: 前置条件验证                        │
│  │   └── 检查是否读取 SKILL.md、是否创建 change-id          │
│  ├─ Post-execution Check: 后置产出物验证                    │
│  │   └── 验证产出物是否符合模板、命名规范                    │
│  └─ Gate Check: 门控检查                                     │
│      └── 模拟 HC0/HC2/HC7，但轻量级实现                     │
│                                                              │
│  Layer 3: 技能封装层（Skills Layer）                          │
│  ├─ request-analysis Skill                                  │
│  ├─ prd-review Skill                                          │
│  ├─ coding-implement Skill                                    │
│  ├─ code-review Skill                                         │
│  └─ func-test Skill                                           │
│                                                              │
│  Layer 4: 规则与记忆（Rules & Memory）                        │
│  ├─ OpenSpec.md（轻量版）                                    │
│  ├─ global-rules（核心规则）                                 │
│  └─ memory（可选加载）                                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.3 与现有架构的对比

| 维度 | 原 LangGraph 架构 | 原 Cursor 架构 | 「第三选择」架构 |
|------|------------------|---------------|-----------------|
| **后端** | FastAPI 独立进程 | 无 | 无（验证层在宿主内） |
| **启动** | 复杂（需手动启动） | 简单 | 简单 |
| **强制力** | 高（StateGraph） | 低（建议性） | 中（验证层可选强制） |
| **验证** | 后端内置 | 无 | 轻量级验证层 |
| **可维护性** | 低 | 高 | 高 |
| **商业化适配** | ❌ 复杂难落地 | ❌ 无法验证 | ✅ 平衡 |
