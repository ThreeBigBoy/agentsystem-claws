## 1. 问题发现

### 1.1 现象

在执行 curiobuddy v0.1beta 的 PRD 评审和技术方案评审时，Agent 只执行了 `check_*.py` 脚本，跳过了 Skill 评审环节。

**评审执行记录**：
- PRD 评审：只执行了 `check_prd.py`
- 技术方案评审：只执行了 `check_solution.py`
- **均未执行**：prd-review Skill、architecture-review Skill、technical-design-review Skill

### 1.2 影响

| 问题 | 影响 |
|------|------|
| 跳过 Skill 评审 | 缺少深度语义评审，只有量化评分 |
| 缺少评审纪要 | 无法追溯评审过程和结论 |
| 阻塞项遗漏 | check_*.py 无法替代 Skill 的深度分析 |

### 1.3 根因分析

**最初错误归因**：
- 最初认为问题是 "run_gates.py 不调用 Skills"
- 进一步分析后认为是 "Agent 执行 ad-hoc 行为"

**正确归因**：
- SKILL.md 定义了执行顺序，但没有定义 "How"（如何确保做到）
- SKILL.md 说 "强制"，但没有说跳过会怎样
- 没有校验机制来检测 Skill 评审是否被执行

## 2. 讨论过程

### 2.1 第一性原理分析

**核心问题**：质量门禁系统的目的是什么？

**答案**：确保每个阶段出口都经过严格检查，防止有问题的产出进入下一阶段。

**约束**（来自 SKILL.md）：
```
Step 2: prd-review Skill → check_prd.py（两者都强制）
Step 4: architecture-review + technical-design-review Skill → check_solution.py（两者都强制）
Step 6: code-review Skill → check_code.py（两者都强制）
```

**问题**：
- 说"强制"，但没有机制确保强制执行
- Agent 可能选择直接调用 check_*.py

### 2.2 SKILL.md 缺陷分析

| # | 缺陷类别 | 问题描述 |
|---|---------|---------|
| 1 | **歧义** | L182-183 "先Agent评审后量化检查" 只说了 2 个步骤，遗漏了"综合判定"，且与 Step 4 双 Skill 结构不符 |
| 2 | **触发机制** | 只说"调用 Skill"，没说怎么触发、怎么判断当前阶段 |
| 3 | **校验机制** | 说"强制校验"但无具体方案；tasks.md 机制太重 |
| 4 | **跳过后果** | 说"强制"但无后果描述 |
| 5 | **异常处理** | 无异常场景说明 |
| 6 | **执行入口** | Agent 可能直接调用 check_*.py 绕过 Skill |
| **7** | **综合判定保障** | **综合判定环节无执行保障（后补充）** |
| **8** | **综合判定 vs Gate** | **综合判定 ≠ Gate 确认，关系和顺序不明确（后补充）** |

### 2.3 关键决策

**决策 1：tasks.md 机制太重，简化**

- 原有思路：用 tasks.md 追踪执行状态
- 决策：改为基于"文件存在性"的校验机制
  - Skill 评审纪要文件存在 = Skill 评审已执行
  - check_*.py 已执行 = 量化评分已完成
  - 两者都存在 = 可以做综合判定

**决策 2：短期方案 vs 长期方案**

- 短期：不求 100% 执行确定性，只求不要遗漏即可
- 长期：建立强制执行机制

### 2.4 改进方向讨论

**改进点 1**：L182-183 歧义优化
- 采用方案 A：分 Step 描述每个阶段的完整步骤

**改进点 2-5**：SKILL.md 增强
- 明确触发机制
- 定义校验机制（基于文件存在性）
- 定义跳过后果
- 补充异常处理

**改进点 6**：check_*.py 警告机制
- 自动检测 Skill 评审纪要是否存在
- 未找到时输出警告（不阻止执行）

### 2.5 综合判定执行保障讨论（后补充）

**发现的问题**：

在审查改进方案时，发现综合判定环节没有执行保障：

| 问题 | 缺失 |
|------|------|
| **谁来执行综合判定？** | Agent？check_*.py？ |
| **综合判定产出位置？** | 评审纪要？check_*.py 输出？ |
| **如何确保执行？** | 无保障机制 |

**补充的改进点 7-8**：

- **改进点 7**：综合判定执行保障
  - 执行者：Agent
  - 产出位置：追加到 Skill 评审纪要末尾
  - 保障机制：未执行不得进入下一阶段

- **改进点 8**：综合判定 vs Gate 确认
  - 综合判定 ≠ Gate 确认
  - 顺序：综合判定 → Gate 确认 → 进入下一阶段

## 3. 教训总结

### 3.1 技术教训

| 教训 | 说明 |
|------|------|
| **"强制"需要机制支撑** | SKILL.md 定义了"强制"，但没有校验机制，实际无法保证执行 |
| **不能假设 Agent 会遵守规范** | Agent 有自主决策能力，可能选择熟悉的路径（check_*.py）而非规范路径 |
| **治标 vs 治本** | 在 check_*.py 加警告是治标，修改 SKILL.md 定义是治本 |
| **简化优于复杂** | tasks.md 追踪机制太重，文件存在性校验更轻量且足够 |

### 3.2 流程教训

| 教训 | 说明 |
|------|------|
| **先讨论再修改** | 避免过早陷入代码实现，确保方案对齐 |
| **第一性原理思考** | 从问题本质出发，而非从现有代码出发 |
| **不求完美但求有效** | 短期方案：警告机制；长期方案：强制验证 |

## 6. 附录

### 6.1 评审纪要文件命名规范

| Skill | 评审纪要路径模式 |
|-------|-----------------|
| prd-review | `records/PRD-[change-id]-评审纪要.md` |
| architecture-review | `records/architecture-review-[change-id]-评审纪要.md` |
| technical-design-review | `records/technical-design-review-[change-id]-评审纪要.md` |
| code-review | `records/code-review-[change-id]-评审纪要.md` |

### 6.2 三步执行顺序

```
Step N 评审（N = 2/4/6）
    ↓
步骤 1：【强制】Skill 评审 → 产出评审纪要
    ↓
步骤 2：【强制】check_*.py 量化评分 → 产出量化分数
    ↓
步骤 3：【强制】综合判定 → Skill 评审结论 + 量化评分
    ↓
Gate 确认 → 用户人工签字 → 进入下一阶段
```
