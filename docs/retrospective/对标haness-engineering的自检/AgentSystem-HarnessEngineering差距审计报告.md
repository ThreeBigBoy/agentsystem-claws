# Harness Engineering视角下的AgentSystem全面审计报告

## 一、审计背景

### 1.1 审计依据

根据 [Harness Engineering第一性原理深度解析](../../docs/temp/Harness%20Engineering第一性原理深度解析.md)，Harness Engineering包含三个核心层次：

| 层次 | 解决的问题 | 核心手段 |
|------|-----------|---------|
| **上下文工程** | "理解"问题 | docs/结构、AGENTS.md、知识图谱 |
| **架构约束** | "正确性"问题 | 分层架构、Linter、品味不变式、测试 |
| **熵管理** | "可持续性"问题 | 循环清理、自动重构、文档治理 |

### 1.2 审计目的

对照Harness Engineering的三层框架，系统性检查AgentSystem的：
1. 当前能力覆盖情况
2. 差距与不足
3. 改进建议与优先级

---

## 二、上下文工程层审计

### 2.1 Harness Engineering要求

| 要求 | 说明 |
|------|------|
| **docs/目录** | 所有文档以机器可读格式存储在仓库中 |
| **AGENTS.md** | 定义Agent行为规则、权限、协作流程 |
| **代码即文档** | 将黄金原则编码到代码中 |
| **渐进式披露** | Agent仅访问当前任务所需上下文 |

### 2.2 AgentSystem现状

| 组件 | 存在？ | 文件 | 说明 |
|------|--------|------|------|
| **docs/目录** | ✅ | `docs/project-prd-changes/[change-id]/` | 项目变更文档 |
| **AGENTS.md** | ⚠️ | 部分项目中存在 | 非系统级强制 |
| **知识图谱** | ✅ | `openspec/changes/[change-id]/` | 变更知识结构化 |
| **代码即文档** | ✅ | 各skill的REFERENCE | 规范编码到文件 |
| **渐进式披露** | ❌ | 无 | Agent一次性加载全部上下文 |

### 2.3 差距分析

| 差距 | 严重程度 | 说明 |
|------|---------|------|
| **AGENTS.md非系统级强制** | 高 | 各项目可自行决定是否创建，缺乏系统级约束 |
| **缺乏渐进式披露机制** | 中 | Agent通常一次性加载所有相关文档，可能导致上下文膨胀 |
| **黄金原则未系统编码** | 中 | 品味不变式散落在各skill中，未形成系统化的编码规则 |

### 2.4 改进建议

**P0（立即实施）**：
1. 在 `sys-root/lib/` 下创建系统级 `AGENTS.md`，定义Agent的通用行为规则
2. 在skill.json或skill.md中明确上下文加载的最佳实践

**P1（短期实施）**：
1. 建立"渐进式上下文加载"规范，避免Agent上下文膨胀
2. 将黄金原则提炼为可验证的编码规则

---

## 三、架构约束层审计

### 3.1 Harness Engineering要求

| 要求 | 说明 |
|------|------|
| **分层架构** | Types→Config→Repo→Service→Runtime→UI |
| **自定义Linter** | 针对业务场景的静态代码检查 |
| **品味不变式** | 将代码风格偏好编码为可验证规则 |
| **单元/集成测试** | 覆盖所有关键路径 |
| **可观测性集成** | LogQL/PromQL直接查询应用状态 |
| **操作权限控制** | 定义Agent可修改的代码范围 |

### 3.2 AgentSystem现状

| 组件 | 存在？ | 文件 | 说明 |
|------|--------|------|------|
| **分层架构规范** | ✅ | `coding-implement/REFERENCE/spec-backend-*.md` | 后端分层定义 |
| **自定义Linter** | ❌ | 无 | 缺乏业务场景Linter |
| **品味不变式** | ❌ | 无 | 未系统化定义 |
| **单元/集成测试** | ⚠️ | `func-test/` | 功能验收测试，非单元测试 |
| **可观测性** | ✅ | `code-review/SKILL.md`提及 | 检查日志、监控埋点 |
| **操作权限控制** | ⚠️ | `tasks.md`任务勾选 | 隐式权限控制 |

### 3.3 差距分析

| 差距 | 严重程度 | 说明 |
|------|---------|------|
| **缺乏自定义Linter** | 高 | 无法自动化检查业务场景特定的代码规范 |
| **品味不变式未定义** | 中 | 代码风格偏好未编码为可验证规则 |
| **单元测试覆盖不足** | 高 | 主要依赖功能验收测试，缺乏细粒度单元测试 |
| **可观测性停留在检查项** | 中 | 可观测性只是Review维度之一，非运行时能力 |

### 3.4 改进建议

**P0（立即实施）**：
1. 创建业务场景Linter规范（如命名规范、架构合规性检查）
2. 将品味不变式编码为 `project-rules/` 中的可验证规则

**P1（短期实施）**：
1. 在 `coding-implement/` 中增加单元测试指导
2. 建立可观测性检查的自动化机制

## 四、熵管理层审计

### 4.1 Harness Engineering要求

| 要求 | 说明 |
|------|------|
| **后台扫描任务** | 定期检测冗余代码、死代码、反模式 |
| **自动发起重构PR** | 可自动修复的问题直接提交PR |
| **文档治理** | Agent自动更新过时文档，保持docs/与代码同步 |
| **架构漂移检测** | 持续监控代码依赖关系、分层架构合规性 |

### 4.2 AgentSystem现状

| 组件 | 存在？ | 文件 | 说明 |
|------|--------|------|------|
| **后台扫描任务** | ❌ | 无 | 缺乏自动化的代码库扫描 |
| **自动重构PR** | ❌ | 无 | 重构依赖人工发起 |
| **文档治理** | ⚠️ | `anti-pattern-*.md` | 有反模式识别，但无自动修复 |
| **架构漂移检测** | ⚠️ | `anti-pattern-terminology-drift.md` | 识别术语漂移，但非架构层面 |

### 4.3 差距分析

| 差距 | 严重程度 | 说明 |
|------|---------|------|
| **熵管理基本空白** | 高 | 这是AgentSystem与Harness Engineering差距最大的层面 |
| **缺乏自动化清理机制** | 高 | 死代码、冗余依赖需人工清理 |
| **文档同步依赖人工** | 中 | docs/与代码同步无自动化保障 |
| **无架构漂移检测** | 中 | 分层架构合规性靠人工Review |

### 4.4 改进建议

**P0（立即实施）**：
1. 创建定期扫描任务，发现反模式时自动发起修复PR
2. 建立文档同步检查机制

**P1（短期实施）**：
1. 开发架构合规性自动检测工具
2. 建立循环清理流程，类似垃圾回收机制

## 五、与Quality Gate的关系

### 5.1 AgentSystem的Quality Gate

AgentSystem已有质量门禁机制：

| 组件 | 说明 |
|------|------|
| **三层门禁** | tasks状态一致性、阶段准入声明、质量评分 |
| **评审修复循环** | 通过→有条件通过→不通过的判定与修复 |
| **评分阈值** | PRD 80%、方案 70%等阈值控制 |

### 5.2 与Harness Engineering的对比

| 维度 | Quality Gate | Harness Engineering |
|------|-------------|---------------------|
| **定位** | 人工触发的事后检查 | Agent执行中的实时约束 |
| **时机** | 代码合并前 | 代码生成→验证→合并全流程 |
| **手段** | 评分阈值 | 架构约束、可观测性、熵管理 |
| **反馈** | 事后评分 | 实时反馈+自动修正 |
| **范围** | 代码质量 | 上下文+架构+知识+代码 |

### 5.3 互补关系分析

**Quality Gate是Harness Engineering的重要组成部分**，但Harness Engineering的范畴更广：

```
┌─────────────────────────────────────────────────────┐
│                   熵管理（Entropy Management）        │
│         AgentSystem：❌ 基本空白                     │
├─────────────────────────────────────────────────────┤
│                   架构约束（Architecture Constraints）│
│         AgentSystem：⚠️ 部分覆盖（Review检查项）      │
├─────────────────────────────────────────────────────┤
│                   上下文工程（Context Engineering）   │
│         AgentSystem：✅ 基本覆盖                     │
└─────────────────────────────────────────────────────┘
                    ↑
            Quality Gate
```

## 六、总体差距矩阵

### 6.1 各层覆盖情况

| 层次 | 要求项 | 覆盖项 | 覆盖率 | 差距 |
|------|--------|--------|--------|------|
| **上下文工程** | 5 | 3 | 60% | 中 |
| **架构约束** | 6 | 2 | 33% | 高 |
| **熵管理** | 4 | 0 | 0% | 极高 |

### 6.2 优先级排序

| 优先级 | 改进项 | 预期效果 | 工作量 |
|--------|--------|---------|--------|
| **P0** | 创建系统级AGENTS.md | 统一Agent行为规范 | 低 |
| **P0** | 开发业务场景Linter | 自动化代码质量检查 | 中 |
| **P1** | 建立品味不变式规则 | 系统化代码风格控制 | 中 |
| **P1** | 增加单元测试指导 | 提升代码质量基线 | 低 |
| **P2** | 开发熵管理扫描任务 | 自动发现并修复问题 | 高 |
| **P2** | 建立架构漂移检测 | 自动化架构合规性 | 高 |

## 七、结论

### 7.1 核心发现

1. **上下文工程层**：AgentSystem基本覆盖，但AGENTS.md非系统级强制
2. **架构约束层**：部分覆盖，缺乏自定义Linter和品味不变式
3. **熵管理层**：基本空白，是与Harness Engineering差距最大的层面

### 7.2 核心建议

> **短期重点**：补齐架构约束层的Linter和品味不变式
>
> **中期重点**：建立熵管理层，实现自动化的代码库健康检查
>
> **长期重点**：将Quality Gate升级为完整的Harness系统

### 7.3 行动计划

| 时间 | 行动 | 目标 |
|------|------|------|
| **本周** | 创建系统级AGENTS.md | 统一Agent行为规范 |
| **两周内** | 开发业务场景Linter规范 | 自动化代码质量检查 |
| **一个月内** | 建立品味不变式规则 | 系统化代码风格控制 |
| **三个月内** | 开发熵管理扫描任务 | 自动发现反模式 |
