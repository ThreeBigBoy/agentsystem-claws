# 技术方案评审双技能质量提升复盘

> **日期**: 2026-03-30
> **参与**: 用户 + Agent
> **背景**: curiobuddy迭代v0.1.1技术方案评审过程中，发现评分机制和Skill存在深层问题

---

## 一、问题发现

### 1.1 评分机制问题

**现象**：第三次评审后，量化评分从73.9提升到78.0，但数据流设计、存储设计、项目结构三个维度仍然停留在60分。

**根本原因分析**：

| 维度 | 改进前得分 | 问题诊断 |
|------|-----------|---------|
| 接口定义清晰 | 100 | 已有完整入参/出参/错误码 |
| 数据流设计 | 60 | 关键词存在即给20分，无法区分"有"和"好" |
| 存储设计 | 60 | 关键词存在即给20分，无法区分"有"和"好" |
| 项目结构 | 60 | 关键词存在即给20分，无法区分"有"和"好" |

**第一性问题**：

| 问题 | 表现 | 后果 |
|------|------|------|
| 可被刷分 | 堆砌关键词但不理解本质 | 技术方案看起来好但实际差 |
| 无法区分深度 | 写一行和写十行都得满分 | 无法反映真实质量 |
| 违背第一性原理 | 关注"有没有"而非"做没做" | 评审不反映方案本质 |

### 1.2 Skill切分问题

**原有两个Skill的定位**：

| Skill | 原定位 | 问题 |
|-------|--------|------|
| architecture-review | 产出物规范性 | 职责与technical-design-review有重叠 |
| technical-design-review | 生产就绪性 | 检查维度不区分项目类型 |

## 二、行业最佳实践研究

### 2.1 ISO/IEC 25010 软件质量模型

| 质量特性 | 子特性 | 适用评审 |
|---------|--------|---------|
| **功能适用性** | 功能完整性、功能正确性 | architecture-review |
| **性能效率** | 时间特性、资源利用 | technical-design-review |
| **安全性** | 保密性、完整性、可审计性 | technical-design-review |
| **可维护性** | 模块性、可重用性、可测试性 | 两者都有 |

### 2.2 IEEE 1012 软件验证与确认标准

| 评审类型 | 时机 | 关注点 |
|---------|------|-------|
| **架构评审** | 设计阶段 | 可行性、一致性、风险 |
| **详细设计评审** | 设计阶段 | 正确性、完整性 |
| **代码评审** | 实现阶段 | 缺陷、风格 |

### 2.3 ATAM (Architecture Tradeoff Analysis Method)

| 阶段 | 输出 | 评审内容 |
|------|------|---------|
| **阶段1: 架构描述** | 架构文档 | 完整性和清晰度 |
| **阶段2: 风险分析** | 风险点列表 | 敏感点和权衡点 |
| **阶段3: 评估** | 分析报告 | 质量属性效用树 |

### 2.4 互联网大厂常见做法

| 公司 | 评审切分 | 评审时机 |
|------|---------|---------|
| 阿里 | 设计评审 + 代码评审 | 开发前 / 开发中 |
| 腾讯 | 技术方案评审 + 专项评审（安全/性能） | 开发前 / 上线前 |
| 字节 | 架构评审 + Code Review | 开发前 / 开发后 |
| Google | Design Review + Test Review | 设计阶段 / 测试阶段 |

**共同规律**：大厂都是**按生命周期阶段切分**，而非按内容维度切分。

### 3.2 核心问题区分

| 问题类型 | 应该用哪个Skill | 说明 |
|---------|----------------|------|
| "这份方案能直接开发吗？" | architecture-review | 检查PRD可追溯性、接口契约、数据模型 |
| "这份方案能稳定上线吗？" | technical-design-review | 检查安全性、稳定性、可观测性 |
| "F-ID都有对应实现吗？" | architecture-review | 检查功能点覆盖 |
| "上线后会不会出问题？" | technical-design-review | 检查生产就绪性 |
| "有没有安全/稳定/性能问题？" | technical-design-review | 检查生产就绪模块 |
| "文档结构是否完整？" | architecture-review | 检查9项自检清单 |
| "命名是否规范？" | architecture-review | 检查文档规范性 |

### 3.3 第一性原理视角

| Skill | 第一性原理问题 | 回答 |
|-------|---------------|------|
| **architecture-review** | "这份技术方案的**好**体现在哪里？" | "能落地"——PRD可追溯、接口清晰、数据模型正确、方案可实施 |
| **technical-design-review** | "这份技术方案的**好**体现在哪里？" | "能上线"——安全有效、稳定有效、可观测有效、性能达标 |

### 3.4 评审维度对比

| 维度 | architecture-review | technical-design-review |
|------|---------------------|-------------------------|
| **PRD可追溯性** | ✓ 检查F-ID覆盖、AC对应 | - 不检查 |
| **接口契约完整性** | ✓ 检查入参/出参/错误码/调用示例 | - 不检查 |
| **数据模型正确性** | ✓ 检查类型定义、关系约束 | - 不检查 |
| **方案可实施性** | ✓ 检查无TODO、无伪实现 | - 不检查 |
| **架构模式一致性** | ✓ 检查与project.md一致 | - 不检查 |
| **多端适配** | ✓ 检查平台特殊要求 | - 不检查 |
| **可测试性** | ✓ 检查F-ID对应测试用例 | - 不检查 |
| **安全性有效性** | - 不检查 | ✓ 检查实际安全措施 |
| **稳定性有效性** | - 不检查 | ✓ 检查限流/熔断/重试 |
| **可观测性有效性** | - 不检查 | ✓ 检查trace/log/metric |
| **性能达标性** | - 不检查 | ✓ 检查响应时间基准 |
| **多端生产就绪** | - 不检查 | ✓ 检查平台生产标准 |

### 3.5 决策逻辑

```
用户问："这份技术方案怎么样？"
    │
    ├── 问："能直接开发吗？" → architecture-review（快速Checklist）
    │       └── 产出：评审纪要（通过/待改进）
    │
    └── 问："能稳定上线吗？" → technical-design-review（深度量化）
            └── 产出：完成度报告（0-100分）
```

### 3.6 执行顺序

| 阶段 | Skill | 目标 |
|------|-------|------|
| Step 4.1 | architecture-review | 确保"方案可落地" |
| Step 4.2 | technical-design-review | 确保"方案可上线性" |
| Step 4.3 | check_solution.py | 量化评分验证 |

**重要**：两个Skill是串联执行关系，必须先通过architecture-review，再执行technical-design-review。

### 3.7 三种切分方案对比

| 方案 | 切分依据 | 优点 | 缺点 |
|------|---------|------|------|
| **方案A: 按生命周期阶段** | 设计/实现/上线 | 符合行业标准 | 需调整Skill体系 |
| **方案B: 按评审深度** | 快速Checklist/深度量化 | 简单明了、用户易选 | 内容仍有重叠 |
| **方案C: 保持当前结构** | - | 改动小、用户已习惯 | 仍有职责重叠 |

### 3.8 最终选择：方案B（按评审深度切分）

**理由**：

1. 符合用户当前流程（Step 4同时使用两个Skill）
2. 符合行业趋势（快速Checklist + 深度专项）
3. 符合第一性原理：
   - architecture-review = "这份方案能让我快速判断要不要做"
   - technical-design-review = "这份方案能让我判断上线后会不会出问题"

## 四、check_solution.py 评分机制改进

### 4.1 改进前（旧机制）

```python
# 旧机制：存在性检测
quality_checks = {
    "数据流描述": ["数据流", "流向", "流程"],  # 有=20分，无=0分
    "模块间接口": ["接口", "调用", "交互"],
    "输入输出定义": ["输入", "输出", "参数", "返回值"],
}
```

### 4.2 改进后（新机制）

```python
# 新机制：深度检测
quality_checks = {
    "数据流描述": ["数据流", "流向", "流程"],  # 基础分20分
    "模块间接口": ["接口", "调用", "交互"],
    "输入输出定义": ["输入", "输出", "参数", "返回值"],
}

depth_checks = {
    "状态管理": _has_state_definition(content),      # 深度分10分
    "缓存策略": _has_cache_strategy(content),        # 深度分10分
    "跨页面数据流": _has_cross_page_dataflow(content), # 深度分10分
}

# 最高可达 60 + 30 = 90分
```

| 接口定义清晰 | 100 | 100 | - |
| 技术选型合理 | 100 | 100 | - |
| 数据流设计 | 60 | **90** | +30 ✓ |
| 存储设计 | 60 | **90** | +30 ✓ |

| **评估** | 10维度定量 | 10维度定量（项目类型适配） |
| **判定** | 0-100%评分 | 0-100%评分 |

#### 新增项目类型适配指南

**轻量级项目（小程序、简单Web应用）判断标准**：
- 单体应用，无分布式协作
- 日活 < 10万
- 无高并发需求
- 开发团队 < 5人

**可裁剪的维度**：
| 维度 | 可裁剪原因 | 替代方案 |
|------|-----------|---------|
| 部署与运维（5%） | 无K8s/容器化需求 | 简化部署文档 |
| 可观测性（部分） | 小程序有自带监控 | 使用微信性能分析工具 |
| 扩展性（部分） | 功能简单，扩展需求少 | 预留扩展点即可 |

**重量级项目（微服务、高并发系统）判断标准**：
- 分布式架构
- 日活 > 100万
- 高并发要求
- 开发团队 > 10人

**完整10维度必须全部检查**
