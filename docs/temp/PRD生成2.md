# PRD文档生成技能

根据页面截图、原型图或文字描述生成结构化的产品需求文档(PRD)。

## PRD文档结构模板

始终使用以下标准PRD模板结构：

```markdown
# [产品功能名称] PRD

## 1. 产品概述

### 1.1 背景与目标
- 产品背景
- 核心目标
- 预期收益

### 1.2 目标用户
- 主要用户群体
- 用户特征
- 使用场景

## 2. 功能需求

### 2.1 核心功能
- 功能1：[功能名称]
  - 功能描述
  - 用户价值
  - 优先级：P0/P1/P2

### 2.2 功能详细说明
对每个功能进行详细描述，包括：
- 功能流程
- 交互逻辑
- 异常处理

## 3. 用户体验设计

### 3.1 页面结构
- 页面布局
- 信息架构
- 导航设计

### 3.2 交互设计
- 操作流程
- 反馈机制
- 状态变化

## 4. 技术需求

### 4.1 功能性需求
- 性能要求
- 兼容性要求
- 安全性要求

### 4.2 非功能性需求
- 可用性
- 可维护性
- 可扩展性

## 5. 验收标准

### 5.1 功能验收
- 核心功能验收点
- 边界条件测试
- 异常场景处理

### 5.2 体验验收
- 用户体验指标
- 性能指标
- 质量标准

## 6. 项目计划

### 6.1 开发排期
- 开发阶段划分
- 关键里程碑
- 交付时间

### 6.2 风险评估
- 技术风险
- 资源风险
- 时间风险
```

## 分析工作流程

### 步骤1：内容分析
根据输入类型进行分析：

**页面截图/原型图分析：**
- 识别页面元素和布局
- 分析交互组件（按钮、表单、导航等）
- 理解信息架构和用户流程
- 推断功能逻辑和业务场景

**文字描述分析：**
- 提取核心功能点
- 识别用户需求和痛点
- 理解业务目标和约束条件
- 补充缺失的细节信息

### 步骤2：需求梳理
- 将分析结果转化为具体功能需求
- 按优先级对功能进行分类（P0核心功能、P1重要功能、P2优化功能）
- 识别功能间的依赖关系
- 考虑边界条件和异常场景

### 步骤3：文档生成
- 使用标准PRD模板结构
- 填充具体的功能描述和需求细节
- 补充技术考虑和实现建议
- 制定验收标准和测试要点

## 内容质量标准

### 功能描述要求
- **具体明确**：避免模糊表述，使用具体的动词和名词
- **用户导向**：从用户角度描述功能价值和使用场景
- **逻辑完整**：包含正常流程、异常处理和边界条件
- **可测试性**：每个功能都有明确的验收标准

### 技术需求要求
- 提供性能指标建议（响应时间、并发量等）
- 考虑兼容性要求（浏览器、设备、系统）
- 包含安全性考虑（数据保护、权限控制）
- 评估技术实现难度和风险

### 项目管理要求
- 合理的开发排期估算
- 识别关键依赖和风险点
- 提供可行的里程碑划分
- 考虑资源配置和团队协作

## 常见场景处理

### 信息不足时的处理
当输入信息不够详细时：
1. 基于行业最佳实践进行合理推断
2. 在文档中标注假设条件
3. 提出需要进一步确认的问题
4. 提供多种可能的实现方案

### 复杂功能的拆解
对于复杂功能：
1. 按用户操作流程进行分解
2. 识别核心路径和分支路径
3. 分别描述每个子功能的需求
4. 说明子功能间的关联关系

### 跨平台需求处理
当涉及多平台时：
1. 分别描述各平台的特殊需求
2. 识别通用功能和平台特定功能
3. 考虑数据同步和一致性要求
4. 提供平台适配的技术建议

## 输出检查清单

生成PRD文档后，检查以下要点：

```
PRD质量检查：
- [ ] 产品目标和用户价值清晰明确
- [ ] 功能需求描述具体可执行
- [ ] 包含完整的用户操作流程
- [ ] 考虑了异常情况和边界条件
- [ ] 技术需求合理可行
- [ ] 验收标准明确可测试
- [ ] 项目计划具有可操作性
- [ ] 文档结构完整，逻辑清晰
```

只有通过所有检查项后，才输出最终的PRD文档。
