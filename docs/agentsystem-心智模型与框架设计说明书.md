# agentsystem 心智模型与框架设计说明书

> **文档类型**：内核备忘 / 迭代对齐用（非对外产品文档）  
> **目的**：在演进 agentsystem 时保持 **分层清晰、职责不漂移、阅读顺序可维护**。  
> **维护**：当 `usr-devclaw/user.json`、`.cursor/rules/AGENT.mdc`、`usr-devclaw/usr-rules/` 或 workspace 约定有重大变更时，应回顾并更新本文。

---

## 1. 初心与边界

### 1.1 agentsystem 是什么

- **多宿主、可复用的研发治理壳**：为 **Cursor 等 IDE** 提供规则入口、（可选）Python 门禁、以及 **可定点加载** 的技能 / 记忆 / 脚本库（`sys-root/lib/`）。
- **与业务仓库解耦**：业务代码与产品文档在 **各业务项目 Git 仓库**；agentsystem **不**承担业务交付物正文，**只**承担「如何协作、如何读、如何被门禁约束」。

### 1.2 agentsystem 不是什么

- **不是**单一业务应用的后端或业务仓库。
- **不是**要求 Agent 通读 `sys-root/lib/` 全库；全库是 **工具箱**，按需 **定点打开**。
- **不是**替代业务项目的 `openspec/AGENTS.md` / PRD；usr-rules 是 **原始规范**，业务仓 `openspec` 是 **落地与指针**。

### 1.3 一句话心智模型

```
宿主规则（.mdc） → 识别范围 → 业务项目入口（claw-config）→ 导航与偏好（user.json）→ 按需下沉（openspec / usr-rules / sys-root 单文件）
```

### 1.4 全局架构图（线框）

> 一眼看清：**系统层（唯一共用）**、**多套用户层 Agent 角色**、**各项目层归属关系**（非加载时序）。

**读图（避免误解）**：**系统层** 全局 **唯一**，全体共用（工具/宿主/门禁）。**用户层** 不是只有一个「全局模板」，而是可演进为 **多套并行的「用户层 Agent 角色」**（**举例**：**独立开发者**、**股票研究员** —— 各为一套 `profile` / usr-rules 组合 / `user.json` 变体）。**当前仓库约定**：**`usr-devclaw/` 即「独立开发者」用户层角色的落点**（`usr-rules` + `user.json`）；**「股票研究员」** 等其它角色 **不**占用当前 `usr-devclaw`，须在演进中 **另设** 用户层根目录或等价配置，避免与独立开发者混读。**每个业务仓库（项目层）在治理语义上仅归属其中 1 个用户层 Agent 角色**（**1:1**）：多仓可 **共用** 同一角色，但 **同一项目不得混用多个角色**。单项目落地时，是 **系统层 + 其选定的那一个用户层角色** 共同支撑，而非「所有用户层条文同时灌进一个仓」。

## 2. 两种视角：产品三层 vs 加载洋葱

> **重要区分**  
> - **产品设计视角**（本节 2.1）：回答 **边界**——系统 / 用户 / 项目 谁管什么；参考 **OpenClaw** 思想，支撑 **不同用户层 Agent 角色**（如 **独立开发者**、**股票研究员**）与 **同一使用者下多个业务项目**（如 Curiobuddy、ShopifyTheme）。  
> - **数据加载与约束视角**（本节 2.2）：回答 **加载顺序与约束强度**（谁先读、谁更「硬」）；即前文的 **洋葱模型**，与 2.1 **正交互补**，不是同一套编号的两种叫法。

---

### 2.1 产品设计：系统层 → 用户层 → 项目层（OpenClaw 思想）

目标：**一层一责**，避免把「内核能力」「个人偏好与角色」「具体业务交付」混在同一仓库或同一文件里。

| 层级 | 职责 | 服务谁 | 典型落点（当前实现） |
|------|------|--------|----------------------|
| **系统层** | **与 Agent「读什么规则」无关的可执行内核**：宿主入口、门禁脚本、**全库**技能 / 记忆 / 脚本（`sys-root/lib/`）；**被所有用户层 Agent 共用**，不包含「给 Agent 看的业务/流程条文」 | 所有使用该 agentsystem 的宿主与上层配置 | `agentsystem` 仓库：`sys-root/lib/`、`.cursor/rules/AGENT.mdc`（范围判定壳）、`sys-root/lib/scripts/agents-md-enforcer/` 等 |
| **用户层** | **面向 Agent 的规则与配置**：跨项目的 **流程与目录规范**（usr-rules）+ 同一席位下的 **偏好、阅读顺序、可扩展多角色**（`user.json` 等）；**与具体业务仓库解耦**，但约束「Agent 如何协作」 | 「用 agentsystem 的这个人」及其定义的 **Agent 行为空间**—跨多个业务仓库；**角色举例**：**独立开发者**（工程交付、迭代节奏）、**股票研究员**（研报口径、数据与合规禁区） | **当前**：**`usr-devclaw/` = 「独立开发者」角色落点**（`usr-rules` + `user.json`）；**其它角色**另设根目录或等价配置（演进）。仍属用户层 |
| **项目层** | **单个业务产品 / 仓库** 的交付与治理：运行时人设、openspec、PRD、源码、`src/` 等 | 单个业务项目（如 Curiobuddy、某 ShopifyTheme 仓） | 各 **业务 Git 仓库** 根下：`claw-config/`、`openspec/`、`docs/`、`src/`；通过 **`agentsystem/workspace/{project}` → 业务仓 `claw-config/`** 软链挂载 |

**与多项目、多角色的关系**：

- **系统层** 全局 **一套**：所有项目共用同一内核（skills、宿主壳、门禁实现等）。  
- **用户层** 可有 **多套「用户层 Agent 角色」**（不同 `profile`、不同 usr-rules 组合、不同 `user.json` 变体等），**并行存在、互不替代**；**示例两档**：**独立开发者**、**股票研究员**（可再扩展其它角色）。**当前**：**`usr-devclaw/` 即独立开发者** 的落点；股票研究员等 **不**使用本目录。  
- **项目层**（每个业务 Git 仓库）**在治理上仅绑定其中一个用户层 Agent 角色**（**1:1**）：多个项目可以 **共用** 同一角色（例如仓库 A、C 同属 **独立开发者**，仓库 B 同属 **股票研究员**），但 **单个项目不得同时按多个用户层角色混读/混配**（避免人设与规则冲突）。  
- **项目层** 仍按仓库隔离具体产品；与 **usr-rules** 的「跨项目复用」是指 **同一角色下的多仓复用条文**，不是「一个仓同时消费多套用户层角色定义」。

---

### 2.2 数据加载与约束：洋葱模型（L0–L5）

本节描述 **运行时/Agent 侧「先读谁、谁更硬」**，侧重 **加载逻辑与约束强度**（由内到外可理解为：从宿主强制到按需工具库）。**与 2.1 的三层不是一一重命名关系**，而是 **实现细节视图**。

| 层级 | 典型位置 | 职责 |
|------|----------|------|
| **L0 宿主入口** | `.cursor/rules/AGENT.mdc` | 判定工作区范围（本仓 agentsystem vs 业务项目）；业务项目时强制 **claw-config/AGENTS.md** + 门禁（若启用）。 |
| **L1 业务运行时** | 业务仓库 `claw-config/AGENTS.md` | **本项目**人设、沟通习惯、禁区；与治理层 **引用关系**，不重复全文规则。 |
| **L2 业务治理** | 业务仓库 `openspec/AGENTS.md`、`openspec/project.md` | **本项目**变更目录、日志、技能原则等；**指针**指向 usr-rules。 |
| **L3 用户层（偏好与导航）** | `usr-devclaw/user.json` | **个人偏好** + **跨仓库阅读顺序** + **禁止通读 sys-root** 的提示；**不**替代 L1/L2。 |
| **L4 用户层（跨项目 Agent 规则）** | `usr-devclaw/usr-rules/OpenSpec.md`、`PROJECT-CONFIG.md` 等 | **面向 Agent** 的流程与目录规范（**非**系统层可执行代码）；业务仓 `openspec` **落地**并引用；加载顺序上常 **晚于 L1/L2**（里程碑/开迭代再读）。 |
| **L5 系统层工具库** | `sys-root/lib/skills/`、`memory/`、`scripts/` | **可执行/可引用** 的技能与脚本；按任务 **定点** 读取；默认 **禁止** 全库浏览。 |

> **产品分层与洋葱编号的对应（修正）**：**`usr-rules` 属于用户层，不属于系统层**——名称中的 `usr` 即 **面向 Agent 的用户规则**；系统层是 **所有用户层 Agent 共用** 的 **内核与工具**，若把 usr-rules 放进系统层，会推出「内核 = 给 Agent 读的条文」，与「系统层 = 全体共用的基础设施」**语义矛盾**。  
> **L3** 与 **L4** 在加载顺序上拆成两步，在 **产品三层** 中 **同属用户层**（偏好/导航 vs 流程规范）。

**对照关系（便于迭代时不混）**：

- **系统层（产品）** ≈ **L0（宿主壳）+ L5（`sys-root` 工具库）+ 门禁代码**；**不含** `usr-rules` 条文。  
- **用户层（产品）** ≈ **L3（`user.json`）+ L4（`usr-rules`）**；二者均为跨项目 **Agent 配置/规则**，仅形态不同（JSON 偏好 vs Markdown 规范）。  
- **项目层（产品）** ≈ **L1 + L2** + 业务仓其余目录。

**加载顺序与产品分层** 仍不必同序：例如 **L4** 常在 **L1/L2** 之后读（里程碑），但 **产品上** 仍属 **用户层**。

## 3. 关键目录与文件职责

| 路径 | 职责 |
|------|------|
| **`agentsystem/.cursor/rules/AGENT.mdc`** | 宿主壳：范围判定、业务项目必读 `claw-config/AGENTS.md`、门禁入口；**不**展开 `read_order` 细节（避免与 `user.json` 漂移）。 |
| **`agentsystem/usr-devclaw/user.json`** | **用户层（独立开发者角色）**：**偏好** + **read_order_hint / read_order_notes**；说明 ①②③④ 与「里程碑再读 usr-rules」。 |
| **`agentsystem/usr-devclaw/usr-rules/*.md`** | **用户层（独立开发者角色）**：**面向 Agent** 的跨项目规则（OpenSpec、PROJECT-CONFIG 等）；**不是**系统层内核代码。 |
| **`agentsystem/sys-root/lib/`** | **系统层**：技能、记忆、脚本等 **可执行/可引用** 工具库；**按需**加载。 |
| **`agentsystem/workspace/{project}/`** | 通常 **软链** 到业务仓库内 `claw-config/`（名称以仓库为准，如 `claw-config`）。 |
| **业务仓库 `claw-config/`** | **项目层**运行时：AGENTS、SKILLS、MEMORY、TOOLS 等。 |
| **业务仓库 `openspec/`** | **项目层**治理与变更：`AGENTS.md`、`project.md`、`changes/`、`specs/` 等。 |
| **业务仓库 `src/`**（usr-rules 约定） | **项目层**：**业务源码根**唯一合法目录名（与 `openspec/project.md` 一致）。 |

## 4. `user.json` 在链路中的位置（结论固化）

> 本节固化架构共识，避免在「第几环读 user.json」上反复摇摆。

### 4.1 是否有必要存在

**有必要保留**。原因：

- **集中**存放「跨项目、跨会话」的 **阅读顺序** 与 **禁止通读 sys-root** 的约束；若删除，易散落到 **AGENT.mdc**（过长、难改）或 **每个业务仓 AGENTS**（重复、易漂移）。
- **偏好**（语言、verbosity 等）属于 **用户层**，不属业务项目规则正文。

### 4.2 它不是「第几份 AGENTS」

- **不替代** `claw-config/AGENTS.md`（项目身份与禁区）。
- **不替代** `openspec/AGENTS.md`（业务治理）。
- **定位**：在 **已识别业务项目身份** 之后、**深读 openspec / usr-rules / sys-root 大块之前** 的 **导航图**。

### 4.3 推荐阅读时机（逻辑顺序）

1. **AGENT.mdc** 已由 Cursor 加载（入口壳）。  
2. **识别** 当前任务属于 **业务项目** 还是 **agentsystem 本仓**。  
3. **业务项目**：读 **`claw-config/AGENTS.md`**（建立项目上下文与禁区）。  
4. **再读** **`usr-devclaw/user.json`** 中的 `read_order_hint` / `read_order_notes`，对齐 **①②③④** 与 **定点打开** 策略。  
5. 按任务继续：**openspec**（②）、**定点 skill/memory**（③）、**里程碑时 usr-rules**（④）。

> **说明**：①②③④ 的编号写在 `user.json` 内；**不必**把「打开 user.json」硬写成与 ① 同级的「第 0 步」产品编号，但 **语义上** 应在 **②③ 展开之前** 完成导航对齐。

## 5. 业务项目 vs agentsystem 本仓

| 维度 | 业务项目（如 Curiobuddy） | agentsystem 本仓 |
|------|---------------------------|------------------|
| 工作区根 | 业务仓库根 | `agentsystem` 根路径（见 AGENT.mdc 别名表） |
| **产品分层** | **项目层** | **系统层**（内核）+ **用户层**（当前 **`usr-devclaw/` = 独立开发者**；物理上在本仓，**分层上**≠ 系统层） |
| **是否**强制读 `claw-config/AGENTS.md` | **是** | **否** |
| **是否**调用 `quick_enforce`（若启用） | **是**（按项目名） | **否** |
| **主要读** | claw → openspec → 按需 sys-root | README、docs、脚本、内核规则 |

## 6. 质量与门禁在架构中的位置

- **规则层**（`.mdc`、AGENTS、openspec）给出 **可执行约束与流程**。  
- **强制层**（Python `agents-md-enforcer` 等）在 **业务项目** 场景下 **可选** 阻断「未读 AGENTS 就改仓」等风险。  
- **质量闭环**（10 步、OpenSpec、复盘）的 **正文**在 **usr-rules** 与 **业务 openspec**；agentsystem 提供 **skills/memory** 支撑 **How**，不替代业务 **What**。

## 7. 迭代时「不忘初心」检查清单

在发起 agentsystem 大改前，建议快速自检：

- [ ] **产品三层是否仍清晰**：系统层 / 用户层 / 项目层 **边界是否未被糊在一起**？  
- [ ] **洋葱加载模型是否仍可用**：宿主 → claw → openspec → user.json 导航 → 按需 sys-root / usr-rules 是否仍说得通？  
- [ ] **是否仍解耦业务仓**：业务代码与 PRD 是否仍在 **项目层** 仓库，agentsystem 是否仍 **不** 塞业务正文？  
- [ ] **sys-root 是否仍按需**：默认 **禁止** 通读全库是否仍成立？  
- [ ] **user.json 与 AGENT.mdc 是否仍分工**：入口壳 **不** 长文化阅读顺序；顺序 **集中** 在 `user.json` 是否仍成立？  
- [ ] **workspace 软链约定**是否仍文档化（路径因本机而异，避免写死唯一绝对路径）？  
- [ ] **业务源码根** 与 usr-rules 是否仍一致（当前约定：**仓库根 **`src/`**）？  
- [ ] **多用户层 Agent 角色 vs 单项目唯一归属**：是否仍能说清 **多角色可并存**、**每项目仅归属其中一个用户层角色**（可多项目共用同一角色、**禁止**单仓混多角色）？  
- [ ] **多业务项目** 是否仍能通过 **项目层隔离 + 用户层（按角色）配置** 表达，而无需复制整套系统层？
